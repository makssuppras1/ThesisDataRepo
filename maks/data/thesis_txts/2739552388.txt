Abstract
This thesis explores the implementation of Vulkan software rasterizer using compute shader,
meshlet and compression techniques to improve rendering performance. Rasterization, a
key method in transforming 3D data into 2D-pixel images, is typically handled by the
hardware in GPUs. However, fixed pipeline stages often suffer from inefficiencies, es-
pecially when processing micro-polygons and non-optimized mesh data. The study in-
vestigates GPU architecture correlating to different pipelines and multiple compression
methods that reduce memory bandwidth and computational load. The meshlet-based ap-
proach is used for improving vertex reuse efficient culling, leading to better performance
than fixed pipelines. The results show that the software rasterizer outperforms traditional
fixed-function hardware pipelines, particularly for objects with high triangle counts. The
thesis concludes with more analysis of profiling data, architecture considerations, and the
potential for extending this work to different specialized rendering platforms.

Introduction
Rasterization is a technique in computer graphics that transforms 3D data into 2D pixel-
based images. It is fundamental in various applications such as computer games, computer-
aided design (CAD) software, and augmented and virtual reality (AR/VR) technologies.
Over the past few decades, this algorithm has become an integral part of hardware pipelines,
primarily in GPUs (Graphics Processing Units), starting from the 1970s in Figure 1.1.
This integration of rasterization into GPUs leverages parallel processing capabilities, sig-
nificantly enhancing computational efficiency. Nowadays, we are seeing consumer-level
graphics cards that can potentially process billions of triangles per second, which are still
improving with every generation.
Figure 1.1: A history of graphics dedicated GPU features in the last century. [Bly08]
Over the past decades of GPU advancements, two key aspects have significantly influ-
enced hardware architectural decisions.
One is the choice between programmable pipelines and fixed-function stages.
Programmable shaders [Las+95], now the essentials of all graphics applications, allow
customization in vertex and fragment processing. While originally, GPUs were purely
fixed-function devices.
Some fixed-function implementations are more cost-effective
and power-efficient [Bly08], and few compelling cases remain for conversion to pro-
grammable ones. Examples include rasterization, texture filtering, depth buffer, etc.
Another aspect comes at the intersection of dedicated components and General-Purpose
Graphics Processing Unit (GPGPU) [VC13]. As the demand for more flexible and pow-
erful scientific computing computations grew, GPU evolved and incorporated general-
purpose workloads. Notable examples would be CUDA and openCL [Nic+08; SGS10],
which allow GPUs to perform non-graphics computations. In terms of dedicated compo-
nents, consumer graphics cards have increasing specialized support in machine learning
(Tensor Cores), ray tracing (RT cores), and mesh shading [Nvi20], alongside traditional
graphics functions.
This opens an optimization space for hardware engineers where dedicated and general-
purpose units must coexist, often sharing some of the same underlying cores. Some manu-
facturers, though, have chosen separation into different products, as in Figure 1.2 by AMD

All these advancements in GPU have given software developers productivity in countless
fields. A variety of libraries are built upon low-level hardware and offer higher-level
control. For graphical applications, developers can use advanced APIs such as OpenGL,
DirectX, and Vulkan to access almost all hardware configurations and implement features
with ease.
1.1
Problem Statement & Motivation
As has been introduced, a versatile GPU has both dedicated and general-purpose compo-
nents. Though manufacturers still put efforts into optimizing the graphics pipeline, there
are certain costs in marshaling, scheduling, and handling physical elements (bigger chip,
higher temperature, etc.). This leads to the implication that graphics applications can cause
a loss in performance. On the other hand, fixed pipeline stages are intuitively designed
on the pixel level. Research has shown that the fixed stage, especially the pixel shader,
performs poorly if the primitive size is much smaller than a pixel [Fat10].

In response, software rasterization on GPUs has become a promising proposal for address-
ing these issues. This project aims to build a compute shader-based rasterizer, offering
flexibility for developers who want to bypass the fixed pipeline and implement custom
changes, such as compression in this project. At the same time, meshlet-based render-
ing offers improved parallelization for the graphics pipeline. Although mesh shading has
gained popularity in recent years, it remains unsupported by many GPUs, as shown by
figure 1.3. This MVP will use meshlets to create a prototype to emulate mesh shaders on
possibly Android and lower-end devices. The project will also consolidate the two-stage

There are also relevant and integral topics in exploring GPU architecture. A deeper dive
into the GPU black box will uncover opportunities to enhance GPU performance, both in
compute and fixed context.

Related Work
In this section, we explore the fundamentals and related work surrounding software ras-
terization and GPU technology.
2.1
Software Rasterization
The rasterization process generally refers to determining the pixel values from data struc-
tures such as 3d models, characters, vectors or even neural networks [KUH18]. [Gha+89]
in the early days provided a systematic overview of implementing raster scan displays,
and today, the framebuffer method has been the mainstream way for raster output.
Rasterization is commonly applied to 3D models, and three geometric methods exist. The
scan line method [CS80; Lan+80] processes one row of pixels at a time and calculates
color and depth sequentially. It can handle parametric surfaces and non-triangle polygons
by identifying odd or even edges, as in figure 2.1. The edge function method [Pin88],
exclusively for triangles, calculates barycentric attribute values of a pixel tested inside
triangles. Lastly, the Bresenham algorithm [Bre77], which rasterizes lines and can be
extended to polygons, is less commonly used than the other two methods.
Figure 2.1: A visualization of scanline rasterization [Yan24].
Academic
As mentioned, hardware rasterization can sometimes underperform, leading to experimen-
tation with software rasterization on GPUs over the past two decades. Larrabee [Sei+08]
was an early GPU software rasterizer that achieved high performance and flexibility through
programmable components. Similarly, Freepipe [Liu+10] demonstrated a fully programmable
rendering pipeline using CUDA atomic operations in parallel, showing a twofold perfor-
mance improvement when rendering large sets of triangles. Laine and Karras [LK11]
developed a tile-based software rasterization pipeline using CUDA, with fine and coarse
4

raster by tiles. Although their method did not outperform hardware graphics, they sug-
gested that specialized tasks could benefit from this approach. More recently, an aca-
demic software rasterization pipeline [Ken+18] constructed a conservative coverage mask
in scanline order directly from triangle edge equations, adding several innovative features
to their pipeline.
Industry
From the industry, Frostbite [Wih16] was one of the first to utilize a compute rasterizer
for culling and improving the AMD rasterizer efficiency. The most notable example is
Nanite [Bri21], which introduced a high-performance hybrid rasterization system using
a visibility buffer technique. This system, combined with other techniques, has achieved
remarkable results. Id Software has also integrated software rasterization into their engine
[GGW20], further showcasing its potential in real-time graphics rendering. Many other
examples follow [Dro17; HA15].
Specialization
Researchers have also developed various techniques for software rasterization in specific
scenarios. [SKW22] implemented a custom compute shader rasterizer to render large point
clouds in real time on commodity hardware. Their approach includes optimizations such
as frustum culling and level-of-detail (LOD) models. [Jak24] created a compute shader
rasterizer to achieve precise order-independent transparency. The graphics community
has also developed a software rasterizer specifically for rendering hair [Mat24].
Micropolygon
[CCC87] first introduced the concept of micropolygons, which generate subpixel primi-
tives in the rendering pipeline. [Fat10] further explored micropolygons, identifying chal-
lenges such as higher processing costs, inefficient parallelization, and shading redundancy
on GPUs. In response, he proposed improved parallelization techniques and rasterization
algorithms in motion blur and tessellation [Fis+09; Fat+09]. Nanite’s mesh-shader-like
implementation [Bri21] claims to achieve three times faster performance than hardware
through its hybrid approach.
Since micropolygon rendering is often associated with complex geometry, many algo-
rithms have been developed to improve data structures for level-of-detail (LOD) and tes-
sellation. A notable new technique, called micro-mesh [MMT23], supported by Nvidia,
has become a fixed feature of the latest GPU architecture [Nvi23]. This method stores
a base mesh and uses tessellation and displacement mapping to render highly detailed
geometry.

2.2.2
GPU architecture and Fixed Graphics Pipeline
Although GPUs can be accessed differently from CUDA, graphics API or other layers,
they all share the same SM and compute units with minor execution differences. The
same holds for the fixed graphics pipeline. The arithmetic operations in the graphics
pipeline also use CUDA cores and warps while relying on graphics-specific components
to function. In this section, we gently go through the pipeline [Chr15] to have a simpli-
8

Figure 2.6: Mesh shading pipeline [Kub18]
fied overview of it, more pertinent to the GPU figure 2.2, the pipeline figure 2.6 and this
project.
1. Draw Command Submission: The graphics API issues draw commands or buffers
to the GPU through the driver.
2. Vertex Attribute Fetch: The Primitive Distributor begins processing the indices
from the index buffer and generates triangle work batches, which are sent to the
GPCs. Inside a GPC, the Poly Morph Engine fetches vertex data from the indices.
3. Vertex Shader Execution: Warps of 32 threads are scheduled by the SM’s sched-
uler to process vertices in the vertex shader. This may require several dispatch
cycles. Some instructions take longer due to memory loads, but the warp scheduler
can switch between tasks to optimize execution.
4. Viewport Transform and Rasterization Prep: After applying viewport transform,
triangles are prepared for rasterization. The L1 and L2 Caches manage communi-
cations.
5. Rasterization: Triangles are sent from the GPC to the raster engines by tiles,
with jobs split accordingly. This stage generates pixel information.
6. Pixel Shading: Groups of 32 pixel threads (8 sets of 2x2 pixel quads) work on
pixel shading. These 2x2 quads also calculate derivatives for mipmap filtering and
other effects. Any thread within the 2x2 quad that doesn’t cover a triangle’s sample
locations is masked out (via gl_HelperInvocation).
7. Render Output: The final colors and depth values of the shaded pixels are writ-
ten to the render targets. This data is then passed to the ROP (Render Output
Processor) subsystems, where depth testing and blending occur atomically in the
frame buffer to avoid write conflicts. Some GPUs apply memory compression at
this stage for efficiency.

2.2.3
Meshlet and Mesh shader
The concept of meshlets originates from the clustering of polygons, and there has been ex-
tensive research in computational geometry on spatial partitioning of objects. Early work
by [ZH97] utilized clustering to perform culling and accelerate rendering. Many game
engines around 2017 have adopted cluster-based rendering systems, as seen in [Dro17;
Bri21; GGW20; Wih16; HA15]. As previously mentioned, these engines unanimously
rely on compute shaders due to their computational power.
The GPU industry has since evolved cluster rendering into a fixed pipeline known as the
mesh shading pipeline. Mesh shaders utilise culling and handle vertex reuse, LOD (Level
of Detail), tessellation, and in-memory compression, maximizing GPU hardware utiliza-
tion. This approach effectively replaces legacy shader stages, as shown in Figure 2.6. As
described by [JFB23], various generation strategies provide a foundation for this project’s
experiments.
2.2.4
Vulkan Graphics API
Vulkan is a low-level, cross-platform API developed by Khronos for 3D graphics and com-
pute operations. It offers efficient, fine-grained control over CPU and GPU tasks. Along
with DirectX, Vulkan remains one of the leading graphics APIs, continuously updated to
support modern GPU technologies such as ray tracing and mesh shading.
For this project, Vulkan facilitates the construction of both compute and fixed-function
graphics pipelines. Additionally, it integrates well with tools like RenderDoc, Nsight
Graphics, and other software for debugging and profiling.
2.3
Mesh Locality and Compression
As a largely black box for software developers, the GPU pipeline can handle highly opti-
mized operations but quite implicitly. One of the most important optimizations is cache
optimization. In the third step of the pipeline 2.6, if input triangles are poorly arranged for
GPU, memory loading times can significantly increase, creating a bottleneck.
Early discussions on optimizing mesh locality were introduced by [Dee95] and [Hop99],
along with their work on mesh compression. Later, [SNB07] presented a method to im-
prove post-transform vertex cache efficiency and reduce overdraw independently of the
viewpoint, enhancing meshlet culling. More recently, [Ker+18] further investigated this is-
sue and highlighted the importance of GPU batch size, where a new method outperformed
existing mesh optimizations.
As mesh compression is closely linked to mesh locality, research has focused on creat-
ing mesh topologies that optimize both locality and compressibility. This is critical, as
memory and CPU-GPU communication bottlenecks continue to impact high-performance
systems and mobile devices.
For Nanite [Bri21], developers emphasize different compression strategies for memory
and disk representations. In-memory compression for graphical systems often involves
specialized techniques for various vertex attributes. For example, [Cig+14] proposed a
method to compress normal vectors by one-third. Quantization, although lossy, is another
common technique favored by engineers.

3
Software Rasterization Pipeline
This section provides a high-level overview of the rasterization pipeline, divided into four
parts.
The first part handles the preprocessing of meshlet generation. Before being sent to GPU
buffers, compression techniques are applied to reduce transmission sizes and optimize
desired properties such as locality. Once the data reaches the GPU, it is unpacked and
checked for culling conditions. In the final stage, each triangle is processed based on
its screen coordinates using the barycentric rasterization algorithm, which determines the
pixel values.
Overall, this pipeline is an experimental model to assess situations where the traditional
hardware graphics pipeline may underperform. Additional features. could be incorpo-
rated into this customizable compute pipeline such as improved shading and texturing
functionality
3.1
Meshlet Data Packing
The pipeline first parses data from 3D format files, which are processed into an indexed
triangle list consisting of a vertex buffer and an indice buffer. We then refer to different
strategies from [JFB23] to generate meshlets.
Figure 3.1: A visual representation of meshlet data structure [Kub18]
The algorithm will pack the meshlet information into four buffers for pipeline usage, as il-
lustrated in Figure 3.1; this data structure originates from Nvidia implementation [Kub18]:
1. Meshlet Description Buffer: This buffer contains details for each meshlet, such as
the starting offset within the vertex index and primitive index buffer, as well as the
number of elements. It also holds other customizable information, such as bounding
box and cone angle/normals for meshlet-level operations.
2. Vertex Buffer: The vertex buffer holds all the per-vertex attributes data. In our
case, we use positions, normals, and texture coordinates in our vertex structure. The
normal will be used for shading the model.
3. Vertex Index Buffer: Each segment of vertex indices defined by the meshlet de-
scription buffer represents a distinct collection of unique vertex indices associated
12
Software Rasterization with Vertex Data Compression
==PAGE:18==
with the corresponding meshlet. These indices refer to the position in the vertex
buffer.
4. Primitive Index Buffer: Similar to the vertex index buffer, the primitive index
buffer stores indices for each triangle in order within each meshlet. These indices,
starting from 0 in each meshlet’s section, correspond to the indices in the vertex
index buffer.
3.2
Buffer Compression
As mentioned in the previous chapter, geometry data compression can be classified into
connectivity-driven methods or quantitative (non-connectivity-driven) methods. In this
system, the latter is more suitable for several reasons: 1) while effective in certain cases,
topology-driven methods can be slow for high-performance systems [Bri21], adding over-
head to this software compute pipeline. This makes it less ideal for comparison with the
fixed-function pipeline. 2) Many quantitative techniques offer near-instant decompression
and modest compression rates (around 50%). This can save computational resources and
free up bandwidth for other operations. 3) The system could be customized to include a
task shader stage for future specific compression needs, similar to the approach taken by
[MMT23] for their newly designed geometry data format.
Based on the discussion above, this system adopts a similar approach to Nanite for real-
time compression of memory representation instead of disk representation. Compression
is considered on four buffers, as outlined in Section 3.1.
The meshlet description buffer occupies minimal space and can be excluded from com-
pression. For the primitive index buffer, Nvidia’s method packs four 8-bit indices into a
32-bit variable. This is because the meshlet generation enforces a vertex limit of 64, which
is representable by 8 bits.
For the vertex buffer, different encoding methods are applied to individual attributes to
optimize storage as below:
• Normal: We use octahedral mapping [Cig+14] to encode world space normals,
which maps a float32 vec3 variable to vec2 variable, saving around 33% space.
• Positions: We apply classical half quantization [IEE08] [Kap17] to the vertex po-
sitions from 32 bit to 16 bit. This process holds feasible for most position data
smaller than 65504 and only suffer minor fraction precision error. The common
crack anomaly [Cho+00] would not be a problem in this system as joining vertices
will be quantized to the same value. This will save 50% of the space.

both decompression and separation of packed variables.

fully. One key technique is meshlet cone culling. When generating meshlets, the mesh
cone and mesh normal information can also be calculated. Nvidia’s approach [Kub18]

Shading and Depth Culling
The implementation also covers shading and depth culling. Depth culling operates using
a depth buffer, which is initialized to values of 0. Before the color value is calculated, the
depth value is interpolated similarly. The depth value is stored in the buffer the first time
the pixel is shaded. If the pixel is shaded again by another triangle, the new depth value
is compared, determining whether the new color interpolation proceeds.
This pipeline is designed to balance complexity and functionality, focusing on producing
solid graphical results while omitting certain features such as MSAA (Multisample Anti-

the double loop in the rasterizer. The primary goal of this project is to leverage GPU
computing capabilities and compare them with the fixed-function pipeline, which will be

4.1.1
Preprocessing Mesh
For all the 3D models used in this project, we apply Blender’s ’Shade Smooth’ operation.
When the OBJ model is processed into the vertex buffer, a single vertex may be duplicated
due to differing normals from adjacent faces. Smooth shading assigns each vertex a fixed,
interpolated normal, eliminating the need for duplication in the vertex buffer.
The index buffer size remains the same regardless of this process. However, without
smooth shading, the vertex buffer might see a duplication factor of 2-4 times. This sig-
nificantly increases memory overhead when accessing vertex data, leading to suboptimal
performance in any pipeline.
The pipeline then proceeds with generating meshlets and applying compression, as out-
lined in the previous chapter.

4.2
Parallelized Compute Shader for Meshlets
This section outlines the structure of the compute shader used for operations before and
during rasterization. The shader is written in GLSL (version 460) and contains a main
entry function, a rasterization function (see Listing 3.2), and several helper functions. The
design of the rasterization compute shader draws inspiration from various systems [Bri21;
Kub18; Tel21], but mostly mimics the graphics pipeline mentioned in GPU architecture
section 2.2.2.
A true emulation of a mesh shader typically involves a two-stage shader system, where
the task shader (first stage) handles operations like culling, tessellation, or LOD (Level-of-
detail) switching, all of which are essentially compute shader tasks. However, in our case,
we opted for a one-stage compute shader based on a few considerations. One key design
rationale behind task shaders is to avoid transferring data between the GPU and CPU, en-
abling fully GPU-driven rendering. In our implementation, a single-stage compute shader
is sufficient to handle all data transfer to the GPU. Meanwhile, having two stages would
also cause extra overhead, considering the light workload of this project.
4.2.1

recommended by [Kub18], we pack up to 64 vertices and 126 primitives per meshlet. Hard-
ware typically allocates blocks in contiguous 128-byte chunks, making a 126-primitive
meshlet the maximum size that fully utilizes this space.

Discussion
In the final chapter, we discuss the most important topics with respect to the system results,
literature study and future potentials.
6.1
Mesh and Compute Shader
GPU-driven rendering has long been a topic in the graphics community. To achieve its
goal of reducing CPU-GPU communication and having GPU handle most work, shaders
are essential in this process of determining how the hardware can execute. It can be fair to
say all shaders are compute shaders, as they use the basic SM, cache, etc mentioned in the
second chapter. This applies that many optimization skills are universal, such as subgroup
intrinsic.
While to achieve optimal performance, it is essential to choose the suitable shader under
the project’s context. NVIDIA [Kub18] has mentioned that when there is little work for
draw calls, adding more stages can impact performance negatively (especially in models
with many small objects). On the opposite, if the task-shader becomes too complex (much
work involving memory), it has negative performance impacts as well. Consolidation into
one stage for better streamlining in this project is one choice made by this note. However,
GPU system is a black box and production software should take all experiment results into
account given time. The same also goes for things such as the dispatch size for compute
shaders.

6.2
Fixed Pipeline, Micropolygon and Better Method
The fixed pipeline, as has been claimed as inefficient for micropolygons [Fat10], will
possibly stay in the current way because of other important features and hardware designs.
LOD (Level-of-detail) techniques are always essential for large scenes to avoid the micro
polygons and, of course, reduce the workload. However, micropolygons are not 100%
avoidable if more detailed models appear in modern applications, such as in figure 6.1. A
much better way is to try to use a hybrid approach to carefully unify both software and
hardware rasterization like Nanite [Bri21].
The system can be generalized by figure 6.2. In the first pass, a compute shader handles
all culling and LOD operations, which decides the objects that should be rendered. This
shader can do indirect rendering, dispatching two paths into rasterization and achieving
GPU-driven rendering. In the second pass, the triangle size decides which route it will
take. By the final pass, they will converge into the visibility buffer with triangle IDs. The
technique visibility buffer [BH13] takes over traditional forward/deferred rendering and
integrates well with compute shader/rasterizer by leaving as little memory footprint as
possible.
Figure 6.2: A overview of pipeline in Unreal Nanite [Cha21].
6.3
Compression
It has been previously discussed that production software compression should have disk/mem-
ory representation, and memory compression matters in this project. It is essential about
the balance for engineering to achieve performance with regard to memory transfer and
processing capability. The benefits for achieving this is very evident, we can fit more data
into GPU memory and speed up data transfer between CPU and GPU, without incurring
too much cost of doing compression as well. A side note of GPU hardware compression,
following this principle, is texture and color compression [SSK12].
For the compression methods, this project’s choice mimic Nanite [Bri21] and handle dif-
ferent attributes each. We have seen many compression schemes explored with mesh
shading in the second chapter. They include more advanced topology encoding. While
for production-ready software, this approach has not been the mainstream, mostly due to
large overhead and difficult system integrations. With the emergence of System-on-Chip
(SoC) [Sal+06] system and new memory transfer technology [Li+20], the future might
hold a new outlook for compression used in the graphics and compute community.

6.4
Extension of the system
Developers can set out to customize this system for different purposes in the future.
Mobile Systems
For mobile device, a tile-based [OA11] memory-efficient variant can be pursued. As
we have mentioned from figure 1.3, mobile devices do not support mesh shading in the
pipeline and this extension could help mobile devices render faster on complex meshes.
Also, it is possible to modify the system to support binning for tiling rasterization, in which
vertices are binned to each subtile on the screen and rendered separately. [APX13] with
figure 6.3 has shown a system that describes this process and they devised a way to do
parallel frame tiled rendering. For this project, the parallelization paradigm could take a
shift for this pattern, which could potentially improve the memory access performance on
mobile devices with expensive memory space. [LK11] can also serve as a founding stone
by integrating their fine raster and coarse raster steps.
Rendering Large Scenes
For production graphics software, having a system that is optimized for many objects in
the scene is important. In that case this system will inevitably require 2 stage compute
shader. It can also include techniques such as frustum culling, meshlet hierarchical z-
culling [Zha+97], LOD switching for better performance.
Implications On Ray Tracing
This project or the topic of mesh shader centers around only rasterization. As we begin to
see the AAA games releasing recently or with Unreal Engine 5, full ray tracing is almost
achieved with de-noising and deep learning super sampling technology. To connect this
project to more photo-realistic ray-tracing, it is meaningful to note Micro-mesh ([MMT23]
now also hardware fixed for newest GPU) as a tessellation-centric solution. It builds a
pipeline that generates different layers of geometric maps in the compute pass. And for the
ray tracing, this has helped improve BVH (Bounding volume hierarchy) build performance

7
Conclusion
GPUs have evolved into very sophisticated, versatile machines, and many potentials exist
to optimize them to bring astonishing performance. Among those, software rasterization
with compute shaders in graphical systems comes into the spotlight. This project set out to
implement a compute shader rasterizer that utilizes the meshlet structure and aims to use
parallelization capability fully. The rasterizer achieves basic functionality such as projec-
tion, shading, depth/face culling, etc. Some tricks were used for optimization, including
cone culling and many other tips in GPU debugging/profiling. By the end, results have
shown that the compute shader can bring improved frame time compared to a configured
minimal fixed pipeline. This system now has fulfilled the project goal of building an MVP,
exploring GPU, and basic emulation of the mesh shading pipeline. While as in the future,
this system can be further customized for different fields of interest. Real-time shading
will continue to be an exciting field with respect to how developers organize and optimize
the pipeline and also, most importantly, how the hardware improves by generation.
