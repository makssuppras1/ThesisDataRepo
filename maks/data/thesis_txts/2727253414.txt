Abstract
Datalog has found uses in many different areas in the last few decades. An application area
will usually take a core Datalog engine and customize it to fit the needs of that specific area.
Such an engine, both the core part and the customizations, need to be able to be trusted to
give a correct output. If the engine has been defined and proven to satisfy its desired properties
in a proof assistant, such as Isabelle, then it can both be trusted to be correct and the formal-
ization itself can potentially be used to generate the executable code for the engine. To this
author’s knowledge, no Isabelle formalization of Datalog, which focuses on formalizing the core
semantics of Datalog, with an eventual goal of generating executable code, currently exist.
In this thesis, Datalog is formalized in the Isabelle proof assistant. The core datatypes and
definitions for Datalog are formalized. The Herbrand base of Datalog programs is formalized
and proven to be finite and a model for all well-formed and safe programs. Model-Theoretic
semantics are formalized and are proven to define the unique minimal model for all programs,
and a finite model for safe and well-formed programs. Fixpoint semantics are formalized and
the immediate consequence operator is proven to be monotone and to be the minimal model
for all programs at a fixpoint, fixpoint semantics are proven to be equivalent to model semantics
and the immediate consequence operator is shown to compute the unique minimal model after
a finite number of iterations for all safe and well-formed programs. Finally, further work based
on this formalization is suggested.

Introduction
Datalog [1] has in recent decades experienced a significant rise in interest. The growth of big
data and the need for efficient data processing and analysis have highlighted the advantages of
logic-based approaches, and advancements in Datalog optimization techniques and the devel-
opment of efficient evaluation algorithms have improved its scalability and performance, mak-
ing it possible to handle large-scale datasets [2]. The adoption of Datalog in fields like program
analysis, security, and artificial intelligence has further contributed to its growing popularity.
Datalog is a declarative logic programming language that is primarily used for querying and rea-
soning in database applications, deductive databases, and knowledge representation systems.
Some of its application areas are:
• Program Analysis: Datalog is increasingly used in program analysis for the purpose of
program understanding, bug detection, and optimization. It can be used to define rules
that capture program semantics, control flow, data dependencies, and other program-
specific information. Datalog queries can then be used to extract information about pro-
gram behavior, identify potential bugs or vulnerabilities, perform data flow analysis, detect
code duplication, and analyze program performance. An example of this is the Soufflé
framework [3], which is a Datalog engine focused on program analysis that compiles down
to parallel C++, and has been used to implement static analysis tools [4].
• Data Integration: Datalog can be used in integrating data from different databases, file
formats, or web services to provide a unified view of the data. It can be used to define
rules that express mappings, transformations, and relationships between different data
sources. The support for relational operations such as projection, selection, and join
also enables efficient data manipulation and filtering during the integration process, and
Datalog with Negation can extend this even further. An example of this is the INFOMIX
system [5], which is an information integration system, where query answering is reduced
to cautious reasoning on Datalog with Negation programs.
• Declarative Networking: Datalog can be used in declarative networking to express net-
work policies, define network configurations, and reason about network behavior. Net-
work policies, such as access control rules or routing decisions, can be defined using Dat-
alog rules, which capture the conditions and constraints of networks. Datalog’s support
for recursive computations is useful in scenarios involving dynamic network topologies,
where routing protocols or network optimizations may require iterative computations or
continuous monitoring of network state. Network Datalog (NDlog) [6] is a language for
declarative network specifications, which is rooted in Datalog.
What applications of Datalog usually have in common is that they take the core of the Datalog
language, and extend and customize it to fit the needs of the specific area. Extensions can
for example be the support of negation, inequalities or disjunction, and customizations can be
evaluation strategies, which are more efficient for that specific use area.
7
==PAGE:8==
To be able to put a Datalog engine to use in the real world, one needs to be able to trust that it
has been implemented correctly. One possible way of doing this is the approach most if not all
major software projects use of software testing, where a program is executed on some inputs
to see if it gives the expected outputs. This approach can be fairly simple to implement, since
one usually knows what output is expected from a given input, but if the tests used are not
broad enough, then edge cases will be missed and the software will be faulty to some degree.
Another approach is formal verification. In formal verification, the system is modeled using a
mathematical model. The properties of the system can then be formally specified and proven
using a formal logic. This approach provides a high level of confidence and guarantees about
the correctness of a system, based on the properties that have been specified. Proof assis-
tants are software tools that assist in the development and verification of such formal models
and proofs. They can automatically check the validity of proof steps, detect errors or incon-
sistencies, and guide users on further steps in proofs, and often they can be used to generate
executable code from the formal specification.
A larger formalization of Datalog has already been made in the Coq proof assistant [7]. In
this formalization [8], fixpoint semantics have been defined, and the positive engine of this
formalization has been proven to be sound and complete. The positive engine has then been
used to formalize the semantics of Stratified Datalog with Negation, and this has been proven
to be sound and complete, and to compute the unique minimal model.
A formalization of stratified Datalog has also been created in the Isabelle proof assistant [9]. In
this formalization [10] the evaluation function of the formalization is proven to compute a model
which is the minimal model. The formalization is then used to formalize the Bit-Vector Frame-
work [11], which is a framework that is used for static program analysis. In the formalization,
model-theoretic semantics are used to define the semantics of Stratified Datalog with Negation.
Fixpoint semantics and the finiteness of models is not something that is dealt with in this paper.
Fixpoint semantics are of interest, since they can be used to define efficient evaluation methods
[12] and the finiteness of models is important in the context of executability and code genera-
tion, since finite models can potentially be computed. To this author’s knowledge, no Isabelle
formalizations of Datalog, which focuses on formalizing the core semantics of Datalog, with an
eventual goal of generating executable code, currently exist.
The main goal of this project is to formalize Positive Datalog and Stratified Datalog with Negation
in the Isabelle proof assistant. The secondary goals is to formalize different evaluation methods
and to adjust the formalization such that it would be able to generate executable code. The work
done can be found in [13].

2
Isabelle
In this chapter, we give an overview of the Isabelle proof assistant and then give an example
of a structured proof in Isabelle.
Throughout this thesis, previously proven lemmas which are being used in ongoing proofs will
be denoted with a blue background, examples will be denoted with a white background, and
definitions and proofs in progress will be denoted with a grey background.
2.1
Isabelle Description
Isabelle is based on a higher-order logic known as Isabelle/HOL [14], which allows users to
reason about complex mathematical concepts. It supports a wide range of formal logic systems,
including first-order logic, higher-order logic, and various modal logics. Users can interact with
Isabelle through a command-line interface or a graphical user interface called Isabelle/jEdit.
The system provides a scripting language called Isar [15], which allows users to construct
structured proofs step-by-step, guiding the proof process and ensuring the correctness of each
step.
Isabelle/HOL is a large library of formalized mathematics, and includes foundational theories,
algebraic structures, analysis, and formalized proofs of various theorems. This library serves
as a foundation for further formal developments and proof automation. Isabelle also supports
code generation [16], allowing users to extract executable code from their formal definitions.
This facilitates the integration of formal methods with practical software development, since it
enables the generation of certified code.
Isabelle has multiple automated proof methods built-in. No complete proof method exists for
HOL, which means all the proof methods are incomplete to some degree. The ones relevant
to this formalization are:
• simp rewrites the first subgoal of a proof using the declared simplification rules. It can be
invoked on all subgoals of a proof with simp_all.
• auto combines the simplifier with classical reasoning and is applied to all subgoals, and
will prove all easy subgoals and possibly create new subgoals.
• blast is a classical tableau prover and is well-suited for complex logical goals. This
method is applied to only one subgoal.
• force applies the classical reasoner and the simplifier to one goal, and attempts to prove
that goal completely, otherwise it fails.
• metis is a resolution-based automated-theorem prover for first order logic with equality
[17]. This method will attempt to prove a goal using a given list of lemmas. This is the
most advanced automated proof method used in this formalization, however we use it
sparingly, since proofs using it can often be difficult to follow.
10
==PAGE:11==
HOL has some basic types that are similar to those of functional programming languages. The
types that are relevant in this formalization are:
• bool, truth values
• nat, the natural numbers
• list, the type of lists
• set, the type of sets
• ⇒, an infix operator which denotes function types
nat, list, and set each have a library of useful functions and lemmas regarding those functions,
which are useful in definitions and proofs. Sets have the usual notation and can be either finite
or infinite. Lists are always finite.
HOL supports formulas with the typical logical connectives from predicate logic, equality, and
the existential and universal quantifier. Isabelle also allows for lambda expressions, denoted
by standard notation for lambda expressions. The symbols ∧and =⇒also exist, and are part
of the Isabelle framework. These are logically equivalent to ∀and −→, but are instead used to
structure proofs and are preferred when a lemma needs to be used in another proof [18]. =⇒
separates the assumptions of a lemma from the conclusion and ∧indicates an arbitrary but
fixed variable in a lemma.
New types can be defined using the datatype command:
datatype tm = Const nat | Var nat
Here we have defined a datatype for terms, using the natural numbers as an identifier. | is used
to separate the two values that tm can have, and Const and Var are the constructors of the
datatype.
Functions can be defined using the primrec command:
primrec assign_tm :: ‹tm ⇒(nat ⇒nat) ⇒tm› (‹[[_]]tm_› [100,100]) where
‹[[Const n]]tmv = Const n› |
‹[[Var n]]tmv = Const (v n)›
primrec is a primitive form of recursion, where each recursive call peels off a datatype con-
structor from one of its arguments. Here we peel off the tm constructor. After the name of the
function (assign_tm) is the signature of the function. This is optional for functions where Isabelle
is able to infer the signature. After the signature is the mixfix declaration of the function. This
is simply used to make definitions and lemmas easier to read and has the same meaning as
using the name of the function. This function declaration will generate two simplification rules,
which are simply its defining equations, that are automatically added to the simplification rules
used by the simplifier.
If a function doesn’t have an argument that needs to be unpacked, and is non-recursive, then
it can be defined by the definition command, which is simply an abbreviated way of writing
11
==PAGE:12==
the body of the function:
definition ‹syms i ≡sym ` i›
Note here the use of ‘ which is the image of a set under a function. A definition will generate a
simplification rule, which is simply its function body, however this definition is not added to the
simplifier, and will need to be manually added in proofs.
2.2
Isar Proof Example
We now present an example of an Isar proof and explain the key parts of the structure of the
proof:
lemma ground_a_assign:
‹ground_a a =⇒
[[a]]av = a›
proof (induct a)
case (Pre n tms)
then show ?case
proof -
assume ‹ground_a (Pre n tms)›
then have ‹∀tm ∈set tms. is_const tm›
by simp
then have ‹∀tm ∈set tms. [[tm]]tmv = tm›
using const_assign_to_self
by simp
then have ‹map (λtm. [[tm]]tmv) tms = tms›
using map_idI
by metis
then show ‹[[Pre n tms]]av = Pre n tms›
by simp
qed
qed
We will go through this proof part by part.
The lemma command begins the proof and gives the lemma the name ground_a_assign, which
can be used to use the lemma in future proofs. Multiple different commands can be used to
start a proof, such as theorem and proposition. These all function the same, and only indicate
how important we think the lemma is.
lemma ground_a_assign:
Next we have the definition of the lemma, which is contained between ‹ and ›. =⇒separates
the assumption of the lemma ground_a a from the conclusion [[a]]av = a.
‹ground_a a =⇒
[[a]]av = a›
12
==PAGE:13==
Now the actual proof begins, using the proof command. This command can take some proof
method. Here we use induct, where we perform structual induction over the datatype of a.
Here we are simply proving that the lemmas holds for the possible values of the datatype,
which in this case is one value.
proof (induct a)
The induct method creates a case for each value of the datatype, which we need to prove. Pre
n tms is the ”unpacked” value of a, where the values of n and tms are universally quantified.
case (Pre n tms)
The then keyword pipes the previously proved fact into the next proof. What is piped here are
our assumptions for the unpacked a. The show keyword states that we are now proving the
overall goal of this part of the proof and ?case is the goal of this case.
then show ?case
We again use the proof command, but instead of giving a proof method, we use -, which tells
Isabelle to not alter the proof. Otherwise Isabelle will attempt to apply some relevant rule to the
goal, for example existential generalization, in the case of an existentially quantified formula.
proof -
Next we introduce the assumption of the proof, which is that ground_a holds for this value of
the datatype. Here the assumption is unnamed, since we only need it at the start of the proof,
but an assumption can also be named, if it is needed multiple times. This is also useful if a
lemma has multiple assumptions.
assume ‹ground_a (Pre n tms)›
We then show an intermediate lemma, which is indicated by the have keyword, using the simp
method. This intermediate lemma is, like the assumption, unnamed, but can be given a name
if needed multiple times in the proof.
then have ‹∀tm ∈set tms. is_const tm›
by simp
To show the next intermediate lemma, we need to use a previously proven lemma, which is
introduced by the using keyword and the name of the lemma const_assign_to_self. This lemma
is then automatically used by the simplifier.
then have ‹∀tm ∈set tms. [[tm]]tmv = tm›
using const_assign_to_self
by simp
After another intermediate step, we can prove the goal of the case, and we can end the proof
using qed, which closes off the body of the proof command.
13
==PAGE:14==
then show ‹[[Pre n tms]]av = Pre n tms›
by simp
qed
qed
This concludes this example proof. This proof is part of this formalization of Datalog and the
steps taken, and the motivation for the lemma, will be explained in the description of the for-
malization.
If a lemma is simple enough, then a structured proof isn’t needed, and the lemma can simply
be shown using relevant lemmas and a proof method:
lemma adom_p_finite:
‹finite p =⇒
finite (adom_p p)›
unfolding adom_p_def
using adom_r_finite
by simp
Here adom_p is a definition and we are using the unfolding command to replace adom_p
with the body of its definition.

3
Basic Definitions
A Datalog rule is of the form:
A0 ←A1, ...., An
where n ≥0 are atoms where no function symbols appear. The atom to the left of ←is the rule
head and the set of atoms to the right is the rule body. A Datalog rule is interpreted as, if all the
atoms of the body are true, then the head of the rule must also be true. A Datalog program is
a finite set of Datalog rules.
A ground atom is an atom where only constants occur in the atom’s terms. A ground rule is a
rule where the atoms of the body and head are ground, and can be obtained by replacing each
variable x in the rule with a constant, using a valuation function v. A valuation function maps
from the variables of the rule to the set of constants that appear in the Datalog program. The
ground instances of a rule are all the unique ground rules that can be obtained by applying all
possible valuations to the rule.
A fact is a ground rule where the body is empty:
p(t1, ..., tn) ←.
and can also be written as:
p(t1, ..., tn).
for simplicity’s sake.
Interpretations are sets of facts. An interpretation satisfies a ground rule if the atoms in the
body are not contained in the interpretation, or if the head of the ground rule is contained in
the interpretation. An interpretation that satisfies all ground instances of a rule is a model for
the rule, and an interpretation that is a model for all the rules of a program is a model for the
program. All Datalog programs have a model.
A safe Datalog rule is a rule where the variables of A0 appear in some atom in the body. A
safe Datalog program is a finite set of safe Datalog rules. The safety condition guarantees that
rules and programs have finite models.
In this chapter, we describe the basic data types and definition for our formalization of Datalog.
We also show some basic properties of our definitions, which will be useful for proofs in later
parts of the formalization.
3.1
Data Types
The syntax of the formalization is defined using three datatypes. The first datatype encodes
terms and uses natural numbers as identifiers for constants and variables:
datatype tm = Const nat | Var nat
15
==PAGE:16==
The second datatype encodes atoms and is defined as a natural number, which is the identifier
for the predicate symbol of the atom, and a list of terms, which are the arguments of the atom:
datatype atom = Pre nat ‹tm list›
Note that we do not enforce the arity of an atom in the datatype itself. Rather, we later make a
definition that specifies the well-formedness property of atoms.
The third datatype encodes rules and is defined as an atom, which is the head of the rule, and
a list of atoms, which is the body of the rule:
datatype rule = Rule atom ‹atom list›
The body of the rule could also have been defined as a set of atoms, rather than a list of atoms,
without needing to make larger changes in the current formalization. The list representation
was chosen, since it is easier to work if there is a need to evaluate the atoms in the body
sequentially, which could be the case in for example resolution, or if one wanted to incrementally
construct substitutions for a rule, such as was done in the Coq formalization of Datalog [8]. It
also guarantees that rules have a finite body, since lists are finite in Isabelle. In general, lists
are also more flexible than sets, since they can easily be converted into sets, while the opposite
is not true, although sets will often allow for simpler definition and proofs.
Interpretations are generally defined as sets of atoms and programs as sets of rules, and as
such they do not need dedicated data types. Similar arguments for choosing list over set, as
with the body of rules, could be made here.
Arity functions and valuation functions are simply total functions that map from the natural num-
bers to the natural numbers. Valuation functions being total functions does not completely
match our description of Datalog, since here we define them as mapping from the variables
of a rule to the set of constants contained in the program of the rule. This means that a more
correct definition would be as a partial function. A total function was chosen since it is easier
to work with and doesn’t affect our final result. We explain why in Section 3.4. An assessment
of the use of partial functions for both valuation and arity functions is made in Chapter 7.
Now for an example of our encoding. Take the Datalog program:
P(a).
Q(b,b).
R(c).
P(X) <- Q(X,b).
Q(X,Y) <- P(X),R(Y).
This would, using our data types, be encoded as:
{Rule (Pre 0 [Const 0]) [],
Rule (Pre 1 [Const 1, Const 1]) [],
Rule (Pre 2 [Const 2]) [],
Rule (Pre 0 [Var 0]) [Pre 1 [Var 0, Const 1]],
Rule (Pre 1 [Var 0, Var 1]) [Pre 0 [Var 0], Pre 2 [Var 1]]}
16
==PAGE:17==
Where the predicates are identified as follows: P=0, Q=1 and R=2.
3.2
Helper Functions
The formalization contains helper functions that are by themselves not especially interesting,
but are used in other definitions and proofs, like for example the function for the active domain
of a program or for the set of predicate symbols of a program. The helper functions also have
some trivial proofs associated them that are also only interesting for their use in other proofs.
One can refer to the theory files[13] for a complete description of the functions used in the
formalization.
3.3
Well-formedness and Safety
This section describes the definitions related to the well-formedness and safety of programs,
and proofs regarding their properties.
3.3.1
Definitions
A safe rule is defined as a rule where all the variables in the head of the rule also occur in the
body of the rule:
primrec safe_r where
‹safe_r (Rule h b) ←→vars_a h ⊆vars (set b)›
A safe program is defined as a program where all the rules in the program are safe:
definition ‹safe_p p ≡∀r ∈p. safe_r r›
A well-formed atom is an atom where the number of arguments in the list of terms is equal to
the arity of the predicate, determined by an arity function:
primrec wf_a where
‹wf_a (Pre n tms) ar ←→ar n = length tms›
A well-formed rule is a rule where the atoms in the head and the body of the rule are well-formed:
primrec wf_r where
‹wf_r (Rule h b) ar ←→wf_a h ar ∧(∀a ∈set b. wf_a a ar)›
In our definition for a well-formed program, we have the standard definition for well-formedness,
which states that all the rules are well-formed with respect to the arity function:
definition ‹wf_p p ar ≡finite p ∧p ̸= {} ∧(∀r ∈p. wf_r r ar)›
We also define it as the program being finite and non-empty. The finiteness condition is in-
cluded, since, for any real-world application, we are working with finite programs, and many of
the proofs that we are interested in require that the programs that we are working with are finite.
17
==PAGE:18==
Similarly with the non-emptiness condition, we are also generally working with programs that
have some content, and proofs regarding empty programs will in general not be very interesting.
A ground atom is defined as an atom where all the terms in the atom’s argument list are con-
stants:
primrec ground_a where
‹ground_a (Pre _ tms) ←→(∀tm ∈set tms. is_const tm)›
We also include a definition for well-formed interpretations, which states that all the atoms in
the interpretation should be well-formed, and that it should be finite:
definition ‹wf_i i ar ≡finite i ∧(∀a ∈i. ground_a a ∧wf_a a ar)›
It also states that all the atoms in the interpretation should be ground. This could be argued
to be closer related to the safety condition for programs, however the safety condition exists
to ensure that finite programs have finite models, and this same concern is not present for
interpretations that are not ground, since we aren’t using the interpretation to determine what
atoms are needed to satisfy the interpretation.
Furthermore, the textbook definition of an interpretation only includes constants [19, p.24] [20,
p.6], and not variables, whereas the definition of a rule does allow for variables that don’t occur
is the body of the rule to occur in the head of the rule, since this just makes the rule unsafe
and does not affect the well-formedness of the rule. This would make it correct to define an
interpretation containing non-ground atoms as not well-formed, since these are not allowed by
definition.

3.4
Satisfiability
The section contains the definitions related to satisfiability of programs and definitions related
to the minimal model of programs. It also contains general proofs related to satisfiability, and
assignment of terms and atoms.
3.4.1
Definitions
We first define the assignment of terms under a valuation v. Constants are always assigned to
themselves and variables are assigned to some constant using the given valuation:
primrec assign_tm :: ‹tm ⇒(nat ⇒nat) ⇒tm› (‹[[_]]tm_› [100,100]) where
‹[[Const n]]tmv = Const n› |
‹[[Var n]]tmv = Const (v n)›
Here we use the notation [[tm]]tmv. A variable assignment is defined as a function that assigns
each variable to some constant, and assigns each constant to itself, which is what this function
does, when given a valuation.
Next we define the assignment of atoms under a valuation v:
primrec assign :: ‹atom ⇒(nat ⇒nat) ⇒atom› (‹[[_]]a_› [100,100]) where
‹[[Pre n tms]]av = Pre n (map (λtm. [[tm]]tmv) tms)›
Here we use the notation [[a]]av.
We now define the satisfaction of a rule by an interpretation i. An interpretation satisfies a rule if
for all valuations v all the atoms of the body under valuation v are contained in the interpretation
i, then the head under valuation v is also contained in the interpretation:
primrec sat_rule :: ‹atom set ⇒rule ⇒bool› (infix ‹|=r› 100) where
‹i |=r Rule h b ←→(∀v. (∀a ∈set b. [[a]]av ∈i) −→[[h]]av ∈i)›
Here we use the notation i |=r p. This definition is equivalent to stating that the interpretation
satisfies all ground instances of the rule.
In this definition v is a total function, which as mentioned previously, doesn’t match our de-
scription of Datalog. Mapping from only the variables of a rule doesn’t make a difference from
mapping from all possible variables, since the valuation function will only be applied to the
variables of the rule. The minimal model of a program, which the semantics we consider are
given by, will only contain constants which are contained in the program (this will be proven in
Chapter 5). This means that, if we have a valuation where we map to some constant that is
not contained in the program, then the ground atom, which contains this constant, obtained by
applying this valuation to some atom, cannot be contained in the minimal model. If this atom
is from the rule of a body, this means that the minimal model will satisfy this ground instance
of the rule by having an atom of its body not contained in it. If this atom is the head of the rule,
and assigned body of the rule is contained in the interpretation, then the rule is unsafe, and we
are mostly interested in safe programs. This means that the ground head of the rule does not
need to be contained in the minimal model, and we then have the same minimal model as if this
19
==PAGE:20==
valuation was not a valuation that we considered, which means that our results aren’t affected
by not using a partial function.
An interpretation satisfies a program if it satisfies all the rules of the program:
definition sat_prog :: ‹atom set ⇒rule set ⇒bool› (infix ‹|=p› 100) where
‹i |=p p ≡∀r ∈p. i |=r r›
Here we use the notation i |=p p.
A minimal model of a program is defined as a model that satisfies the program and where no
proper subset of the model satisfies the program:
definition min_model :: ‹atom set ⇒rule set ⇒bool› (infix ‹|=min› 100) where
‹i |=min p ≡i |=p p ∧¬(∃m ⊂i. m |=p p)›
Here we use the notation i |=min p.
Next we define the unique minimal model for a program, using the definite description operator
THE, which is usually denoted as ι. This operator only returns a value if a unique solution exists,
otherwise an arbitrary value is returned. Another term for this is uniqueness quantification:
definition min :: ‹rule set ⇒atom set› where
‹min p ≡THE i. i |=min p›
An actual proof regarding the existence of a unique minimal model won’t be performed until
Chapter 5, since that is when we first have a definition for the unique minimal model of a
program. Proving the existence of a unique minimal model without a definition for the unique
minimal model would likely be difficult.

4
Herbrand Base
Let P be a Datalog program. The Herbrand universe HP of the program is the set of constants
contained in P. HP is also known as the active domain of P. The Herbrand base BP is the set
of ground atoms that can be constructed from the predicate symbols contained in P and the
constants in HP . The Herbrand base of a program is a model of the program, which we will be
proving in this chapter. A Herbrand interpretation is any subset of the Herbrand base.
In this chapter, we describe our definitions for the Herbrand base of a program and show rel-
evant properties. We want to use the Herbrand base to prove that there exists a finite model
for all well-formed and safe programs. This is an important property, since finite models can
be computed. This formalization is based on the description of the Herbrand base found in
[20, p.19], however here there is not provided a proof of the Herbrand base being a model for
the program. The proof for the satisfiability of the Herbrand base that we perform is based on
similar reasoning as the proof found in [19, p.280-281]. Here is not the Herbrand base that
the proof is made for, but for a subset of it, which is also a model, and therefore the same
arguments hold for the Herbrand base.
4.1
Definitions
We first define lists_len, which takes a set of some arbitrary type and a natural number, and
gives a set of lists, which is the set of all lists of length len that can be created from the elements
in i:
definition ‹lists_len i len ≡{tms. set tms ⊆i ∧length tms = len}›
For example, the input:
i = {a, b}, len = 2
gives the output:
{[a,a], [a,b], [b,a], [b,b]}
The definition sym_atoms takes a natural number, a set of terms and a natural number, and
gives a set of atoms, which is the set of all the atoms can be created with the predicate symbol
n and the term lists from lists_len:
definition ‹sym_atoms n i len ≡{Pre n tms | tms. tms ∈lists_len i len}›
For example, the input:
n = 0, i = {Const 0, Const 1}, len = 2
gives the output:
28
==PAGE:29==
{Pre 0 [Const 0, Const 0], Pre 0 [Const 0, Const 1],
Pre 0 [Const 1, Const 0], Pre 0 [Const 1, Const 1]}
The definition for the Herbrand base takes a set of rules and an arity function, and gives a set
of atoms, which is the Herbrand base of the program for the given arity function. syms_p gives
the set of all the predicate symbols of a program and adom_p gives the active domain program,
which is what is needed to create the Herbrand base:
definition herbrand :: ‹rule set ⇒(nat ⇒nat) ⇒atom set› (‹(|_|)_›) where
‹(|p|)ar≡∪n ∈syms_p p. sym_atoms n (adom_p p) (ar n)›
Here we use the notation (|p|)ar .
For example, the input:
p = {Rule (Pre 0 [Const 0]) [Pre 1 [Const 1, Const 0]]}, ar = (0 := 1, 1 := 2)
gives the output:
{Pre 0 [Const 0], Pre 0 [Const 1], Pre 1 [Const 0, Const 0],
Pre 1 [Const 0, Const 1], Pre 1 [Const 1, Const 0], Pre 1 [Const 1, Const 1]}
The definition does not require that the program is well-formed with respect to the given arity
function, however, if this is not the case, then it will per definition not be the Herbrand base of
the program, since each predicate is associated with a fixed arity, and an incorrect arity function
would create atoms with incorrect arities.
The distinction between whether the program is incorrect with respect to the arity, or the arity is
incorrect with respect to the program is not especially important, however, for all arity functions
there will exists programs which are well-formed with respect to the arity function. This will
not always be the case the other way around, since a program can contain atoms of the same
predicate with inconsistent arities. This would argue in favor of arity functions always being
correct.
4.2
Proofs
We want to prove two separate properties of (|p|)ar . The first is that (|p|)ar is finite for a
finite program, and the second is that it is a model for programs that are safe and well-formed,
with respect to the arity function given to it. We will also show some other useful properties of
(|p|)ar .

4.2.2
Herbrand Base Satisfiability
We now want to show that, given a program p that is safe and well-formed with respect to a
given arity ar, (|p|)ar is a model for the program.
We first need show an important property of (|p|)ar , which will motivate the steps taken in
the proof of the satisfiability of (|p|)ar , and which will be used in the proof. If an atom is well-
formed, ground, its predicate symbol is contained in the program, and the active domain of the
atom is a subset of the active domain of the program, then the atom is contained in (|p|)ar .
theorem wf_ground_a_in_herbrand:
‹wf_a a ar =⇒
ground_a a =⇒
sym a ∈syms_p p =⇒
adom_a a ⊆adom_p p =⇒
a ∈(|p|)ar›
unfolding herbrand_def sym_atoms_def lists_len_def
by (induct a) auto
This follows from the definition of (|p|)ar , and the definitions therein, which is why we are able
to simply unfold its definitions to prove the theorem. We use the symbols of the program, the
active domain of the program, and the arity function of the program to construct the atoms. We
also only use constants in the arguments of the atoms, which means that they will be ground.
This theorem can be used to prove that an atom is contained in (|p|)ar , which is what we will
be doing in the satisfiability proof.
Before starting the proof of the satisfiability (|p|)ar , we need to show two properties of its active
domain, which will require a longer proof. After this, the proof of the satisfiability of (|p|)ar will
be performed.
Active Domain Proofs
The first property we show is that the active domain of (|p|)ar is a subset of the active domain
of its program.
lemma adom_herbrand_adom_p:
‹adom (|p|)ar ⊆adom_p p›
unfolding herbrand_def sym_atoms_def lists_len_def adom_def
by auto

function could give all predicates arity 0, while the program could contain some predicate with

5
Model-Theoretic Semantics
A model for a program P is minimal if no proper subset of it is a model for the program. All
Datalog programs have a unique minimal model, which is equivalent to the intersection of all the
models of the program [20, p.20]. Model-theoretic semantics are given by the unique minimal
model of programs.
The minimal model is chosen since the model should not contained more atoms than necessary
to be a model for a program P. Then the model contains only the facts contained in the program,
and the facts that can be derived from the rules and facts in the program.
In this chapter, we describe our definitions for model-theoretic semantics and show relevant
properties of our definitions, most importantly that model-theoretic semantics define the unique
minimal model for programs and that this model is finite for safe and well-formed programs. We
also show other useful or interesting properties of model-theoretic semantics. This formalization
is based on the description of model-theoretic semantics found in [19, p.278-282].
5.1
Definitions
We define the intersection of all the models that satisfy the program p, which is the definition
given in [19, p.278-282].
definition solve_m :: ‹rule set ⇒atom set› (‹(|_|)›) where
‹(|p|) ≡∩{m. m |=p p}›
Here we use the notation (|p|) .
Note that we do not use the usual database perspective of model-theoretic semantics, where
we have some database instance i, containing a set of facts, and the program p, containing
the relations/rules that are used to generate new facts from the database instance like in [20,
p.19-20][19, p.278-282]. We simply treat facts as if they are rules with an empty body, since
this makes the formalization simpler, and is a valid view of facts.
The definition we use here performs unbounded quantification, since there is an infinite number
of models that satisfy a given program. Since we know that (|p|)ar is a model for all safe and
well-formed programs, then (|p|) must always be a subset of (|p|)ar for safe and well-formed
programs. This means that we can, instead of quantifying over all the models that are a model
for p, settle for just quantifying over all the subsets of (|p|)ar , which is a finite set of sets for a
finite program.
definition solve_m_herbrand :: ‹rule set ⇒(nat ⇒nat) ⇒atom set› (‹(|_|)_›)
where
‹(|p|)ar ≡∩{m. m ⊆(|p|)ar ∧m |=p p}›

6
Fixpoint Semantics
Fixpoint semantics stem from fixpoint theory [24]. Here we use an operator called the immedi-
ate consequence operator. The operator derives facts from interpretations, using the rules of a
Datalog program. The immediate consequence operator P for a program P and an interpre-
tation I is defined as:
P (I) = {v(A0) | A0 ←A1, ..., An is a rule in P and
for some valuation v v(Ai) ∈I for all 1 ≤i ≤n}
An interpretation I is a fixpoint of P if P (I) = I. The immediate consequence operator P
is monotone:
I ⊆J =⇒P (I) ⊆P (J)
From this, and the Knaster–Tarski theorem, it follows that P has a least fixpoint, which is
a fixpoint that is included in any other fixpoint [25]. A fixpoint can, for safe and well-formed
programs, be computed in a finite amount of steps by iteratively applying TP to an interpretation
I. The least fixpoint of a program can be computed by iteratively applying TP to the empty set,
and is equivalent to (|p|) .
In this chapter, we describe our definitions for fixpoint semantics and show that they define the
unique minimal model for programs, that they are equivalent to model-theoretic semantics, and
that it is possible to define a fixpoint evaluation method, which terminates in a finite number of
steps, using our definitions for fixpoint semantics. This formalization is based on the description
of fixpoint semantics found in [19, p.282-286].
6.1
Definitions
We define the immediate consequence operator for a Rule h b, given an interpretation i:
primrec imm_conseq_r :: ‹atom set ⇒rule ⇒atom set› (‹r(|_, _|)›) where
‹r(|i, Rule h b|) = {[[h]]av | v. ∀a ∈set b. [[a]]av ∈i}›
Here we use the notation r(|i, r|) .
We define the immediate consequence operator for a program p, given an interpretation i:
definition imm_conseq_p :: ‹atom set ⇒rule set ⇒atom set› (‹p(|_, _|)›) where
‹p(|i, p|) ≡∪r ∈p. r(|i, r|)›
Here we use the notation p(|i, p|) .
We define the function for iteratively applying p(|i, p|) :
52
==PAGE:53==
primrec imm_conseq_p_iter :: ‹nat ⇒atom set ⇒rule set ⇒atom set› (‹p_(|_, _|)›)
where
‹p0(|i, p|) = i› |
‹pSuc n(|i, p|) = pn(|p(|i, p|) ∪i, p|)›
Here we use the notation pn(|i, p|) . For our fixpoint semantics, we use a naive evaluation
method called the Jacobi method. In this evaluation method, the immediate consequences
of all the rules for an interpretation in a given iteration are computed and passed on to the
next iteration. An alternative naive method is the Gauss-Seidel method. Here, the immediate
consequences of a program are evaluated sequentially, and the immediate consequences of
one rule is added to the interpretation for the immediate consequences of the next rule. This
method generally has better convergence than the Jacobi method [26], which means that it
will usually reach a fixpoint in a fewer number of iterations, but it is still fairly inefficient [1].
The Jacobi method was chosen here, since it allows for a simpler definition, and showing that
the properties of the Jacobi method hold for other methods should only involve showing that
the methods are equivalent at a fixpoint. These evaluation methods are called naive, since
they compute every previously proven facts in each iteration, as opposed to the semi-naive
methods, which partially eliminate this inefficiency [27].
Note that we have defined iteration 0, where we apply the immediate consequence operator 0
times, as the input interpretation, where in literature only 1 or more iterations are considered.
This is a consequence of how primitive recursion has to be defined in Isabelle, since each
case of the recursion has to use a constructor of the datatype. The choice of outputting the
input interpretation in the base case might seem out of line with literature, but consider that the
empty set will generally be what is given to pn(|i, p|) initially, and in the cases where it is not,
the interpretation given will be the output of previous iterations, which we would want to include
in the output of the definition.
We then define the unique minimal model for a program p using fixpoint semantics. Here we
use card (min p) as the number of iterations for p(|i, p|) . This definition can also be called
the fixpoint of p(|i, p|) :
definition solve_f :: ‹rule set ⇒atom set› (‹pω(|_|)›) where
‹pω(|p|) = pcard (min p)(|{}, p|)›
Here we use the notation pω(|p|) . We can use the value of card (min p) for the number of
iterations since, if our definitions for fixpoint semantics are correct, then in every iteration of
pn(|i, p|) , we will either generate at least one new fact, and we should continue for at least one
more iteration, or we don’t generate a new fact, and we have the minimal model. This means
that we will have at most the amount of facts in the minimal model number of iterations. This is
similar to the argument used in [19, p.284]. Another possibility would be to defined the number
of iterations as the number of facts in the Herbrand base. Both arguments can be used to show
that model can be computed in a finite amount of iterations for safe and well-formed programs,
however the result using card (min p) can be said to be tighter, since the minimal model will
always be a subset of the Herbrand base, while the opposite is necessarily true.

This definition essentially requires that min p computes the model and is only really interesting

definition does in fact define the minimal model for the program, this means that it is possible to

6.2.3
Relation to Model-Theoretic Semantics
We now want to show that fixpoint semantics and model-theoretic semantics are equivalent.
The reasoning for this is that they should be equivalent, but also that it will allow us to easily
show properties that we have already shown for model-theoretic semantics for fixpoint seman-
tics, for example model minimality.
For our proofs related to this, we have the assumption that the input interpretation i to pn(|i, p|) is
a subset of (|p|) . This is opposed to having just it be equal to the empty set, which is generally
what we would give as input when we want to find the model using pn(|i, p|) . We do this, since
it can be easier to work with variables rather than constants in proofs. This also makes it easier
to relate fixpoint semantics to model-theoretic semantics.
Furthermore, it is a reasonable assumption to have i be a subset of (|p|) , both because
this also includes the empty set, but also because, if our definitions for fixpoint semantics are
correct, then we should compute the minimal model at a fixpoint, which means that our fixpoint
semantics would be equal to model-theoretic semantics. This would mean that in every iteration
of pn(|i, p|) , we operate on an interpretation which is a subset of (|p|) . This then makes
our proofs regarding pn(|i, p|) more general, since we don’t just consider the input for when
p(|i, p|) is first applied, but also for when it has been applied an amount of times.

7
Discussion
In this chapter we discuss some formalization work that was attempted and left unfinished, and
some possible future formalization work.
7.1
Proof-Theoretic Semantics
Another semantics that was planned to be formalized was proof-theoretic semantics[20, p.21-
24]. Proof-theoretic semantics defines the semantics of a Datalog program by the set of all
facts that can be proven using the facts and rules in a Datalog program. Here there are two
approaches. One is bottom-up, where we start with the ground atoms of the program and derive
new facts using the rules of the program, which is essentially the same as fixpoint semantics.
The other is top-down, where we start from some goal that we want to prove, and use the facts
and rules of the program to derive and prove sub-goals that we need to prove to prove the main
goal.
Proof-theoretic semantics are usually based on SLD resolution. Early on in the project, it was
decided that this would not be formalized in this project, since it is seemingly a large task in of
itself to define and prove the soundness and completeness of SLD Resolution, even when we
don’t have to consider formulas with function symbols. Proof-theoretic semantics are interesting
to define, since they allow one to search prove specific facts of a program, rather than computing
the whole model. Proof-theoretic semantics are also equivalent to model-theoretic and fixpoint
semantics, which would be an interesting property to show. This would likely be better suited
for a separate project, which could build upon what has been formalized in this project.
7.2
Executable Code
In this section we discuss the steps that would need to be taken in order to generate code from
the different semantics.
7.2.1
Model-Theoretic Semantics
The definition:
definition solve_m_herbrand :: ‹rule set ⇒(nat ⇒nat) ⇒atom set› (‹(|_|)_›)
where
‹(|p|)ar ≡∩{m. m ⊆(|p|)ar ∧m |=p p}›
is close to being able to generate executable code, since it, for safe and well-formed pro-
grams quantifies over a finite set (|p|)ar . However, there are two things that would need to
be changed in the formalization. The first thing is that valuation function we quantify over in
i |=r p is defined as a total function. Since it is defined as nat ⇒nat, it is an infinite type, and
as such the definition can’t be computed in an executable program, which means that code
can’t be generated. An alternative solution would be to use a partial function instead. Here, we
would only need to define its domain as the variables contained in a rule and the range as the
72
==PAGE:73==
constants contained in the program, and if we have a finite program, then we could compute
all the possible variable assignments. This is similar to how how substitutions are constructed
in [8].
The second thing that would need to be done is to define most definitions using a list of rules as
the program instead of a set of rules, and then use the corresponding list functions to define the
method. This is both because the Herbrand base will be infinite for infinite programs, and a list
is guaranteed to be finite, but also because Isabelle is not able to generate code for many of the
definitions used in the set definitions, even for a finite sets, since set functions are used. The
relevant set functions usually have some equivalent way of defining them using list functions,
so the definitions themselves might not be too difficult to create. It might also be possible to
not have to redo proofs, since it could be sufficient to show that the definitions for the rule lists
gives the same output as for rule sets. This should be trivial for functions related to the active
domain and predicate symbols of programs, and might be more challenging for the definition
of (|p|)ar itself.
It would likely also be necessary to compute an arity function for a program, since we need it
to compute the Herbrand base. As with the valuation function, we would need to use a partial
function instead of a total function, and we could attempt compute it by going though all the
rules in the program.
Model-theoretic semantics are fairly inefficient in computing the minimal model, since they need
to compute a lot of other models first. Generating executable code from model semantics might
not then be especially useful, but it could still be an interesting exercise.

Fixpoint Semantics
Fixpoint semantics are also close to be able to generate code. The definition:

which does compute the minimal model, but it is dependent on min p. If one wanted to create an
evaluation function that was independent of min p, then it would need to be defined as such that
it doesn’t stop applying p(|i, p|) until the output of an iteration is equal to the output of the pre-

Another approach would be to use a coinductive definition of the evaluation. Coinduction [28]
is a way to reason about infinite and potentially non-terminating processes and coinductive def-
initions specify the behavior or properties of the objects using predicates or rules. Since the
evaluation of a program is potentially infinite for unsafe programs, this way of defining an eval-
uation function is useful, since Isabelle is able to generate code from inductive predicates even
is they are potentially non-terminating. We would then be able to prove that our coinductive
definition does compute a model for safe and well-formed programs.
The While_Combinator library [29] of Isabelle could also potentially be used. This library de-
fines the while combinator, which can be used to iterate a function on some input until a stop
condition is met. Defining it this way does not require an immediate termination proof, however
proving properties of this definition will require some proofs based on the definition of the while
combinator, which might make it as troublesome as any other approach.
7.3
Formalization of Datalog with Negation
In Stratified Datalog with Negation, the bodies of rules consist of literals, and the same safety
and well-formedness conditions for programs apply. Here, programs are divided into a se-
quence of strata, where each strata is a partition of the predicate symbols contained in the
head of rules. A stratified program is evaluated strata by strata, where the model of the pre-
ceding stratas is given as input to the current strata. In a stratification, it must hold for each
strata that the predicate symbols of the heads of its rules do not appear negatively in itself or
a preceding strata. Each strata is then a semi-positive program, which is a Negative Datalog
program, where negation is only applied to base atoms. Semi-positive programs have a natural
translation into Positive Datalog, since the atoms where negation are applied can just be re-
placed with a new positive atom. This means that it is possible to evaluate a Stratified Datalog
with Negation program using a Positive Datalog engine, by transforming each of the strata into
a Positive Datalog program, and evaluating them using the Positive Datalog engine.
An attempt was made in this project to formalize Stratified Datalog with Negation. This proved
unfruitful for three main reasons.
The first reason is the complexity of transforming from Positive Datalog to Negative Datalog.
To evaluate a Negative Datalog program using the stratified semantics, the strata of a program
would first need to be translated to Positive Datalog. The model for this strata would then be
translated back to Negative Datalog, and it would then have to be proven that it was a model
for the Negative Datalog version of the strata. The actual mapping function could be defined
fairly simply, since it should just be some bijective function, that takes the predicate symbol of
the literal and its truth value, and gives the predicate symbol of the Positive Datalog predicate.
74
==PAGE:75==
The mapping from Positive to Negative Datalog can then be accomplished simply by inverting
the mapping. What proved difficult was reasoning in proofs when these transformation of the
program and of the model were taking place.
The second reason is the way models need to be handled. When the model for the previous
stratas has been computed, and is being input into the evaluation of the current strata, it will not
contain the negative atoms in the model i.e. the atoms where the positive atom is not contained
in the model. These need to be added somehow. One possibility is using the Herbrand base
to compute all the possible atoms of the program that constitutes the previous strata. If some
positive atom is contained in the Herbrand base, but not in the model, then the negative version
of the atom needs to be added to the model. This is similar to what was done in [8] but reasoning
about this proved difficult.
The third reason is how to input the model for the previous strata into the evaluation of the
current strata. As we’ve touched upon in the formalization for model-theoretic semantics, we’ve
only defined the evaluation method to take a program as an input. This means that, if we
wanted to include some preexisting model in the evaluation, we would either need to define a
new evaluation method, where our existing proofs for the most part wouldn’t hold for arbitrary
input models, or we would need to add the model to the program itself as rules with an empty
body. While the latter option is possible, it is still difficult to reason about. If the database view
had been taken for the semantics, it might’ve been easier to work with, since the separation of
facts and rules could make it easier to reason when evaluating a strata.

8
Conclusion
In this thesis, the core datatypes and definitions of Datalog have been formalized. The Herbrand
base of Datalog programs has been formalized and proven to be finite and a model for all well-
formed and safe programs. Model-Theoretic semantics have been formalized and proven to
define the unique minimal model for all programs, and a finite model for safe and well-formed
programs. Fixpoint semantics have been formalized and the immediate consequence operator
has been proven to be monotone and to be the unique minimal model for all programs at a
fixpoint, fixpoint semantics have been proven to be equivalent to model semantics and the
immediate consequence operator has been shown to compute the unique minimal model after
a finite number of iterations for all safe and well-formed programs.
Originally, it was planned to also have formalized Stratified Datalog with Negation, however this
proved more difficult than first anticipated, and this part of the formalization was abandoned in
favor of refining the existing formalization. The main difficulties faced here were grounded in
the complexity of proofs when needing to translate from Positive to Negative Datalog, the fact
that the negative atoms need to be added to the model of a previous strata and the fact that we
didn’t consider the formalization of Negative Datalog when formalizing the semantics of Positive
Datalog, which made inputting the model for a previous strata into the evaluation of a current
strata difficult.
It was also planned to formalize proof-theoretic semantics, however it was decided early on
that formalizing SLD resolution would require too much work, and not much effort was spent
on this part of the formalization.
The secondary goals of being able to generate executable code from the formalization and to
formalize different evaluation methods weren’t explored in depth, however some suggestions
as to how the formalization could be modified to be able to generate code have been made.
Possible further work based on this formalization could be:
• Negation: Datalog with Negation is more expressive than relational algebra, and has a
wide range of uses, which motivates the creation of a formalization of it. This formalization
has laid the groundwork for a formalization of Stratified Datalog with Negation that’s based
on translating strata from Negative to Positive Datalog and evaluating them using the
Positive Datalog engine.
• Executable Code: With some amount of work it would be possible to generate exe-
cutable code from the different semantics. This would be interesting, since it would make
it possible to use this verified code for some application.
• Proof-Theoretic Semantics: Datalog has three main semantics and we have formalized
two of them. Proof-theoretic semantics are of interest to define, both for completeness’
sake, but also because of the way proof-theoretic semantics can be used, where facts
that are not of interest are not computed.
76
==PAGE:77==
• Evaluation Methods: The evaluation method used for fixpoint semantics is naive and
very inefficient. Defining more efficient evaluation methods, and proving their equivalence
to naive methods, would be interesting in the context of code generation, both because
efficiency is desireable, but also to be able to benchmark the different methods.
