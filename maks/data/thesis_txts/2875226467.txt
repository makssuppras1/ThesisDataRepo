Abstract
KernelôÙmode rootkits pose a significant security challenge due to their ability to execute
with elevated privileges and evade conventional monitoring. Emerging techniques exploitôÙ
ing extended Berkeley Packet Filter (eBPF) subsystems and the Linux io_uring interface
further complicate detection by bypassing syscallôÙbased monitoring and tampering with
inôÙguest detection solutions.
This thesis presents a hypervisorôÙbased framework designed to provide a solid foundation
for rootkit detection and analysis. The framework inspects kernel structures directly from
the hypervisor to identify potential signs of compromise, such as modifications to system
call tables and control registers, persistence through eBPF and io_uring programs activity,
and direct manipulation of kernel objects (DKOM). CrossôÙview analysis with an inôÙguest
framework is proposed as a supplementary technique to strengthen detection.
The framework demonstrates ability, albeit with limitations, to detect both classical and
modern rootkit techniques. Although not a complete defense system, the contribution
of this work lies in establishing an easyôÙtoôÙuse and extensible platform for systematic
rootkit analysis. By formalizing indicators of compromise and providing reproducible inôÙ
trospection, the framework provides a basis for future research that focuses on machine
learningãbased detection methods once suitable datasets become available.

Introduction
1.1
Motivation
Although rootkits do not dominate the malware ecosystem in terms of prevalence, acôÙ
counting for only a small percentage of detected attacks [1], their advanced evasion
techniques and use by sophisticated attackers [2] create significant security concerns.
The discovery of Bootkitty [3] in 2024 marked the first UEFI bootkit targeting Linux sysôÙ
tems, breaking the longôÙheld assumption that such firmwareôÙlevel attacks were limited
to Windows environments.
Simultaneously, attackers are exploiting newly introduced
kernel subsystems like io_uring, a highôÙperformance asynchronous I/O interface, to
achieve kernelôÙlevel persistence while bypassing traditional security monitoring tools such
as Falco [4]. The Skidmap malware also highlights this trend, employing kernel rootkit
capabilities to hide cryptocurrency mining operations [5].
These emerging attack vectors reveal a critical gap in current Linux security monitoring.
While the Linux kernel is often considered more secure due to openôÙsource development,
new interfaces like eBPF [6] and io_uring [7] provide attackers with unexpected pathôÙ
ways to achieve kernelôÙlevel access. The situation is further complicated by the fact that
traditional inôÙguest monitoring solutions are vulnerable to tampering by the very rootkits
they aim to detect [8]. This vulnerability highlights the need to revisit hypervisorôÙbased deôÙ
tection approaches that can maintain highôÙprivilege observation points beyond the reach
of kernelôÙlevel threats.
Current monitoring frameworks such as DRAKVUF [9] and HVMI [10] offer generalôÙpurpose
forensics and introspection capabilities but lack the specialized focus needed for rootkit
analysis. Additionally, these frameworks often require substantial expertise to configôÙ
ure and extend, creating barriers for researchers. This gap motivates the development of
frameworks that combine specialization for contemporary rootkit threats with userôÙfriendly
design, enabling systematic analysis of evolving attack techniques without overwhelming
complexity.
1.2
Gap
The previous observations outline a clear problem: contemporary Linux kernel rootkits,
including those leveraging eBPF tracepoints and io_uring, can subvert or evade inôÙguest
monitors by tampering with ãuserôÙvisibleã state. Therefore, detection should rely on outôÙofôÙ
guest observation to obtain a trustworthy view of kernel memory and control flow. In this
work, crossôÙview analysis denotes a principled comparison between (i) inôÙguest observôÙ
ables (e.g., /proc listings, kernelôÙexported metadata) and (ii) hypervisorôÙderived ground
truth obtained via Virtual Machine Introspection (VMI). We further define rootkit attack inôÙ
dicators as operational signals of compromise at the kernel level (e.g., invariant violations
in dispatch tables or control registers) that can be collected with minimal invasion from
the hypervisor and we extend them with crossôÙview discrepancies.

1.4
Research Questions
Scope and assumptions. The hypervisor/virtual machine monitor (VMM) is trusted. Only
the guest OS may be compromised. The target platform is limited to Linux kernel 5.x and
tested solely on kernel 5.15.139 on x86_64. Observations are performed outôÙofôÙguest via
virtual machine introspection. Any inôÙguest agent is considered untrusted.
1. Hypervisor Introspection Which features is it feasible to collect from an infected
domain via a privileged domain using Virtual Machine Introspection (VMI) and what
is the complexity of such approach?
2. CrossôÙview detection feasibility. Can a hypervisorôÙresident introspection frameôÙ
work detect active kernelôÙmode rootkits on modern Linux by identifying crossôÙview
discrepancies between (i) inôÙguest observables (e.g., /proc views, kernelôÙexposed
metadata) and (ii) outôÙofôÙguest ground truth obtained via direct inspection of kernel
memory and structures?
3. Rootkit attack indicators (RAIs). Which rootkit attack indicators can be precisely
defined for modern Linux kernels and extracted nonôÙinvasively from the hypervisor?
Examples include, but are not limited to: integrity violations of the sys_call_table,
anomalous transitions of control registers (e.g., CR0.WP) or modelôÙspecific regisôÙ
ters (e.g., IA32_LSTAR), unexpected modifications of IDT entries, persistent eBPF
program/map attachments consistent with getdents64 tampering, io_uringôÙcentric
I/O sequences lacking corresponding syscall activity, and direct kernel object maôÙ
nipulation (DKOM) patterns such as task lists.

1.5.1
Machine Learning Based Approaches
Trace of the Times: Rootkit Detection through Temporal Anomalies in Kernel AcôÙ
tivity
by Landauer M. and Alton L. introduces a novel behaviorôÙbased detection method
that leverages machine learning to uncover temporal anomalies. The core principle of this
approach is that when a rootkit injects code to interfere with a system function, the exeôÙ
cution of that modified code block will inevitably take more time than the original function,
2
Verdict from Above: HypervisorôÙBased Linux Rootkit Analysis
==PAGE:15==
creating a measurable time delay. The methodology employs Extended Berkeley Packet
Filter (eBPF) probes to measure fineôÙgrained time stamps of functions within system calls.
Their approach analyzes the distributions of these function timings and applies statistical
tests to detect significant shifts that are not attributable to normal system variation. The
implementation is openôÙsource and it achieves high detection accuracy with an F1 score
of 98.7% across five scenarios with varying system states [11].
The reliability of the proposed method is heavily dependent on the stability of function timôÙ
ing distributions. The authors note that function runtimes can vary based on their context
of execution, which may lead to false positives. Moreover, the method is semiôÙsupervised,
relying on a baseline of measurements from a system assumed to be free of rootkits. Its
practical application requires an approach that can handle and adapt to natural system
variations and concept drift over time. A significant challenge for all learningôÙbased detecôÙ
tion methods is the lack of public, highôÙquality datasets for evaluating these approaches.
As a result, the approach was tested on a custom LKM ftrace rootkit, CARAXES [12].
Trusted Kernel Rootkit Detection (TKRD)
proposes a machine learningãdriven frameôÙ
work that leverages memory forensic analysis to extract malicious features from virtual
machine (VM) memory dumps. By training classifiers such as Random Forest, J48, PART,
and SVM, TKRD achieved its best performance with Random Forest, reporting an accuôÙ
racy of 0.986 and an AUC of 0.998. The main strength of TKRD lies in its trusted memory
acquisition mechanism, which collects VM memory from the hypervisor layer, rendering
the dumping process resistant to manipulation by kernelôÙmode rootkits.
However, its reliance on static forensic snapshots introduces a detection delay and may
miss highly transient or stealthy rootkits that do not leave persistent traces. Furthermore,
the training process is datasetôÙdependent, raising concerns regarding generalization to
unseen or obfuscated malware families. No openôÙsource implementation of TKRD has
been released, limiting reproducibility [13] .
VirtualizationôÙbased Kernel Rootkit Detection (VKRD)
adopts a different strategy,
leveraging hardwareôÙassisted virtualization and virtual machine introspection (VMI) to dyôÙ
namically monitor kernel modules. It constructs behavioral feature vectors that capture
code, data, and register access operations, and applies classifiers such as SVM, DeôÙ
cision Trees, Random Forest, and KNN to distinguish benign from malicious modules.
Implemented on the Xen hypervisor, VKRD demonstrated high detection effectiveness
in experiments against Windows kernel rootkits, with applicationôÙlevel benchmarks (e.g.,
ApacheBench, NTFS decompression) showing observable but acceptable slowdowns relôÙ
ative to native execution.
VKRDãs limitation lies primarily in the performance overhead introduced by continuous
monitoring, which scales with the frequency of kernel resource accesses, and the chalôÙ
lenges of handling obfuscated or highly adaptive rootkits. Similar to TKRD, no openôÙ
source code for VKRD is available [14].

is a hypervisorôÙlevel rootkit detection mechanism that reconstructs seôÙ

sistency checks among multiple data sources, such as process lists, file descriptors, and
network sockets, to expose anomalies introduced by kernelôÙmode rootkits. Implemented

on Xen 4.1, the prototype successfully detected the KBeast rootkit, which was able to

1.5.3
Forensics Based Tools
Volatility
is an openôÙsource memory forensics framework designed to extract runtime
artifacts from volatile memory images across multiple operating systems and architecôÙ
tures. Its extensible plugin system allows focused analyses of suspicious behavior and is
frequently employed in incident response and academic research [16].
In the repository of Volatility 3, several Linux plugins (some may be deprecated) are
explicitly categorized as malwareôÙfocused and provide coverage against kernelôÙmode
rootkit techniques. These include check_syscall and check_idt, which verify the inôÙ
tegrity of system call and interrupt descriptor tables to uncover hookôÙbased tampering,
check_modules, hidden_modules, and modxview, which detect inconsistencies or conôÙ
cealment of kernel modules, and netfilter together with check_afinfo, which examine
network filtering and protocol structures for unauthorized modifications. Additional rootkit
vectors are targeted through keyboard_notifiers and tty_check, which expose maliôÙ
cious input interception, and check_creds, which inspects altered credential structures.
Finally, the malfind plugin highlights anomalous executable memory regions that may
correspond to injected payloads. Collectively, these plugins provide systematic inspecôÙ
tion of common kernel attack surfaces.
ModXRef
is a recently introduced Volatility plugin by Nagy R., specifically for Linux kerôÙ
nel module analysis. This plugin adopts a crossôÙview detection methodology by collectôÙ
ing kernel module information from a total of seven distinct kernel data structures and
meticulously crossôÙreferencing them for inconsistencies. In an evaluation, this new plugin
successfully detected all 55 tested rootkits, including those that evaded Volatilityãs native
detection plugin by tampering with multiple kernel data structures simultaneously, such
as the module_list and module_kset. The authors provide memory dumps for future
analysis [17].
As a memory forensics tool, Volatility is designed for postôÙmortem analysis of a memory
dump, not for realôÙtime monitoring. This makes it a diagnostic tool for incident response
rather than a proactive defense mechanism. Furthermore, the effectiveness of any memôÙ
ory forensics tool, including Volatility, is dependent on the integrity and completeness of
the memory dump itself.
1.5.4
Virtual Machine Introspection (VMI)
Virtual Machine Introspection (VMI),
as a principle was first formalized by Garfinkel
and Rosenblum, who introduced the concept of moving intrusion detection outside of the
guest system and into the virtual machine monitor (VMM) [18]. In their architecture, the
hypervisor acts as an isolated and privileged observer of the guest operating system. The
key advantage of VMI lies in its strong isolation guarantees: attackers operating within
4

==PAGE:17==
the compromised guest cannot directly tamper with the monitoring logic. However, the
approach introduces a semantic gap, as raw memory pages and registers observed at
the hypervisor level must be reconstructed into meaningful operating system semantics
before they can be analyzed. In addition, hypervisorôÙlevel rootkits exist which invalidate
the approach groundtruth.
LibVMI
is the most widely adopted openôÙsource VMI library and evolved from the earôÙ
lier XenAccess project [19]. Written in C, with bindings available for other languages, it
provides hypervisorôÙagnostic primitives for reading and writing guest memory, accessing
processor registers, translating virtual to physical addresses, and setting up traps on speôÙ
cific events. Tools such as DRAKVUF, a popular agentless dynamic malware analysis
framework, directly builds on LibVMI to provide transparent monitoring of guest execuôÙ
tion. While LibVMI offers broad support across hypervisors including Xen and KVM (onôÙ
development), it was designed with correctness and flexibility in mind, not scalability on
multiôÙcore systems.
RapidVMI
extends the baseline provided by libraries such as LibVMI by introducing a
performanceôÙfocused, multiôÙcore aware active introspection framework [20]. Unlike purely
passive approaches, RapidVMI enables fineôÙgrained, synchronous event interception by
setting traps that redirect execution to the monitor. Its design explicitly addresses the
contention and synchronization bottlenecks of multiôÙvCPU environments by orchestrating
concurrent guest memory access more efficiently. Evaluation demonstrated that RapidôÙ
VMI achieves significantly lower overhead than existing VMI solutions, making it more
suitable for scenarios that demand highôÙfrequency monitoring. Nevertheless, RapidVMIãs

2.1
KernelôÙmode Rootkits
2.1.1
Taxonomy
KernelôÙmode rootkits represent one of the most sophisticated and challenging forms of
malware in cybersecurity due to their stealthy nature and ability to operate at the highest
privilege level within an operating system.
According to the framework established by Bravo and GarcûÙa [24], kernelôÙmode rootkits
traditionally take the form of a loadable kernel module which after being loaded, tamper the
execution path of system calls and modify critical inôÙkernel structures. In a recent survey
by Nadim and Lee [25], kernelôÙmode rootkits are malware capable of hiding their presôÙ
ence and obtain high privileges. By operating with kernelôÙlevel privileges, these rootkits
can interfere with fundamental OS security mechanisms, allowing them to hide malicious
processes, conceal loaded modules, obscure network traffic, and perform system manipôÙ
ulation with complete administrative access.
We expand the definition of kernelôÙmode to malware that encompasses not only traditional
LKMôÙbased rootkits but also modern threats that exploit kernel interfaces and subsystems
to perform highôÙprivilege operations. This taxonomy includes:
1. Traditional LKMôÙbased approaches which represent the classical model of kernel
rootkits, where malicious code is inserted into the kernel through the standard modôÙ
ule (.ko) loading mechanisms (e.g., insmod).
2. Kernel interface exploitation which encompasses leveraging legitimate kernel interôÙ
faces such as eBPF [26], io_uring [7], and ftrace [27] for malicious purposes.
3. A combination of the previous techniques.
We consider hypervisorôÙbased rootkits out of scope for this study, because our detecôÙ
tion framework explicitly assumes the hypervisor (VMM) is a trusted, uncompromised
entity. If the hypervisor itself is subverted (for example by HVM/VMôÙlevel rootkits such
as BluePill [28] or SubVirt [29]), outôÙofôÙguest monitoring and virtualôÙmachineôÙintrospection
guarantees are invalidated and the monitor can be controlled or evaded by the adversary.

System Call Table Modification:
System calls constitute the primary interface between
user space and the Linux kernel [38]. Consequently, the systemôÙcall dispatch table is a
highôÙvalue target for kernelôÙmode rootkits that wish to intercept or modify the behavior
of userôÙfacing applications stealthily. Rootkits typically modify entries in the syscall table
to redirect execution into attacker code, thereby implementing covert behaviors such as
filtering directory listings, blocking signals, and hiding connections [39, 40].
Common targets observed in the literature and in proofôÙofôÙconcept rootkits include the
directoryôÙenumeration calls getdents() and getdents64() (used to hide files and direcôÙ

and also trigger escalation on specific signal codes. Multiple analyses of real rootkits
document these exact hooks as a standard technique for hiding artifacts and controlling
process lifetime [41, 42, 43].

less trivial target. With the introduction of modern kernel versions, system call tables are
readôÙonly sections and kernel symbol visibility is often restricted. Moreover, KASLR (KerôÙ

location every boot making it harder for attackers to find the addresses of critical kernel
structures, was merged into the Linux kernel mainline in kernel version 3.14, released
on 30 March 2014 [44]. Historically, at x86_64 systems, attackers clear the CPU writeôÙ
protect (CR0.WP) bit or modify page table entries to obtain write access to the table. The
attacker must already be root, which is still possible from within kernel modules [45, 46].

the system call table and knowing the system call number they wish to target, assuming

IDT table modification:
The Interrupt Descriptor Table (IDT) is a critical data structure
that contains descriptors for interrupt and exception handlers in x86ã64 systems [59].
The IDT maps interrupt vectors to their corresponding handler addresses, governing how
the processor responds to hardware interrupts, software interrupts (via the INT instrucôÙ
tion), and processor exceptions. The location of the IDT is stored in the IDTR (Interrupt
Descriptor Table Register). The IDTR can be read at any privilege level using the SIDT
instruction, whereas updating it requires the privileged LIDT instruction. Executing LIDT
from user mode raises a #GP (General Protection Fault).
IDT centered kernelôÙmode attacks may either modify individual IDT descriptors or reloôÙ
cate the entire IDT and alter a copied table via LIDT. The latter evades integrity verifiers
that check only the canonical IDT address [60]. Because this technique requires careful
preservation of observable state and can be mitigated by comprehensive integrity meaôÙ
surement, documented instances and open proofôÙofôÙconcept code for IDT hooking are
relatively rare.

Lately, it has also been the case that rootkit repositories are banned (e.g, Reptile) [61]

2.3
Kernel Security
This section surveys the evolution of kernel security on Linux, focusing on features that
directly or indirectly mitigate kernelôÙmode rootkit techniques. We mostly organize by major
releases (2.2, 2.6, 3.x, 4.x, 5.x/6.x) and wherever appropriate, we connect mitigations to
common rootkit techniques (e.g., syscall table modification, symbol resolution).
The progression of kernel security features shown in Table 2.3 has a direct impact on the
effectiveness of known Linux rootkits (Table 2.2). Early rootkits such as AdoreôÙng, Rooty,
and JynKbeast, which relied on writable syscall tables and exported symbols, ceased to
function reliably once the syscall table was marked readôÙonly in 2.6.24 and unexported
in 2.6.18. Similarly, the introduction of KASLR in 3.14 invalidated the static address asôÙ
sumptions embedded in several proofôÙofôÙconcept projects (e.g., rickrolly, Toykit). Later
rootkits like Reptile and Suterusu adopt more resilient symbol resolution strategies by
using kallsyms_lookup_name().
Modern features such as the syscall ABI switch (4.17) and the unexporting of the funcôÙ
tion kallsyms_lookup_name() (5.7) break compatibility across a number of openôÙsource
rootkits. Nurupo, Reptile, and puszek, all of which previously used direct kallsyms_lookup
_name() calls, requires significant modifications to build on new kernels. By contrast,
rootkits that leverage eBPF (e.g., ebpfkit) remain viable on 4.xã5.x kernels, showing
that defensive measures against traditional syscall hooking and symbol resolution have
pushed adversaries towards exploiting other surfaces.
Another noteworthy effort, that was marked in 6.x kernel series is the introduction of
Rust as a supported language for inôÙtree kernel code. The RustôÙforôÙLinux effort provides
memoryôÙsafety guarantees absent in C, particularly eliminating entire classes of vulneraôÙ
bilities such as useôÙafterôÙfree, buffer overflows, and iterator invalidation. Rust is explicitly
designed to enforce memory and concurrency safety in its ãsafeã subset through its ownôÙ
ership and type system, thereby reducing the reliability of traditional memory corruption
primitives [81].
Last but not least, one important dimension of kernel protection is hardening, which arises
not from new hardware features or compiler protections alone, but from configurationôÙ
level hardening (i.e. enabling or disabling kernel build options that mitigate known atôÙ
tack vectors). The Timesys presentation ãLinux Kernel Security: Overview of Security
Features and Hardeningã highlights how many kernel config flags have inherent secuôÙ
rity implications: for example, disabling unused subsystems (CONFIG_USB_USBNET=n,
CONFIG_DEVMEM=n), enabling stack protector (CONFIG_STACKPROTECTOR=y), hardened userôÙ
copy checks(CONFIG_HARDENED_USERCOPY=y), module signature enforcement (CONFIG_M

2.4
Virtualization
2.4.1
Background
Definition:
Virtualization is a fundamental technology going back to the 1960s with IBM
CP/CMS [93]. The central idea is that it enables the creation and management of virôÙ
tual machines (VMs) by abstracting physical hardware resources and allowing multiple
operating systems to run simultaneously under a single physical machine. At its birth,
virtualization involves the implementation and insertion of a software layer, called a VirôÙ
tual Machine Monitor (VMM), which is interchangeably referred in literature as hypervisor,
[94] and lies between the physical hardware and the operating systems [95].
Motivation:
The primary motivation of virtualization in multiprocessor systems has been
workload consolidation, which has been driven by reducing computer center costs, enôÙ
abling clustering with message passing for distributed computing, facilitating system miôÙ
gration and isolation through partitioning, reducing system downtime during failures through
improved fault isolation, and supporting heterogeneous system environments on unified
hardware platforms [96].
Software Virtualization Caveats:
Before hardware extensions, x86 virtualization relied
on software techniques such as binary translation or ring deprivileging, where a guest
kernel was forced to run at a less privileged ring (e.g., ring 1 or even ring 3) instead of
ring 0 (see Figure 2.2 for ring topology). Sensitive instructions were trapped and emulated
by the hypervisor, while nonôÙsensitive instructions executed natively. Although effective,
this approach suffered from high overhead and semantic inconsistencies, since many inôÙ
structions behaved differently outside ring 0 and required complex binary rewriting. As
virtualization was applied to increasingly demanding multiprocessor systems, these limiôÙ
tations became more pronounced. Even with optimizations such as binary translation and
shadow paging, the complexity and inefficiency of purely software solutions motivated the
introduction of hardware virtualization extensions, namely Intel VTôÙx and AMDôÙV [97, 98].

frameworks like LibVMI, RapidVMI, DRAKVUF, and HVMI.

Setup Details
3.3.1
Hypervisor / Framework Selection
Initially, KVM [105] was selected as the target hypervisor due to prior familiarity and its
prevalence in cloud infrastructure. Surveys indicate that approximately 94% of users in
OpenStack platform prefer KVM as their primary hypervisor [106]. This widespread adopôÙ
tion suggested that a KVMôÙbased detection framework would have broader applicability
to realôÙworld cloud security scenarios.
During the KVMI (LibVMI for KVM) [107] setup, we encountered several practical chalôÙ
lenges. Building a kernel with KVMI support was harder than expected because it relies
on nonôÙmainlined patches and has limited documentation for recent kernels. As a result,
multiple build attempts failed to produce a working KVMIôÙenabled kernel.
Beyond the build failures, the key factor in transitioning to Xen was the availability of
DRAKVUF. Running Xen as the hypervisor made it possible to integrate DRAKVUF, which
we used as a complementary mechanism to support and validate virtual machine introôÙ
spection.
3.3.2
Storage Management
The storage subsystem utilizes LVM [108] for flexible volume management in the dom0.
LVM provides several advantages including dynamic volume resizing, snapshot capabiliôÙ
ties, and efficient space utilization.
While LVM snapshots offer copyôÙonôÙwrite efficiency, our implementation revealed chalôÙ
lenges with snapshot merging during restoration operations. The complexity of managing
snapshot chains and potential corruption during merge operations led to the adoption of
disk cloning using dd [109] for state preservation.
LVM combined with xl tools offer the capability to also attach a LVM at an active domain.
This capability proved useful for transferring files. Moreover, memory snapshots (dumps)
are possible via LibVMI vmi-dump-memory [110].
3.3.3
Security and Isolation
Minimal security isolation was implemented using Uncomplicated Firewall (ufw) [111] in
the dom0. The principle of least privilege [112] was applied by restricting network access
to SSH connections only. We also followed bestôÙpractices for SSH accessing [113].
3.3.4
Display and Remote Access
The implementation employs VNC (Virtual Network Computing) for guest VM display,
proxied through noVNC for browserôÙbased access [114]. The SSH tunnel approach proôÙ
vides several benefits: encryption of VNC traffic which is by default unencrypted, authenôÙ
tication through SSH key infrastructure and avoidance of additional port exposure.
The Xen toolstack also provides native console access to guest VMs through the xl
console command, which establishes a TTYôÙbased connection to the guestãs serial conôÙ
sole. This approach offers a lightweight alternative to VNCôÙbased graphical access. Both
guest and host require additional configuration to support TTY communication.
3.3.5
Network Configuration
Network connectivity for guest VMs is provided through Linux bridge networking rather
than NAT. This decision was influenced by both practical implementation considerations
and technical constraints of the underlying hardware, and the underlying issue of enslavôÙ
ing a wireless interface to a Linux bridge due to driver and protocol restrictions [115].

Diamorphine
Diamorphine is a classic LKM rootkit that disables CR0.WP bit to write to syscall table
entries, performs self hiding, hides processes and files with specific prefix and intercepts
signals for privilege escalation actions.
Attack Techniques
Diamorphine employs classic rootkit techniques:
1. SelfôÙhides by removing itself from the module list (less advanced than bds_lkm_ftrace).

Windows function hooking mechanisms (e.g., inline patching,

which intercept file operations at the VFS layer, are not currently implemented but repreôÙ

Modern kernel interfaces (extension)
Contemporary Linux rootkits exploit modern
kernel interfaces beyond traditional hooking mechanisms. We examine eBPF artifacts by
enumerating loaded programs and maps, correlating them with their owning processes to
detect orphaned or suspicious eBPF objects that may persist after their controlling process
terminates. We also inspect io_uring structures by verifying that the completion queue

manipulation) and by detecting unusual worker threads (iou-wrk* and iou-sqp*) that

6
Conclusion
This study set out to determine whether a hypervisorôÙbased framework can reliably obôÙ
serve indicators of kernelôÙmode rootkits from outside a potentially compromised Linux
guest and expand those indicators through crossôÙview analysis.
First of all, regarding which features are feasible to collect and the practical complexity
of doing so from the hypervisor, the prototype successfully captured a broad set of state
and event signals that are meaningful for kernelôÙmode compromise. These included inôÙ
tegrity of the system call table and IDT, ftrace hooks, module lists, and selected MSR
and CR registers on the state side, along with precise events for syscallôÙtable writes, CR0
and MSR writes, and interrupts on kprobe attachment, eBPF tracepoint registration, and
io_uring ring activity on the event side. The full table of features is illustrated in Section
4. Table 4.1.
Secondly, concerning whether crossôÙview comparison adds detection value, the DiscrepôÙ
ancy Checker consistently showed mismatches between inôÙguest reported and hypervisor
reported observations in an infection state. This was most evident in the eBPF processôÙ
hiding and bds_lkm_ftrace scenarios, where the hidden userôÙspace program was absent
from guest enumeration yet present in hypervisor observations. We also confirmed addiôÙ
tional crossôÙview indicators, specifically those related to established TCP connections. In
the bds_lkm_ftrace scenario, this was demonstrated through a reverse shell established
via nc between Clueless-Attacker and Clueless-Admin, which remained hidden from
inôÙguest observation while still detectable by the hypervisor.
Last but not least, regarding the definition and extraction of rootkit attack indicators suitôÙ
able for modern kernels, this work formalized these indicators in relation to the prior study
by Nadim et al., extending their classification across both classical and contemporary
kernel surfaces.
Across these results, the framework met the studyãs objectives. It demonstrated that
VMI can collect kernel evidence of kernelôÙmode rootkits with precision and that crossôÙ
view comparison exposes inconsistencies concealed from inôÙguest observation, and that
a wellôÙdefined set of rootkit attack indicators (RAIs) can be extracted for both classic techôÙ
niques and modern interfaces. The contributions include a reproducible architecture, a
structured configuration and output format, and minimal yet effective crossôÙview checks
that together provide a basis for future dataset construction and learningôÙbased methods
when appropriate data become available. Detailed limitations and potential directions,
including automation of offset discovery and broader platform coverage, are discussed in
the following sections.

6.1
Limitations
The Rhadamanthus framework has several limitations that affect its applicability and comôÙ
pleteness. The STATE_KERNEL_MODULE_LIST callback does not handle RBôÙtree traversal
for modules references, missing detached modules from the global linked list that still reôÙ
side in memory. Static signature analysis through directory string matching (STATE_DIR_ST
RING_MATCHING) and CPUID instruction event callback is not implemented.
Memory offset handling requires manual configuration using the pahole [130] tool to exôÙ
tract structure offsets for the target kernel version. These offsets must be manually popuôÙ
lated in the offsets library file, creating a maintenance burden and limiting portability across
different kernel versions. The framework is designed and tested primarily for Linux kernel
5.x (5.15.0ôÙ139) versions, and assumptions about kernel data structure layouts may not
hold for older (4.x) or newer (6.x) kernels without modification. The specific kernel version
(5.15.0ôÙ139) was chosen as it supports both eBPF and io_uring interfaces.
The eBPF and io_uring monitoring capabilities provide general indicators of usage rather
than distinguishing between legitimate and malicious applications. These callbacks report
the presence of eBPF programs and io_uring instances but do not perform behavioral
analysis to determine intent. Meaningful detection requires correlation with context from
event callbacks and temporal analysis to identify suspicious patterns.
The framework does not implement a comprehensive scoring or risk assessment system.
While certain callbacks mark findings as ãsuspiciousã based on heuristics (e.g., function
pointers outside kernel text sections), there is no unified mechanism for aggregating these
signals into an overall system compromise score. The framework provides raw monitoring
data and basic indicators but requires external analysis tools or manual interpretation to
draw conclusions about security state.
Testing coverage is limited. While core state and event callbacks have been validated in
experimental environments, integration testing across different kernel configurations and
comprehensive unit testing of edge cases remain incomplete. The framework requires furôÙ
ther validation in diverse deployment scenarios to ensure reliability and accuracy across
varying system configurations and kernel versions.
Despite its name suggesting limited capabilities, CluelessôÙAdmin has certain advantages
over LibVMIôÙbased approaches in specific scenarios. Accessing kernel debug information
through ftrace and debugfs interfaces is often more straightforward than implementing
equivalent functionality through VMI, as these interfaces are designed for introspection
and provide structured access to kernel internals. The tool can leverage existing kernel
debugging infrastructure without requiring complex address translation or memory parsôÙ
ing.
However, the name accurately reflects the toolãs vulnerability to rootkit manipulation. As
an inôÙguest monitoring solution, CluelessôÙAdmin operates within the potentially comproôÙ
mised trust boundary and can be deceived by kernelôÙmode rootkits that hook system calls
or manipulate kernel data structures. In realôÙworld deployments, far more advanced seôÙ
curity tools would typically be deployed, including kernel integrity monitoring systems,
securityôÙenhanced Linux distributions, and dedicated endpoint detection and response
platforms. The CluelessôÙAdmin serves primarily as a baseline comparison point for demonôÙ
strating the value of outôÙofôÙband VMIôÙbased monitoring rather than as a production secuôÙ
rity tool.
The CluelessôÙAttacker component is intentionally limited in scope and functionality. It
implements only specific attack scenarios designed to demonstrate the detection capabilôÙ

6.2
Future Work
The frameworkãs performance overhead has not been systematically measured. Future
work should benchmark introspection latency, memory overhead, and throughput under
various monitoring configurations. While preliminary observations suggest acceptable
overhead, quantitative evaluation is needed to characterize performance at scale and
identify optimization opportunities. Comparative evaluation against existing VMI systems
is challenging due to limited availability of rootkitôÙspecific detection frameworks, though
behavioral comparison with generalôÙpurpose systems like DRAKVUF could provide baseôÙ
line context.
Several callbacks remain unimplemented. The CPUID instruction event callback could
detect virtualizationôÙaware rootkits that query CPUID to identify VMI presence and alôÙ
ter behavior accordingly. Directory string matching (STATE_DIR_STRING_MATCHING) could
monitor filesystem modifications contextôÙagnostically, detecting suspicious file or direcôÙ
tory creation patterns regardless of the specific syscall interface used. Obtaining further
context for EBPF and io_uring callbacks would also strengthen detection.
The current development workflow, manual offset extraction with pahole, custom structure
traversal implementation, perôÙkernel configuration is not scalable. Future work should
investigate automation strategies such as leveraging kernel BTF (BPF Type Format) for
runtime offset resolution.
Complex data structures beyond simple linked lists (RBôÙtrees, XArrays, hash tables) reôÙ
quire extensive testing and perôÙstructure library support, increasing maintenance burden
substantially. Development effort could be reduced by building or finding a reusable data
structure traversal library that handles common kernel container types generically, simiôÙ
lar to how LibVMI provides generic address translation primitives. Such a library would
separate traversal logic from structureôÙspecific offset handling, improving maintainability
and reducing duplication across callbacks.
Both Rhadamanthus and DRAKVUF are built on LibVMI, suggesting potential for integraôÙ
tion or callback sharing. Adapting Rhadamanthus callbacks as DRAKVUF plugins could
leverage DRAKVUFãs mature infrastructure while contributing rootkitôÙspecific detection
logic. This would reduce duplication and allow crossôÙvalidation of detection approaches.
The frameworkãs ultimate purpose is rootkit detection in realistic environments. However,
obtaining labeled datasets of rootkit activity in production systems is extremely challengôÙ
ing due to legal, ethical, and availability constraints. Future work should explore safe data
collection methodologies, potentially through partnerships with security organizations or
controlled deployment in research networks. Validating detection accuracy requires diôÙ
verse rootkit samples and realistic baseline system behavior.
The frameworkãs architecture, transparent monitoring from outside the guest, aligns well
with highôÙinteraction honeypot requirements. Deploying instrumented VMs as honeypots
could collect rootkit behaviors in controlled environments where attackers believe they
have achieved full system compromise. The VMI layer would record attack techniques,
persistence mechanisms, and postôÙexploitation activities without alerting the attacker.
This application warrants investigation into related work on VMIôÙbased honeypots and
sandbox systems, and could address the data collection challenges noted above while
advancing understanding of realôÙworld rootkit behaviors.
