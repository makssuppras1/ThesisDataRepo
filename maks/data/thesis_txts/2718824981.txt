Abstract
Digital media, such as videos, images, and audio, are widely consumed on the internet.
The process of creating these files can be time­consuming. Developing an application that
allows users to create digital media files with long­running processes is not trivial. It re­
quires distributed system design skills that most application developers do not commonly
possess.
Backend as a Service (BaaS) products have been successful in helping developers ac­
celerate the development of simple web applications by providing pre­built backend func­
tionality and abstracting away the complexity of server management. However, no BaaS
product currently can help developers bootstrap applications that support long­running
processes to create media files.
This thesis presents the foundations of a BaaS software that allows for quickly developing
cloud­agnostic web applications that allow users to initiate, monitor, and evaluate long­
running processes that produce media files as output. The BaaS software manages long­
running processes and their output media files. It is distributed and horizontally scalable,
even for a small user base.

Introduction
Digital media like videos, images and audio are widely consumed on the internet all over
the world. The creation of digital media files is a long process compared to the manipula­
tion of simple data like text or numbers in a database. The development of an application
that allows users to create digital media files is not trivial. A system that manages the
long­running processes and their output media files has to be distributed and scalable,
even for a small userbase. A developer who wants to build this application has to have
system integration skills that are not commonly possessed by application developers.
Simple applications have the opportunity to be built on top of existing and successful
Backend as a Service (BaaS) products. There is no BaaS product to help developers
bootstrap an application that creates and stores media files using long­running processes.
This thesis describes the design and implementation of a BaaS software allowing the easy
development of cloud­agnostic web applications that allow users to initiate, monitor and
evaluate long­running processes that produce media files as output.
1.1
Problem domain
The creation of digital videos, images and audio (hereafter referred to as media files) can
not be considered an instantaneous action. The process of creating a media file involves
capturing or generating raw data, processing that data to add effects or manipulate the
content in some way, and then rendering the final product, which can take a significant
amount of time. The creation process is also dependent on the complexity of the media
file being produced and the computational resources available for producing it. Therefore,
the creation of media files should be considered a long­running process.
This theis defines a long­running process as the execution of a program that is expected to
take over 10 seconds to produce some output. 10 seconds is the limit after which feedback
is required to tell the user to take a break or perform other tasks while they wait for the
process to finish. Otherwise they would assume that the application is stuck or faulty [1].
Examples of long­running processes include the rendering of a video, a machine learning
inference calculation, digesting and processing vast amounts of data, or transcribing the
video of a very long lecture. This thesis focuses on long­running processes with timespans
of minutes, hours or even days, depending on the complexity of the media file being
produced and the computational resources available for producing it.
In contrast to media file creation, database operations on primitive data, such as integers
or strings, are relatively simple operations that do not require the same level of computa­
tional resources. As a result, they are considered to be nearly instantaneous [2]. Simple
applications that mainly use database operations on primitive data in their business logic
are referred to as CRUD (create, read, update, delete) applications [3].
This thesis introduces the concept of Long­running Process Media Applications (LPMAs).
An LPMA is an application which allows its users to initiate long­running processes that
produce some media file as output. This thesis focuses on cloud­native LPMAs that run
their long­running processes server­side.

1.2
Problem statement
The development of an LPMA is not trivial. Long­running processes consume a signifi­
cant portion of the computational resources on a server, like CPU and memory. Because
of this, the deployed application’s runtime could be overwhelmed even by a small amount
of users. Therefore, the application should be designed in a way that the long­running
processes scale separately from the business logic of the application. The development
and maintenance of a distributed system requires a different set of software engineer­
ing skills, than the development and maintenance of a simple CRUD application. The
skills required to build a distributed system require more training to acquire, than the skill
required to build CRUD applications.
Simple CRUD applications can be built upon a wide variety of third­party Backend as a
Service products that provide a layer of abstraction over common problems. Authentica­
tion and authorization, user management, databases, file storage, analytics and hosting
are examples of modern web application components that can be outsourced to third­
party services. BaaS solutions reduce amount of code required to be written by develop­
ers and also reduce the responsibility of managing cloud resources. This enables product
companies to focus on their core business [4]. The availability and ease of use of these
backing services has led to the creation of new digital products and services that would
have otherwise struggled to enter the market with their idea [5].
The lack of a widely adopted service to help developers bootstrap LPMAs is a barrier
to entry for new ideas. This barrier is felt mainly by developers at the beginning of their
carreers who do not possess the skills to build and maintain distributed systems.
Most of the BaaS products available to build CRUD applications are closed­source and
can not be deployed to private clouds or air­gapped environments. A notable exception is
Supabase, which is discussed further in Section 2.2.1: Supabase. If a BaaS for developing
LPMAs is ever realized, it should be open­source and cloud­agnostic, allowing developers
to build LPMAs on any cloud provider’s platform. A self­hosted backend also allows local
test environments to be set up for no additional infrastructure cost.
We at Colossyan [6] would have greatly benefited from using a BaaS that we could have
built our product on. Colossyan is a video editing application that provides deepfake
actors to narrate educational and marketing videos. Due to the lack of an established
BaaS for building applications that uses long­running processes, the software engineers
at Colossyan had to build their product by provisioning and integrating multiple third­party
services. A hosted backend for LPMAs would have decreased the development costs
and development time to create Colossyan.
1.3
Solution
This thesis introduces the Long­running Process Media Application Utility (LPMAU). LP­
MAU is a Backend as a Service that allows beginner developers to build Long­running
Process Media Applications. LPMAU focuses on managing the lifecycle of user­initiated
long­running processes and their output media files. A detailed specification of LPMAU
can be found in Chapter 3: Specification. LPMAU is open­source and cloud­agnostic,
which means it can be self­hosted or deployed on any cloud provider’s platform.
LPMAU has the potential to accelerate the development of existing and new LPMAs by
taking on the responsibility of managing the long­running processes and their outputs in
the cloud. The benefit of integrating third­party services into an application can be seen
by the application’s end­users as a more stable product, accelerated innovation and more

Background
This chapter provides a brief overview of the concepts and technologies that are relevant
to this thesis. Several potential technologies are discussed, along with their relevance
to the problem outlined in Section 1.2: Problem statement. Furthermore, this chapter ex­
plains why an LPMA has to be designed as a distributed system. The chapter concludes
with a literature review to explore long­running process management in the scientific com­

2.1
CRUD and LPMA
This section explains the concepts of CRUD and LPMA, and highlights the differences
between them.
2.1.1
CRUD
CRUD applications are simple applications that mainly use database operations on primi­
tive data in their business logic [3]. The acronym CRUD refers to the create, read, update
and delete operations, which are the four basic operations of persistent storage [7]. These
four operations are the core of many applications. CRUD applications are used in many
different contexts, such as web applications, mobile applications and desktop applica­
tions.
2.1.2
LPMA
This thesis defines the concept of a Long­running Process Media Application (LPMA). An
LPMA is an extension of a CRUD application with the following differentiating character­
istics:
• Long­running processes
An LPMA includes long­running processes as a part of its core user flow. Contrary
to common CRUD applications, in LPMAs the create operation does not instantly
yield a created resources, but rather a promise that the resource will be created by
the application over a period of time. In the context of LPMAs, the resources created
in these applications are media files such as images, videos or audio files.
The concept of a long­running process is defined in Section 1.1: Problem domain.
• Media file storage
An LPMA is guaranteed to have a media file storage system, which is used to store
media files such as images, videos, and audio files produced by the long­running
processes of the application.
2.2
Concepts
This section provides a brief overview of the concepts that are relevant to this thesis.
Some of the following concepts come close to solving the problem outlined in Section 1.2: Prob­
lem statement, but lack some key feature or component to be considered a complete
solution.
2.2.1
Backend as a Service (BaaS)
Backend as a service (BaaS) or ”mobile backend as a service” (MBaaS), is a service that
enables web and mobile app developers to connect their apps to backend cloud storage
and access features such as user management, push notifications, and integration with
social media services. This is achieved through the use of custom software development

kits (SDKs) and application programming interfaces (APIs). BaaS is a relatively new de­
velopment in cloud computing, with most BaaS companies founded after 2011. In 2012,
the global BaaS market was valued at approximately $216.5 million [8].
This section lists some of the most popular BaaS providers and their features. BaaS
products in this section are evaluated based on how well they fit the requirements of
building an LPMA.
AWS Amplify
Amazon Web Services (AWS) Amplify is a set of tools and services that enables mobile
and frontend web developers to build and deploy secure, scalable full stack applications.
AWS Amplify provides a declarative interface across a variety of cloud services, including
ones for authentication, storage, serverless APIs, and analytics. With Amplify, developers
can build and deploy applications with features such as authentication, serverless back­
ends, and real­time data synchronization, without having to worry about the underlying
cloud infrastructure. AWS Amplify is suitable for a wide range of applications, from sim­
ple web or mobile applications to complex enterprise­grade applications. It is designed
to be flexible, scalable, and easy to use, so that developers can focus on building their
application’s main logic and not on managing the cloud infrastructure [9].
AWS Amplify provides most of the functionality required by an LPMA. However, a com­
ponent to run long­running processes is missing from the services it offers. AWS Lambda
functions, which can be used with AWS Amplify, are not suitable to run long­running pro­
cesses of arbitrary running time. Therefore, AWS Amplify alone can not be used to de­
velop an LPMA.
Firebase
Firebase is a mobile and web application development platform developed by Firebase,
Inc. in 2011, then acquired by Google in 2014. Firebase provides a set of tools and ser­
vices that enables developers to build, scale, and grow their applications more efficiently.
Firebase offers a range of services for building mobile and web applications, including:
• Real­time database: a cloud­hosted NoSQL database that allows developers to
store and sync data across multiple devices in real­time.
• Cloud Firestore: a NoSQL document database that simplifies storing, syncing, and
querying data for mobile and web applications.
• Cloud Functions: a serverless platform for building and connecting cloud services.
• Authentication: a set of tools and services for handling user authentication and au­
thorization.
• Hosting: a fully managed static file hosting service for hosting web applications and
static files.
Firebase is suitable for building a wide range of applications, from simple prototypes to
complex, data­driven applications. It is designed to be flexible, scalable, and easy to use
[10].
Similarly to AWS Amplify, Firebase also lacks a service to host long­running processes.
Cloud Functions, which can be used with Firebase, are not suitable to run long­running
processes of arbitrary running time. To conclude, Firebase alone can not be used to
develop an LPMA.

Supabase
Supabase is an open­source alternative to Firebase. It includes a real­time database,
user authentication and authorization, block storage and hosting for serverless functions.
Supabase can be initialized as a self­hosted service, or it can be rented for a monthly
fee as a managed BaaS. Both the self­hosted and the managed versions offer the same
functionalities and features. The self­hosted version has official support for deployment
with Docker. The open source community has created other deployment options, that are
not officially supported by Supabase. This includes deployment guides for Kubernetes,
Terraform and Traefik.
Supabase is continuously growing in popularity, which means that there is demand for
cloud­agnostic backend as a service solutions. Figure 2.1 shows the weekly downloads
of the Supabase JavaScript SDK from the Node Package Manager (NPM) over the last
year.
Figure 2.1: Weekly downloads of the Supabase JavaScript SDK from NPM.
Supabase currently offers SDKs in JavaScript and Flutter, a Management API in the form
of a well­documented REST API, and a CLI for managing the service [11].
Since the features of Supabase are a subset of the features offered by Firebase, Su­
pabase also lacks a service to run long­running processes of arbitrary length. Edge Func­
tions, which can be used with Supabase, are not suitable to run the types of long­running
processes discussed in Section 1.1: Problem domain. Therefore, Supabase alone can
not be used to develop an LPMA. However, supabase is the only BaaS discussed so far
that offers a self­hosted deployment option. This means that an extension of Supabase
could become a cloud­agnostic LPMA framework.
2.2.2
Serverless computing
Serverless functions are a type of cloud computing service that allows developers to run
small pieces of code in response to events. They are often used to implement small
pieces of functionality in web applications, such as sending an email when a user signs
up, or to implement a webhook that is triggered when a user clicks a button. Serverless
functions are often used in conjunction with other cloud services, such as databases,
storage, and messaging services. They are popular among developers because they
allow them to quickly and easily implement small pieces of functionality without having
to manage the underlying infrastructure. Most BaaS providers offer serverless functions
in their feature set. Examples of serverless function environments are AWS Lambda,
Google Cloud Functions, Azure Functions, Cloudflare Workers and Supabase Functions
[12].

AWS Lambda is a serverless computing service provided by Amazon Web Services (AWS).

2.2.3
Batch computing
Batch computing involves running large numbers of jobs, typically concurrently or at least
in a parallel manner, with a focus on high throughput rather than low latency. Batch com­
puting is used to process large data sets, perform long­running calculations, or execute
tasks that do not require real­time processing.
In a batch computing system, jobs are typically submitted in groups, or batches, and are
executed in a first­come, first­served manner. Batch computing systems are designed to
maximize the utilization of the underlying compute resources, and may use techniques
such as job scheduling and resource allocation to optimize performance.
Batch computing is often contrasted with interactive computing, in which users interact
with a computing system in real­time, and with streaming computing, in which data is
processed as it is received, in a continuous stream. Batch computing is well­suited to
many workloads, including scientific simulations, data analysis and processing, and other
types of compute­intensive tasks [15].
Even though batch computing would be the ideal environment for the execution of long­
running processes, it can only serve as a potential component of a BaaS that supports
LPMAs. A batch computing environment does not include a file storage system by default,
meaning a service to store and access the outputs of the long­running processes needs
to be provisioned and maintained by the application developer. Another disadvantage of
batch computing environments is that they are only provided by major cloud providers.
This means that an application that uses a batch computing environment would not be
cloud­agnostic.

2.2.4
Master­worker architecture
The master­worker architecture is a common approach for distributing tasks among mul­
tiple processes in order to complete them more quickly. In this architecture, a central
”master” process manages the overall process and assigns specific tasks to ”worker” pro­
cesses. The worker processes then complete these tasks independently and in parallel.
The master process is responsible for coordinating the entire process and is typically the
source of information about the overall progress of the computation. The worker pro­
cesses typically do not communicate with each other or use the master to exchange mes­
sages [18]. In another variation of the master­worker architecture, the worker processes
are responsible for starting processes by asking the master for a job.
Master­worker differs from batch computing in that the worker processes execute more
than one job. In batch computing, a worker process is only responsible for the completion
of one single job, after which the process terminates. In master­worker, a worker process
can ask the master for new jobs after it has finished executing a job. The worker processes
only stop if they crash or they are manually restarted. Therefore, there are no cold starts
and there are no delays in execution between jobs.
The master­worker architecture could be used to design an application that supports long­
running processes. The master component would delegate the long­running processes to
the worker components. The workers would execute the processes and send the results to
the master. The master then could store the results and notify the users of the completion
of their process.

2.3
Decoupling the long­running process execution
This section makes the case for decoupling the long­running process execution from the
rest of the business logic in LPMAs. An application that does not decouple long­running
processes from its main execution flow can run into scalability issues even with a small
amount of users. The following experiments show how long­running processes can cause
a bottleneck in applications.
Two experiments have been performed to measure video rendering times. Both experi­
ments measure the rendering time of videos. The experiments use the Node.js video ren­
dering package Remotion [22], written in TypeScript, to render videos. Remotion uses a
Chromium browser instance to produce each frame of the video. The videos are slideshow
videos consisting of only images. The videos in the experiments are rendered at 30 frames
per second.
A program was developed to produce one video of variable length. The program takes
an input property of imageCount, which specifies the amount of images the video should

==PAGE:20==
consist of. Each image increases the length of the video by 5 seconds. For each data
point in the charts, three measurements were taken and averaged to yield the value of the
data point. The programs have been executed before measuring the execution times to
cache all files required by the videos into memory. The measured execution times include
the following tasks:
• Initialization of the program. This includes the startup of the Node.js runtime, the
startup of the Chromium instance required by Remotion, and any initial checks for
external dependencies required by the program. The execution time of this task is
identical in all video rendering attempts.
• Producing the individual frames of the video. Due to the longer videos having more
frames, this task takes longer for longer videos.
• Combining the individual frames into a single video. This task also takes longer for
longer videos.
• Writing the video file to the filesystem. This task takes longer for longer videos.
Both experiments measure the rendering time of the videos with the Linux time command.
The Linux time command takes another command as an argument and measures the
execution time of that command. The time command returns with three measurements:
• The real time it took to execute the command, also referred to as the wall clock time.
The real time is the time that elapsed between the start and end of the command.
This may include any time taken by other processes and the time spent waiting for
them to be complete.
• The user time it took to execute the command. This is the CPU time spent executing
instructions of the calling process.
• The system time it took to execute the command. This is CPU time spent executing
inside the kernel while performing tasks on behalf of the calling process.
The experiments take the system time into account as a conservative estimate of the real
computation cycles spent on rendering the video.
The experiments were conducted on a mid­range laptop with an Intel i5­8250U processor
and 8 GB of RAM. The processor has 4 cores and a total of 8 threads. The operating
system the experiments were conducted on is Fedora Linux 37.

2.3.3
Conclusion
The results of the experiments in the previous sections can be generalized to any comput­
ing environment and any long­running process. Processors with more cores and higher
clock speeds suffer the same scaling issues for longer videos. Changing the input prop­
erties of a video rendering process, like the resolution of the video or the number of video
tracks, only extends the video rendering time, accelerating the scaling issue.
Running video rendering processes in parallel on a single processor is also not viable.
The second experiment shows that the overhead of context switching between execution
branches accumulates, causing the rendering times to increase with the number of parallel
processes. The only way to reduce the overhead is to execute the processes sequentially.
The parallel execution model of the second experiment also has the drawback of only
supporting video rendering engines that are optimized to be run in parallel. Remotion is a
popular video rendering engine for programmatically creating videos using web technolo­
gies like HTML and CSS, but it suffers the disadvantage of relying on a browser instance
to produce each frame of a video.
Based on the results of the experiments, the execution of the long­running process must
be decoupled from the rest of the business logic in LPMAs to reduce performance bottle­
necks in application backends.

SlideApp specification
SlideApp is a web application that allows users to create slideshow videos from images
and music. The process of creating a video with SlideApp involves selecting images and
a music track from the web, then arranging the images in a desired order. Once the user
is satisfied with the setup of the video, SlideApp can be instructed to render the video.
The rendered videos are saved into cloud storage to be downloaded by the user later.
SlideApp is designed to be used by video editors of all skill levels, including those with no

professional video editing software such as Adobe Premiere Pro, Final Cut Pro or Sony

videos in their browsers.
The strength of SlideApp lies its simplicity and portability. One of the key benefits of
SlideApp is its accessibility from any device with a web browser, without the need for
installation. This means that users can access and use the application from any location,
as long as they have an internet connection. Additionally, since SlideApp does not rely
on local computing resources such as the client’s CPU or memory, it is able to provide
a consistent and reliable user experience on any device. This makes SlideApp an ideal
solution for users who need a simple and convenient way to create and edit slideshow
videos.

SlideApp should be cloud­agnostic, meaning that its back­end should be deployable on
any cloud provider’s platform. This requirement prevents the use of proprietary third­party
services that are tightly coupled to a cloud provider like AWS, Azure or CloudFlare.

Scalability
SlideApp should be able to support hundreds of paralell video rendering processes. Scal­
ing should be defined in SlideApp’s backend configuration and should be easily modifiable
to handle heavy loads.
Minimum delay
The time delay between a video rendering request and the start of a video rendering

the network is stable. This requirement prevents the use of computing environments that
provision computing resources for a noticeable amount of time.
3.2
Generalization
The requirements of SlideApp can be generalized to any application that includes long­
running processes in its core user flow. This generalization serves as a guide to designing
a BaaS that allows the creation and development of LPMAs.
In the case of non­functional requirements, the requirements defined in Section 3.1.3: Non­
functional requirements are reasonable for any LPMA to ensure optimal user experience.
3.2.1
Glossary
The following list of terms will be used in the specification, design and implementation of
a BaaS that helps developers create LPMAs.
Application is a publicly accessible piece of software.
User is a person who uses the application.
Developer is a person or group of people who creates the application.
Resource is a media file that is the output of some long­running process.
Job is a user­initiated request to create a resource.
A job has a status property of
”queued”, ”processing”, ”finished” or ”failed”. A finished job has a reference to the re­
source it created.
Creation is the process during which a resource is created. In the context of LPMAs,
creation is a long­running process.
3.2.2
Actions
The actions defined for videos in section Section 3.1.2: Actions can be defined for any
resource that is the product of a long­running process. The following is a list of actions
that a BaaS for creating LPMAs should provide.
Get jobs
The get jobs action returns a list of jobs initiated by the user. User­defined metadata
attached to the jobs is also returned.
Submit job
The submit job action instructs the application to create a resource. The input to this
action is list of user­defined values that define a resource. The action creates a new job
with a status of ”queued”. Once the creation can be started, the status of the job changes
to ”processing”. When the creation finishes, a reference to the created resource is added
to the job and the job’s status changes to ”finished”. If an error occurs during creation, the
status of the job will change to ”failed”.
Delete job
This action deletes a job and its referenced resource. The action can only be called on
jobs with a status of ”finished” or ”failed”.

To ensure that LPMAU is built on a solid foundation, it uses a number of established open­

as part of its normal operation [30].
Minio
Minio is a high­performance, self­hosted object storage server, compatible with the Ama­
zon S3 API. Similarly to Amazon S3, Minio stores files in its systems as objects in buckets.
Minio is Kubernetes­native, making it an ideal candidate for object storage in a private
cloud. The features of Minio include high availability, active data replication, encryption, a

For the purposes of this thesis, Minio is deployed as a container in a Deployment. In
a live production environment, Minio would be deployed with high availability and safety

Minio is used by the LPMAU backend to store and access the media files generated by

Minio was chosen for LPMAU because it is the most well­known self­hostable object stor­

backing service was made to ensure that any software product built on LPMAU is scalable
even in private cloud systems.
RabbitMQ
RabbitMQ is an open­source message broker that implements the Advanced Message
Queuing Protocol (AMQP). The features of RabbitMQ include high availability with dis­
tributed deployments, monitoring, message streaming, access control, and computing
resource management. RabbitMQ can be extended using plugins to support additional
features and protocols [32].
This thesis deploys RabbitMQ as a single container in a Deployment. A live production
deployment of RabbitMQ would follow the deployment guide described in RabbitMQ’s
documentation to achieve data consistency and queue persistence.
LPMAU uses RabbitMQ to propagate job-start messages between the LPMAU backend
and the LPMAU worker. A job-start message indicates that a job has been submitted
and is ready to be started by the worker. Each worker instance connects to the RabbitMQ
deployment and listens for messages on the job-start queue. When a message is added
to the queue, one of the workers picks up the message and executes the job. A message
contains information about the job, such as the job ID and the user input for the job. The
LPMAU Backend’s Queue component is responsible for communicating with RabbitMQ.
RabbitMQ was chosen for LPMAU because it is the most widely used message broker
in the cloud. The propagation of messages and a queue system is delegated to Rab­
bitMQ, instead of using an in­memory queue in the LPMAU backend. This decision was
made to potentially support hundreds of concurrent workers and thousands of concurrent
messages.
PostgreSQL
PostgreSQL is an open­source object­relational database management system (DBMS).
PostgreSQL is ACID­compliant and it is compatible with all major operating systems.
Many web and mobile applications have chosen PostgreSQL as their primary database
[33].
PostgreSQL is deployed in this thesis as a single container in a Deployment. A live deploy­
ment of a PostgreSQL database would involve a persistent volume claim from a storage
class in the cluster and multiple instances of the PostgreSQL container.
PostgreSQL is used in LPMAU to store records of the jobs initiated by users and managed
by LPMAU. The LPMAU Backend’s Database component is responsible for communicat­
ing with the PostgreSQL deployment.
The reason for choosing PostgreSQL for LPMAU is because it is an established open­
source database management system that scales well. PostgreSQL also integrates well

This section describes the implementation of the components of the LPMAU Backend, as

The LPMAU backend is a Node.js application written in TypeScript. The application uses

A developer could take two approaches to develop a proof­of­concept LPMA:
• Use a single cloud provider’s services. This method has the advantage of easier
integration of services, because a cloud provider’s different services are designed
to be compatible with each other. Another advantage of this method is that only one
cloud account is created and maintained. However, this method has the disadvan­

• Integrate a variety of third­party hosted services. With this method, the developer
could leverage the free tier of any hosted service to develop a prototype. However,
this method may suffer from incompatibility issues between services.
Both methods are explored in this validation. For each method, a comprehensive plan is
constructed to build a simple LPMA. The first method will use AWS as the single cloud
provider.
The second method will use the following collection of separate third­party
hosted services: Vercel [40] for hosting a web client, Railway [41] for hosting a worker,
CloudAMQP [42] for a message queue and Supabase [11] for data and object storage.
For the purpose of the comparison, LPMAU is imagined as a hosted third­party service
with a free plan.
The plans are described as tasks to be completed by the developer to develop an LPMA.
The perceived complexity of the tasks will be taken into account when comparing the
methods.

in LPMAU. I would also appreciate some more advanced monitoring and observabil­
ity in LPMAU.
Conclusion
The interviews revealed that software engineers have mixed opinions about the initial in­
terface of LPMAU. Subject A would use LPMAU to build their LPMA. Subject B would
only use LPMAU to develop a basic prototype of their LPMA. After validating the proto­
type, subject B would switch to more flexible, established frameworks. Both interviewees
expressed an interest in the idea of a self­hosted backend solution.
Suggestions to improve LPMAU include a dashboard, monitoring, alerting and debug­
ging capabilities, checkpointing, and the ability to suspend and resume long­running pro­
cesses. These suggestions are further discussed in Chapter 7: Future work.
6.2.2
Comparison to other methods
This section details the plans to build an LPMA using the two methods described in the
previous section, as well as a plan involving LPMAU. The findings are summarized in the
conclusion at the end of this section.
LPMA built on the AWS stack
The plan to build an LPMA using the AWS stack is described in the following steps:
1. Create an AWS account.
2. Create an AWS Amplify project and use the Amplify dashboard to create a CRUD
backend the long­running jobs.
3. Develop the worker program and package it as a Docker container. Upload the
container image to AWS ECR.
4. Configure a compute environment, a job queue and a job definition in AWS Batch.
Set the job definition to use the container image of the worker program developed
in the previous step.
5. Create an AWS Lambda function to call submitJob on AWS Batch upon creating a
job in the AWS Amplify backend.
6. Create an AWS Lambda function to collect the results of the jobs. Use the file stor­
age functionality provided by AWS Amplify to store the resulting media files.

7. Develop and deploy a web client that uses the Amplify SDK to query, initiate and
delete jobs.
LPMA built on a variety of services
The plan to build an LPMA using a variety of services is described in the following steps:
1. Create an account for each of the dependent services: Vercel, Railway, CloudAMQP
and Supabase.
2. Create projects in each of the services.
3. Provision a message queue in CloudAMQP.
4. Provision a database and an object storage in Supabase.
5. Develop and deploy a worker program on Railway that connects to the CloudAMQP
message queue and the Supabase datastores.
6. Develop and deploy a web client on Vercel that can publish messages to the CloudAMQP
message queue and also query the Supabase datastores.
LPMA built with LPMAU
The plan to build an LPMA using LPMAU is described in the following steps:
1. Create an account for the hosted LPMAU service.
2. Create a new LPMA project and provision an LPMAU instance.
3. Develop and deploy a worker program that connects to the LPMAU instance using
the LPMAU SDK.
4. Develop and deploy a web client that calls LPMAU upon user actions using the
LPMAU SDK.
Conclusion
It is clear that all plans involve the development and deployment of a web client, as well
as a worker program, which encapsulate the core business logic of the application. We
consider these two steps as having the same complexity across plans. We also consider
that creating cloud accounts and provisioning cloud resources on web dashboards have
the same complexity across all cloud providers and resources.
The first plan has the disadvantage of having to develop backend logic to handle the
initiations and the results of the long­running jobs. This logic is common to all LPMAs and
it is not directly related to the core business logic of the application. This disadvantage is
not present in the plan using LPMAU.
The disadvantage of the second plan is that it requires the creation and maintenance of
multiple cloud accounts and projects. This can fragment a developer’s attention and adds
extra complexity to further feature developments, fixes and migrations. This disadvantage
is not present in the plan using LPMAU.
To conclude, LPMAU saves the developer from the overhead of provisioning, integrating
and maintaining multiple cloud services. It also provides a unified interface for the de­
veloper to interact with the backend of their application. This makes LPMAU a superior
method for developing LPMAs.

7
Future work
This chapter lists some ideas of how LPMAU could be extended in the future. With these
extensions implemented, LPMAU could be used in a wider range of applications. LPMAU
could also enter the market as a commercial BaaS product.
7.1
Access control
Access control is the concept of limiting operations that a user can perform in a com­
puter system. The responsibility of access control is to prevent the unauthorized use of
resources in systems [43].
In CRUD systems, access control limits which of the four operations (create, read, update
and delete) can a user perform on a resource.
Access control was out of the scope of this thesis for the presented prototype of LPMAU.
However, in a production environment, access control is required to prevent misuse.
Access control can prevent a user from submitting too many jobs that overwhelm a sys­
tem. It can also prevent a user from accessing other users’ jobs and result files.
A potential implementation of an access control mechanism for LPMAU would be an ab­
straction of the Row­Level Security (RLS) system of Postgres databases, also provided by
Supabase. RLS restricts the rows that can be queried, modified or deleted on a per­user
basis. RLS works by applying policies to database tables that are specific to commands or
roles. An RLS policy can be created, altered or dropped. RLS policies require a Boolean
expression that is evaluated for each row before the user’s query. To apply an RLS policy
to a database table, the ALTER TABLE SQL command must be used. Superusers in the
database have the ability to bypass or force RLS policies [44].
7.2
Security
Security is the concept of protecting data and systems from unauthorized access. Given
that LPMAU has multiple software components running on a network, it is essential that
every component and their communications are protected.
Security measures for LPMAU would include the following:
• LPMAU’s job database must be protected from unauthorized access.
• LPMAU’s job queue component must be protected from unauthorized insertion or
deletion of messages.
• LPMAU’s media files storage component must be protected from unauthorized file
access, file uploads, file deletions or file modifications.
• The LPMAU API must be protected from unauthorized access.
A potential security mechanism for LPMAU would be access control for each of the com­
ponents. Components of LPMAU would have to make sure that they are receiving data
from authorized sources. For example, the worker program would have to make sure that
it is receiving jobs from the job queue component and not from an unauthorized source.

7.3
Multiple worker types
The current implementation of LPMAU only supports one kind of worker.
A workaround is available in the current version of LPMAU that allows for multiple worker
types. A worker_type parameter can be added as a potential input parameter to the long­
running process. The worker program would include all implementations of the various
worker types. The worker program can use the worker_type parameter to determine
which worker type to execute.
The workaround has some disadvantages. The worker program would have to include
all implementations and dependencies of the various worker types. This would result
in a larger binary size and a longer compilation time. Furthermore, observability would
suffer because the developer could not easily see which worker type is being executed.
Therefore, first class support of multiple worker types in LPMAU would be preferable.
7.4
Multiple outputs and output types
The current implementation of LPMAU is focused heavily on one media file as an output.
However, there are other kinds of long­running processes that do not produce media files,
or produce more than one file. These kinds of long­running processes are currently not
supported by LPMAU. An adjustment of the job database schema and the file storage
system would be required to support these new features.
Examples of long­running processes that do not produce media files are:
• Image classification by Artificial Intelligence (AI) that produces a short text output.
• Mathematical calculations that take a long time to complete and produce a single
value.
• Making a decision by parsing a large amount of data that produces a boolean result.
Use cases for allowing multiple file outputs are as follows:
• AI image generation of multiple images.
• Video editing professionals may need the video track and the audio track separately
from SlideApp for post­processing.
• Obtaining the result of a long­running process as well as a log file of the process
execution.
7.5
Interdependencies between jobs
A possible extension to LPMAU would be to allow jobs to depend on other jobs. This
would allow for a more complex workflow to be executed.
For example, a job could depend on the result of another job, taking the result in as
input parameters. This would allow for a job to be executed only after another job has
finished. The failure of a job could also trigger the execution of another job defined as its
dependency.
This new feature would turn LPMAU into a workflow management system like Apache
Airflow or Temporal, with a built­in file storage mechanism.
The workaround that enables this feature in the current version of LPMAU is to write a
monolithic worker program that has multiple possible long­running execution branches
that can come after one another based on pre­defined execution paths.
40

==PAGE:51==
The described workaround does not have the advantages that a workflow distributed
among multiple worker instances could have. For example, the more sub­jobs a job needs
to execute, the more CPU time the job would absorb from the other jobs in the queue.
This would result in a longer wait time for the other jobs in the queue. Therefore, a first
class implementation of interdependencies as part of the LPMAU API and SDK would be
preferable.
7.6
Caching
It is possible that two separate jobs may be submitted with the same input parameters.
In this case, long­running process would be executed multiple times, yielding the same
result. This is a waste of computing resources.
A solution to prevent this waste of resources is to cache the results of jobs. This solu­
tion would introduce a new component to LPMAU: the cache component. When a job is
submitted, the cache would be checked for a result with the same input parameters. If a
result is found, the cache component would add the same result to the job and mark the
job as finished, without ever starting the long­running process that would have produced
the result. If a result is not found, the job would be executed and the result would be
stored in the cache component.
In simple applications, a has of the user input could correspond to one record in the cache.
The cache must also be invalidated on each update to the long­running process imple­
mentation. This is because the long­running process may have changed and the cached
results may no longer be valid.
7.7
Checkpointing
Checkpointing is the process of saving the state of a long­running process at a certain
point in time. This allows for the long­running process to be resumed from the saved
state in case of a failure. Processes could also be paused and resumed by users at will.
This feature was suggested by both interviewees in Section 6.2.1: Developer interviews.

8
Conclusion
This thesis addresses the gap in the market for a BaaS product that can help developers
create Long­running Process Media Applications. The development of these applications
is simplified, making them accessible to a broader range of developers.
The Long­Running Process Media Application Utility (LPMAU) has proved to be a more
accessible approach to developing long­running process applications from scratch. LP­
MAU hides the complexity of managing a distributed cloud service system that supports
long­running process applications. Furthermore, the LPMAU’s cloud­agnostic design al­
lows the deployment of the software on developer workstations and multiple cloud plat­
forms, which makes it more flexible and adaptable to different environments. Scalability
is addressed directly in LPMAU by decoupling the long­running processes from the rest of
the business logic. Overall, LPMAU has the potential to open new possibilities for digital
media consumption and creation by making it easier for developers to create new and
innovative applications.
The evaluation of LPMAU also proves that there is demand for such a service. If the
features and extensions discussed as future work are realized, LPMAU could also be
marketed as a hosted service for a monthly fee.
