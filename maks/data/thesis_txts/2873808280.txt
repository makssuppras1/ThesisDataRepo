Abstract
In mid­February 2025, with the release of DeepSeek, it became one of the most popular
large language models for many people to use in various fields. In this paper, the au­
thor uses LeetCode and static analysis tools to analyze the generated code of DeepSeek
and its ability to improve imperfect or failed code by using the suggestions from static
analysis tools or by simply telling DeepSeek the purpose of the improvement, and com­
pares some similar results with OpenAI’s GPT3.5. The results show that DeepSeek can
give an average acceptance rate of 73.31 % and a very good acceptance rate for easy
questions(96.53%). DeepSeek improves code more effectively without relying on static
analysis tools. Compared with OpenAI GPT 3.5, DeepSeek v3 achieves higher accuracy
than GPT 3.5, and DeepSeek R1 performs worse than GPT 3.5 in fixing and improving
code..
Keywords: DeepSeek, Code Quality, Static Analysis

Introduction
In recent years, with the development of artificial intelligence, large language models have
become one of the most popular research directions in computer science and have been
used by people to analyze various areas and provide solutions as an assistant. Large
Language Models (LLMs) are generative mathematical models of the statistical distribu­
tion of tokens in the vast public corpus of human­generated text[1]. This kind of model can
also be used in code generation to solve code problems. At the end of 2024, a new LLM
model, DeepSeek­v3, was launched and immediately became one of the most popular
large­language models in the world.
DeepSeek­v3 is a half­open­source, free large language model that provides code gen­
eration, code completion, code infilling, and instruction following [2]. It is free in daily con­
versation and has better performance in code generation than other free large language
models, GPT v3.5, in their thesis [2]. According to the thesis, this model has excellent
performance in code generation and outperforms another popular free LLM, ChatGPT
3.5.
Due to its popularity and importance in code generation, scientists have already tried
to evaluate the performance of DeepSeek AI on code quality. Although DeepSeek was
only proposed 8 months ago, scientists have published a thesis to assess the quality
of DeepSeek­generated code and compared it with ChatGPT to evaluate the quality of
generated code. According to Md Motaleb Hossen Manik, DeepSeek has better accuracy
performance in code generation, ChatGPT encountered fewer issues, used comparable
or slightly less memory, consumed less execution time, and wrote fewer lines of code [3].
However, in this thesis, it only evaluates 10 Python problems and doesn’t mention the
performance of DeepSeek on code improvement ability, nor does it compare the difficulty
of these Python problems. It can not be considered universally applicable and highly
persuasive regarding the performance of DeepSeek code quality analysis.
In this thesis, 1124 LeetCode questions will be used to evaluate the quality of DeepSeek
AI­generated code performance. They will be tested in different difficulty levels(easy,
medium, and hard) by the code’s passing rate, time complexity, and memory complexity.
This thesis will mainly focus on three objectives of DeepSeek in the experiment:
1. The passing rate and quality of DeepSeek on LeetCode questions
2. The performance of DeepSeek in improving code
3. The difference in performance on code generation between DeepSeek and Chat­
GPT
In the experiment, DeepSeek will generate code to solve LeetCode questions and submit
it to LeetCode to test its passing rate and code quality. We will choose time complexity
as the measurement dimension to ask DeepSeek to improve the code that did not test
ideally and failing code by using static analysis tools (pylint[4] and Radon[5]) or without
analysis. After this, for some code that still does not pass on LeetCode, the reason for this
will be analyzed and a comparison between the performance of DeepSeek and the result
of ChatGPT (based on the Liu et al.[6] analyzing result of ChatGPT in 2024), which also
used Leetcode to test the quality of ChatGPT’s generated code and ask ChatGPT to fix
the problem by using pylint and fix without analyze so that these results will be compared
with same test in DeepSeek.

The result of the experiments shows that DeepSeek has a 73.74% passing rate on the
first try(96.52% on easy questions, 70.50% on medium questions, and 55.95% on hard
questions), for the code that do not have a perfect solution on time complexity, DeepSeek
achieved 4.86% codes perfect and the average passing rate increased from 59.56% to
67.16%. DeepSeek also improves 18.37% of the code to pass the exam.
By using static analysis tools, DeepSeek improved the 3.26% of perfect code rate, and
the average passing rate decreased from 67.19% to 64.78%.
The codes which are not pass on LeetCode after being fixed are also analyzed, and
found that some of them are because of the algorithm fault(like LeetCode wish them to
use dynamic planning or greedy algorithms but DeepSeek did not use), some of them
are because of the initial assignment is fault (like a=x, b=y but DeepSeek makes it a=y,
b=x as input). Some of them are because DeepSeek did not understand the question
clearly. This will be discussed in the Results chart with detailed examples. But after
telling DeepSeek the wrong cases’ input and the correct output. DeepSeek can find out
most of the problems.
As a result, compared with ChatGPT, Deepseek has better performance in first­time code
generation but has poor improvement in code fixing. DeepSeek can improve the code bet­
ter without static analysis tools’ suggestions, by analyze some detailed solution, it mainly
because by using Radon and Pylint, DeepSeek are more usual to change the structure
of Code to make it more readable and reduce complexity which make the solution more
complicated and DeepSeek are not good to deal complicated question which makes it
cause more mistake. If DeepSeek is asked to change the code by itself, it is more likely
to change the code with reasonable certainty, so it causes fewer wrong solutions. In the
improvement section, it highlights weaknesses in edge detection, problem understanding,
and the handling of complex algorithms.
This experiment and thesis can help people better understand DeepSeek and large lan­
guage models, and help programmers choose a better tool to help them, and for beginners
to consider using DeepSeek to teach them programming.

Techniques
In this chapter, some relevant techniques will be briefly introduced, and some relevant
work on the code generation of LLM(Large Language Model) and DeepSeek will also be
introduced, providing a reason why such research is important.
2.1
DeepSeek
DeepSeek is a code­focused large language model published in 2024. Among other pop­
ular large language models, DeepSeek utilises more Chinese, math, and code to train
models, making it more efficient in these areas [2]. DeepSeek uses a ’chain of thought’ to
improve its ability to solve complex problems like humans, and uses a ‘mixture­of­experts’
architecture to only let the model use the part relevant to the problem, which helps it cost
less than thirteth cost of OpenAI o1 [7]. Not only those techniques, but DeepSeek also
used Multi­Token Prediction, Multi­Head Latent Attention to keep the accuracy and im­
prove efficiency [8]. DeepSeek is popular and gets attention from other scientists because
it is part of open sources, which makes the former promote more innovative technical re­
search in language models than the latter, than OpenAI’s ChatGPT [9].
2.2
LeetCode
LeetCode is an online platform that provides numerous problems and solutions for users
to practice their coding skills and share their coding experiences with other users. It
supports multiple code languages for training and will give the tester information about
time complexity and memory complexity compared with other testers when the tester
has passed the questions. It includes many companies’ questions in applying for the
examination and helps companies select candidates. This website has become one of the
most popular websites for job seekers to find a job in computer companies because it has
many real questions that those companies will ask in an interview. Many scientists also
use LeetCode to test the quality of AI­generated code performance [10, 11]. LeetCode
can be directly compiled by copying and pasting the contents from the LeetCode website,
and it has different difficulty tasks to test the programmers’ ability [12].
2.3
Static analysis
Static analysis is a kind of analysis that analyzes the program without running it. It is a
popular software technique performed by automated tools for verifying large­scale soft­
ware systems. It works without executing the program, just analysing the source code and
applying various heuristics to find possible errors, code smells, and style discrepancies
[13]. Static analysis tools can be used to support developers during code reviews and can
help developers reduce the time and effort on code reviews, have a positive influence on
checking import and regular expressions problems [14].

Since large language models have been used in code generation and teaching purposes,
many scientists have discussed the reliability and quality of AI­generated code and how
to improve AI­generated code quality [15].
Many researchers study to analyze the possibility of AI in automated code generation
in software development work because AI can increase productivity and efficiency, and

Encoder­Decoder, Causal Decoder, Prefix Decoder, and Mixture­of­Experts[21]. Trans­

models’ generation code to ensure the quality of the study and programming by using
large language models’ assistance. There are many researchers who want to compare
different models’ generation code ability and try to give official methods to measure the
models [24, 6, 25]. However, there is not enough paper to measure a new open­source
model, DeepSeek, which has become very popular since its release to the public. Md
Motaleb Hossen Manik publishes the only paper that measures its code quality. In this

number of task cases. It can not be considered a comprehensive and universal quality
evaluation of DeepSeek.

3.2
Code generation
To deal with a large number of code generations. The application of the API for DeepSeek
is necessary. This website: https://api-docs.deepseek.com/ can help new users apply for
the first time API, and also beginners can ask DeepSeek chat to get detailed instructions
on how to use DeepSeek API.
In this code, we set temperature as 0.1 and max_tokens as 2000. Temperature is relevant
to the creativity of DeepSeek. The higher the temperature, the more creative DeepSeek’s
answers will be. The suggestion of DeepSeek on code generation is 0.0 because the
temperature is 0. DeepSeek will create code more conservative and meticulous, and with

Discussion and future work
In this chapter, the author will discuss the feeling of DeepSeek after the experiments and
provide suggestions on how to use DeepSeek more effectively and professionally in the
programming area. Additionally, the author will outline the limitations of this research and
suggest how other researchers could analyze the data more deeply in light of this thesis.
5.1
Discussion
Nowadays, with the development of artificial intelligence, more and more people are notic­
ing that AI can influence people in many different areas; programming is one of the most
prominent areas of AI. Many people are using AI to help with programming, studying pro­
gramming [20], and analyzing code. People need to measure the prospects and quality
of AI in these areas. This experiment measures DeepSeek in many different areas. In
these experiments, static analysis tools and LeetCode are used to measure the quality of
the generated code of DeepSeek and to assess DeepSeek’s improved ability. In those
tests, DeepSeek shows it is effective for easy questions. Compared with the average
solution passing rate of easy questions( 70%), DeepSeek’s accuracy is much higher than
this data by over 20 percent. This proves its ability to generate code. And this ability is
the same as the developer said in their paper, which has better performance than GPT
3.5 in accuracy, regardless of the difficulty.
However, there are also some shortcomings of DeepSeek, as the wrong solutions indi­
cate that DeepSeek struggles with problem analysis. When facing a complicated problem,
DeepSeek’s accuracy will decline rapidly as the complexity increases. DeepSeek some­
times will forget about edge cases, which may also cause wrong instances.
About Improvement, with the requirement of time complexity reduction, DeepSeek will
make some faults in some accepted code to make it unpassable. The test cases show that
DeepSeek can improve the wrong cases and time complexity without suggestions, which
is better than with the help of Pylint and Radon’s suggestions. In some cases, with the
suggestions of Pylint and Radon, DeepSeek is more willing to follow those suggestions
to make the code more readable and reduce Cyclomatic Complexity by using Radon’s
suggestion. However, this sometimes means great changes to the structure of the code.
Those operations cause DeepSeek to make more faults.
In the author’s opinion, DeepSeek is a good and useful large language model for those
seeking a free large language model in programming. Because it has proven to have a
better ability in programming than the current popular free models, OpenAI’s GPT 3.5, and
it shows a high passing rate on easy questions, it can be useful for beginners to study
programming by using DeepSeek to assist and help programmers to quickly generate
some code for easy tasks. However, because DeepSeek sometimes can not understand
the meaning of tasks and forgets some edge cases, for professional programmers who
want to seek accuracy, it is not a wise choice to use DeepSeek to generate code. The
best way to improve the quality of DeepSeek’s generated code is to use dynamic code
to identify potential issues or have humans test it to find bugs. Sometimes programmers
cannot use dynamic code or manually review the code. It would be better to ask DeepSeek
to improve the quality of the Code or fix the problem instead of using Pylint and Radon to
analyze.

5.2
Future work and Limitations
Due to the time limit and shortage of human power, these experiments still have some
areas that need improvement and enhancement. These experiments only checked 50
wrong cases and found some reasons for DeepSeek’s shortage, and in these experi­
ments, DeepSeek has around 200 wrong solutions. So if all the wrong cases are analyzed,
maybe more shortcomings will be discovered, and the understanding of DeepSeek’s gen­
erated code will be more objective and impartial. This paper only used Pylint and Radon
to analyze the code and found that DeepSeek can improve the code better without static
analysis tools. This result has some surprises for the author and is different from the re­
sult of Yue Liu et al.’s assessment of GPT 3.5. In their research, GPT 3.5 can improve
the code better with static analysis tools [6]. So the authors think that by changing the
static analysis tools, this result will maybe be different. In this paper, DeepSeek was only
compared with GPT 3.5, and there are some other artificial intelligence used for program­
ming, like LeetCode AI, GitHub Copilot, Codeium, among others. Those AIs can also be
compared with DeepSeek to compare the quality. Finally, the improvement of the gen­
eration code can achieve multiple rounds of experiments by comparing results without
or using static analysis to help see whether the gap between these two methods will be
enlarged or reduced. Also, it is compared by comparing the structure of DeepSeek and
other open­source’s large language models to check the structural differences and find a
better structure for large language models.
