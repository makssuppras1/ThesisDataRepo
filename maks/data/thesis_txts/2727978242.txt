Abstract
Forward error correction (FEC) is a key factor affecting the signal-to-noise ratio index of a high-speed trans-
mission system receiver.
With the continuous improvement of transmission rate and transmission distance
requirements, its importance in the communication system is also increasing. Open forward error correcting
(oFEC) error-correcting code is an improved braided error-correcting code with low delay, low implementation
complexity, and high net coding gain (NCG). That makes the code welcomed by many standards such as open
reconfigurable add-drop multiplexer (Open ROADM) and the international telecommunication union (ITU).
The investigation into oFEC will help the development of high-speed communication to reach the next level.
The decoding ability of the oFEC error-correcting code is still under investigation.
Due to computational
limitations, the decoding rate of the code cannot be simulated thoroughly by using software code. To reach a
lower bit error rate (BER), such as 10−15, one method would be to run the simulation on a hardware platform.
Field Programmable Gate Arrays (FPGA) are a kind of chip that can be programmed to meet specific use and
is perfect for the case because it is faster than running the simulation using the software.
Some simulations have been done in this field. In Wang’s paper, 100 pieces of FPGA boards were used to
simulate the decoding ability of the oFEC decoder under different parameters, and the output BER is able to
reach 10−15, which proves the method is doable.
One of the goals of this master’s thesis is to implement the proposed oFEC decoding algorithm on FPGA
hardware so that the output BER can reach 10−15. Also, minimizing the time it takes to reach that level is a
key goal of the thesis because it is supposed to perform faster than the software method. Another goal is to fit
the design into an 800 Gb/s data speed scenario. Hence the circuit speed of the design also needs to be fast to
speed up the overall data speed.
This thesis introduced a way of implementing the soft decision decoder of oFEC error-correcting code on Xilinx
xcvu9p-flga2104-2L-e ultra-scale FPGA platform. The algorithm for soft decision decoding is chosen to be the
Chase-Pyndiah algorithm, introduced in 1998 by Pyndiah in his paper. The data speed of the design reaches 64
Gb/s. Hence simulating output BER to 10−15 is estimated to take 87 hours, given the fact that the decoding
rate is 32 Gb/s. With this kind of data speed, it is challenging to fulfill the goal of fitting this single decoder
into the 800 Gb/s scenario. However, it is still possible to fit the decoding unit into an 800 Gb/s scenario, if 25
copies of the decoder are implemented in parallel to form the unit.

Introduction
1.1
Basic model of digital communication system
The purpose of communication is to transmit information that the other side does not know. Therefore, it is
required that a communication system must be efficient and reliable in transmitting messages. However, to
seek high efficiency, the time occupied by each data symbol must be shortened, and the waveform needs to be
narrowed. In this case, the possibility of errors will increase after the signal is disturbed, which reduces the
reliability of the transmitted message. If the reliability of transmission is required, the rate at which messages
are transmitted is slowed down. Therefore, how to reasonably solve the contradiction between reliability and
speed is one of the key issues in correctly designing a communication system. The communication theory itself
is also constantly developed in the process of resolving this pair of contradictions.
Figure 1: Basic model of digital communication system
All the digital communication systems can be depicted in the model that is shown in Figure 1. The raw infor-
mation bits contain information generated directly by the sender. Encoder functions to convert messages into
signals suitable for channel transmission. The channel is the intermediate medium through which the signal
is transmitted from the encoder to the decoder. After being transmitted in the channel, the signal will be
corrupted because of the existence of noise in the channel. The decoder functions the opposite of that of the
encoder, that is, to restore the message generated by the source from the signal interfered with by noise. After
the signal is decoded, the raw information bits should be able to be recovered. And it is expected to be identical
to the ones on the sender side.
In 1948, Claude Shannon proposed the Noisy-channel coding theorem, which is considered the fundamental
theory of channel coding. It states that given a noisy channel with channel capacity C and information trans-
mitted at a rate R, then if R < C, there exist codes that allow the probability of error at the receiver to be
made arbitrarily small. This means that, theoretically, it is possible to transmit information without error at
any rate below a limiting rate, C[9].
Since the proposal of the theory, numerous scholars worldwide have focused their efforts on researching channel
coding and striving to identify codes that can help data to be transferred comparably to the Shannon limit
in the channel. Over the past 60 years, various channel coding methods have emerged. Some of them have a
channel capacity that is very close to Shannon’s limit. Before going deep into the theory of the code in this
thesis, some fundamental knowledge of channel coding needs to be introduced.
1.2
Background of Error Correcting Code
As it has been mentioned, reliable data transmission is critical, especially in the presence of noise, interference,
or channel impairments. In recent years, with the rapid development of high-speed data networks, there has
been an increased demand for enhanced reliability in transmitting data. Therefore, how to control errors and
improve the reliability of data transmission have become an important issue faced by designers of modern digital
communication systems. Forward Error Correction (FEC) is a method used in data transmission to detect and
correct errors that may occur during the transmission of digital information. Unlike traditional error correction
methods that rely on the retransmission of data, such as Cyclic Redundancy Check (CRC), which can only
detect the existence of errors, FEC can correct errors on the receiving end without retransmission. This is
accomplished by adding redundancy to the transmitted data as extra bits, allowing the receiver to detect and
correct a predetermined number of bit errors in the received data.
Compared to traditional error correction methods, FEC has some advantages.
Firstly, it reduces the need
1
==PAGE:8==
for retransmission, saving time and resources. Secondly, it can be used to correct errors in real time, which is
particularly important in applications where latency is a concern. Finally, FEC can provide higher reliability
and robustness in data transmission, which is particularly important in applications where errors can have
serious consequences. Although there are some downsides to utilizing FEC, such as increased bandwidth usage
and limited error correction capabilities, it remains a popular choice in various scenarios, and industry demands
for faster processing speeds and improved error correction abilities continue to persist.
Error-correcting codes (ECCs) are typically classified into two categories: block codes and convolutional codes.
Block codes divide the data into fixed-size blocks, with each block independently encoded and decoded. Convo-
lutional codes, on the other hand, operate on a continuous stream of data and generate an output based on the
current input and past inputs. The block codes are flexible in code design and easily implemented. However,
the convolutional code is more efficient in encoding and decoding and has lower redundancy regarding error
correction capabilities and the utilization of limited bandwidth. The choice between these two kinds of code
depends on the trade-offs between implementation complexity, coding efficiency, and error correction capabilities.
Significant advancements have been made in the development of FEC techniques. In 1950, Hamming proposed
the Hamming code, a complete linear block code that can correct one error. Elias discovered the convolutional
code in 1955. It has a small time delay, which makes it suitable for wireless communication systems. After that,
Viterbi discovered the maximum likelihood algorithm (ML) for the convolutional code decoding algorithm. In
terms of block codes, the Hamming code that corrects one error is insufficient to meet practical applications’
needs. Around 1960, the discovery of the Bose-Chaudhuri-Hocquenghem (BCH) code family, which could cor-
rect multiple errors, was an important milestone. Then, Reed and Solomon discovered a subset of BCH codes,
which were named Reed-Solomon (RS) codes. This code has been widely used in Compact Disc (CD) players
and the Digital Video Broadcasting (DVB) scheme. Over time, several ECCs that can approach Shannon’s limit
with reasonable complexity have been proposed successively, such as Turbo codes[10] and low-density parity-
check (LDPC) codes[11]. The decoding algorithms of these error-correcting codes are relatively simple, and the
integrated circuit technology has made great progress so that these error-correcting codes can be implemented
and used in real life.
Using FEC allows the messages to be encoded into codewords on the sending side.
After passing through
a noisy channel, the received codewords need to be decoded to recover the original information. There are
generally two ways to decode FEC: soft decision and hard decision. In the soft decision, the receiver assigns a
probability value to each received bit based on the amplitude or phase information, indicating the likelihood
that the bit is either a 0 or a 1. This information is then used to calculate the most likely data sequence, which
can be used to correct errors. With probabilities, the decoder can perform iterative decoding. This will increase
the accuracy of the decoded result. In the hard decision method, the receiver makes a binary decision about
each received bit, treating it as a 0 or a 1. The hard decision is simpler and faster than the soft decision. Still,
it may not be as accurate in some situations as the soft decision decoder since it makes binary decisions based
on discrete thresholds.
With the development of the decoding algorithms, some FEC algorithms, which could only be decoded in
a hard decision manner before, can now also be decoded in a soft decision manner. Due to the accuracy of the
soft decision, it has become increasingly prevalent, especially in applications requiring higher error correction
capabilities, such as wireless communications, digital broadcasting, and optical communication systems. In the
5G standard, using the LDPC decoder as the FEC decoder has contributed to the increasing popularity of soft
decision FEC. The algorithm can effectively exploit the extra information provided by soft decision decoding,
resulting in improved error correction performance.

Research Objectives

Theory behind oFEC Error-Correcting codes
This section first briefly introduces the BCH code encoding algorithm. Then it goes through hard-decision
decoding and soft-decision decoding of the BCH code, which includes the introduction of the Chase-Phydiah
decoding algorithm. This algorithm is the same one used for decoding oFEC EEC. In the end, the interleaving
process and the decoding ability of oFEC will be discussed.
2.1
Structure of oFEC System
The theory of the oFEC ECC can be introduced based on different phases of data processing. The oFEC system
can be simplified into three parts, as shown in Figure 2. The raw data bits at the beginning are 222 bits long
and come from a scrambler. This contains the real information in the message that the device should receive
on the receiving side without any error after decoding. The codeword for oFEC ECC is chosen to be extended
Bose–Chaudhuri–Hocquenghem codes (BCH codes), which will be introduced in section 2.2.
Two encoders
encode the raw information bits into BCH codewords, and they are running in parallel to speed up the process,
so the 222 bits of data will be separated and input into those two encoders based on their index. If the index
is even, the bit will go to encoder 0. Otherwise, it goes into encoder 1.
After passing the information bits through both encoders, a message with 17 bits will be added to the end
of the information bit. The name of the bits it adds here is parity bits, which do not contain information
but are only for error-correcting. With more parity bits in the message, the information bits will be more pro-
tected against noise. Still, on the other hand, it will reduce the bit rate, making the decoder receive less valuable
information in a certain period of time. After adding 17 bits to the data, the current bit rate is 111/128 = 0.867.
Now, the message now has 2x128 bits in total, and now it needs to be interleaved. The encoder interleavers
will shuffle the locations of bits and make them more robust to the noise in the channel. After the processing is
done, two interleaved messages will be combined together to make the codeword into a staircase structure in the
channel interleaver. The positions of the bits will be rearranged again to reduce the effect of burst errors and
improve the overall error correction performance. When the message goes through the channel, errors can occur
due to noise, interference, or other factors. Burst errors are a type of error where consecutive bits or symbols are
affected. FEC codes are designed to detect and correct errors but are most effective when errors are distributed
randomly. An interleaver rearranges the order of the data before it reaches the channel. It shuffles the input data
so that consecutive symbols are spread out and dispersed throughout the encoded output, reducing their impact.
Currently, the message contains 256 bits and is prepared for transmission through the channel.
However,
the channel will introduce noise, causing some of the bits in the message to flip. The message then goes through
a channel deinterleaver system and a decoder deinterleaver system to restore the codewords. These two blocks
function just the opposite as the channel interleaver and oFEC encoder interleaver. Once the bits are put back
to their original order, the decoding process can start.
The data retrieved by the decoder interleaver can be viewed in two segments.
The first half of the code-
word has 128 bits, which is the data that has already been decoded once. They are fetched again for the second
round of decoding. The first round of decoding is called horizontal decoding, the second round is called vertical
decoding, in terms of storing data in a matrix. This will be further explained in section 2.5. The second half
of the codeword also has 128 bits. This part of the codeword has not been decoded yet. Thus, it needs to
be decoded horizontally. Together, these two parts of the codeword assemble a full codeword, which can be
decoded by the decoder. And after going through all the data stored in a matrix, all the bits can be decoded
both horizontally and vertically, which fulfilled the decoding concept of Turbo Product Code (TPC). This will
be further explained in section 2.4.
According to the standard of oFEC EEC, Any of the iterative algorithms designed for turbo decoding of
Product Codes can easily be adapted to decode oFEC codewords[6]. Thus, the Chase-Pyndiah algorithm can
be seen as a suitable decoding algorithm for the soft decision part of the oFEC. It has a good trade-off between
decoding ability and complexity. The Chase-Pyndiah decoding algorithm for Turbo Product Code was proposed
by Pyndiah R[5]. It is an iterative algorithm that aims to minimize the number of decoding errors by iteratively
improving the estimated codeword. It supports the soft-input/soft-output (SISO) data mode and is perfect for
decoding block code such as BCH code in a soft decision way in a Turbo Product code case. The message after
decoding will be as correct as possible. Even though the BCH decoder might not correct all the errors, it is the
nature of the FEC that only some of the errors can be corrected. The decoder of oFEC has many stages, and
the most well-known one is the three soft-decision decoders concatenated with two hard-decision decoders. This
4
==PAGE:11==
combination will make the likelihood of error-correcting higher, as well as occupying not too many hardware
resources. By having all five stages of the decoder connect to each other, it realizes iterative decoding.
Finally, a separator can cut the information bits out of the rest parity check bits. And the information bits
suppose to be the same as the one that was input into the system.
Figure 2: The simplified structure of oFEC system
For the three parts of the oFEC system, the decoder is more complicated than the encoder and the channel
sector. Thus the decoder part is what this thesis investigates. Also, this thesis aims for the most interesting
part of the decoder, which is the soft decision Chase-Pyndiah decoder. In the upcoming sections, It provides a
brief overview of encoder theories, followed by a detailed explanation of the decoder.

2.4
BCH Soft Decoder
Soft decision decoding is another way to decode the BCH codewords. It works in a similar way to the BCH
hard decision decoding. Instead, it deals with likelihoods. In figure 2, this part of the theory is applied in the
decoder block.
Figure 5: Illustration of Soft Decision Input Data[2]
The appearance of soft decision input data can be seen in Figure 5. When the codeword is received by the
decoder, each position contains a likelihood ratio indicating how much this bit should be positive or negative.
If the value of the likelihood ratio falls at the locations of 0 or 1, then the value of that bit is almost certain.
Otherwise, if it falls in the middle of 0 and 1, the certainty would drop, indicating that its current value might
be wrong.
For the BCH soft decision decoder, it simply categorizes the likelihood ratios into hard decision bits. Then
the decoding process is the same as the hard decision decoder. Until it reaches the final output stage, the
absolute value of the likelihood ratio of each bit will be assigned to the hard decision bit, which provides the

sign. After the assigning process is done, the soft decision decoding of the BCH code is finished.
The soft decision decoder is a crucial part of the oFEC decoding system. It is more complicated than the
BCH soft decision decoder because oFEC introduced a staircase structure to encode the information bits. The
structure is quite similar to the structure of the TPC, which is a concatenated code. The structure of such
a code can be visualized as a two-dimensional array or matrix, like what is shown in 6. By using this struc-
ture, each bit can be encoded both vertically and horizontally, which increases the chances of correcting the

2.4.1
Chase Algorithm
The received data is of soft decision type, which allows for iterative decoding. If an iterative decoding algorithm
is used, it can reflect the excellent performance of Turbo product code. The decoding methods that were used
to decode TPC were the "bit-flipping" algorithm or the "maximum a posteriori" (MAP) algorithm. Still, these
two methods have their limitations on decoding thresholds and error correction capabilities. Until Pyndiah pro-
posed a SISO Turbo product code iterative decoding algorithm based on the Chase algorithm. In this section,
it will first introduce how the Chase algorithm works. And in the following section, the contribution of Pyndiah
to the Chase algorithm will be mentioned.
The Chase decoding algorithm basically converts any hard-decision decoder into a soft-decision decoder. It
aims to estimate the most probable transmitted message by considering the statistical properties of the encod-
ing scheme and the transmission channel. The Chase decoding algorithm is highly advantageous because it
handles various error patterns effectively. It accomplishes this by analyzing some error patterns and determin-
ing the likelihood of each one based on the received signal. This enables the algorithm to decide on the most
probable transmitted message.
To understand the Chase algorithm better, the theory of Bounded Distance Decoding (BDD) needs to be
introduced. This Chase decoding algorithm uses BDD as a crucial component of its process. BDD helps esti-
mate the most probable message transmitted by searching within a limited distance or radius from the received
codeword. The term "bounded distance" refers to the maximum number of symbol errors that are allowed
between the received codeword and the potentially transmitted codeword during the decoding process.
The use of BDD in the Chase algorithm allows for error correction in situations where the symbols received
may not be entirely trustworthy or when the channel introduces a small number of errors. By considering a
bounded distance around the received codeword, the algorithm tries to find a codeword within that radius that
is closest to the received codeword, giving the Chase decoding algorithm more opportunity to correct the error
in the received codeword.

2.4.2
Calculation of the Extrinsic Information
As has been introduced above, the Chase algorithm allows converting a hard-decision decoder into a soft-decision
decoder. One of the outstanding features that the Chase-Pyndiah algorithm has is its iterative feature. This
feature allows the same location to be decoded multiple times, meaning the same information passes between
different decoding nodes. This information is about the likelihood of each possible transmitted symbol. At
each iteration, the algorithm updates these messages by combining the extrinsic information from neighboring
nodes. The extrinsic information is obtained by comparing the received data with the decoded data under
the current iteration. Given the received data, it represents the difference between the likelihoods of different
transmitted symbols. By having this extrinsic information in the decoding process, the algorithm can enhance
its estimations and enhance the precision of the decoded output.

calculating the reliability of symbols in the decoded codeword, which is called the Chase-Pyndiah algorithm[5],
and the specific process is as follows.

to have the smallest Euclidean distance to the received codeword R, except for D. If a codeword like C can be

credibility of the external information is gradually increasing, and the weight factor is also increasing.

needs to be adjusted based on multiple simulations to determine how much the decoding result in a certain

To sum up, by adding extrinsic information and weight factor, the Chase-Pyndiah algorithm makes the it-
erative decoding feasible and has better accuracy.
The same algorithm will also be used in the hardware
implementation of oFEC regarding the soft decision decoder.
2.5
The Interleaving of oFEC
After the soft decoding algorithm for oFEC has been introduced. Here comes another important feature of
oFEC, which is the interleaving process. This part of the theory is used in the oFEC encoder interleaver and
the oFEC decoder deinterleaver block in figure 2. The interleaver process is an important part of the oFEC for
managing the data in the transmission process. This is needed in the system because it helps improve the decod-
ing performance. Having the bits well spread can improve the error correction capabilities of FECs, as they are
13
==PAGE:20==
designed to handle a certain number of errors within a given block. It can also make the data more tolerant of
some noises, such as burst errors. Burst errors are a type of error where consecutive bits or symbols are affected.
FECs are designed to detect and correct errors but are most effective when errors are distributed randomly.
An interleaver rearranges the order of the data before it is encoded using an FEC. It shuffles the input data
so that consecutive symbols are spread out and dispersed throughout the encoded output, reducing their impact.
The interleaver process reorganizes the data from the encoder, preparing it for transmission over a commu-
nication channel with lower affection by the noise. However, its job doesn’t end there. After the decoder side
receives the data, the interleaver block must reconstruct the original codewords, essentially reversing its initial
process to retrieve the original information. As such, the interleaver block is integrated into both the encoding
and decoding stages, ensuring smooth and accurate data flow. In this thesis, the block for interleaving on the
encoder side is called interleaver, and the block for interleaver on the decoder side is called deinterleaver. As
mentioned in the previous section, the interleaver for the channel is not introduced in this thesis. This section
only discusses the interleaver in the encoding and the decoding process.
Once the soft data is received, they are sequentially placed in the matrix.
One crucial aspect of this pro-
cess involves identifying the locations of the codewords within this data. Codewords are blocks of data that
the encoder has processed for error correction. To find these codewords, we must apply two specific formulas
that are also used by the interleaver on the encoder side. By utilizing these formulas, we can pinpoint the exact
positions of the codewords within the data stored in the matrix.

Matlab Simulation on Quantization Level
In order to implement an oFEC decoder on hardware, it is essential to select a suitable number of binary bits
to quantize the channel input data, which are decimal numbers. This is because hardware cannot store decimal
numbers directly. Consequently, the choice of the binary representation becomes crucial as it will impact the
results generated by the hardware. Therefore, it is necessary to conduct a simulation using Matlab to determine
the optimal number of binary bits required to quantize the channel input data.
output = ⌊(input(i)/saturation_value ∗(2quantization_level))⌋+ 0.5
(9)
Formula 9 illustrates the quantization process, where the variable ’quantization level’ determines the number of
binary bits utilized. The ’saturation value’ signifies the maximum decimal value that can be represented within
the given number of binary bits. Increasing the value of the ’saturation value’ makes it possible to convert even
small decimal numbers to binary format with only minor information scarification. By employing this formula,
it becomes feasible to construct the quantization function in Matlab.

The simulations with different quantization levels are built based on the result from the five iterations case

rithm with different quantization levels. As the quantization level increases, a higher number of errors will be
detected at a specific BER level. This trend is accurate since greater quantization of the signal leads to greater
signal corruption, resulting in more errors in the decoded output. The light blue line (4-bit case) is noticeably
above the red line (non-quantized case), which means its decoding result is much worse than the non-quantized
case. For the rest of the quantization levels, their results are closer to the non-quantized case, sometimes even
better than it. That is because they are not decoding based on the same input data. Since the input data is
randomly generated, it could be the case that the quantized decoder accidentally received a set of easier input
data for decoding. If the decoders on each quantization level have the same input data to decode, then the
non-quantized one will have the best decoding result among others.
To ensure the performance, it was determined that 5 bits would be used to represent the absolute value of
the decimal number and 1 bit to denote its sign. This results in a total of 6 binary bits needed to represent a
decimal number. And with the help of 2’s complement, it is able to convert the received decimal number into
6 binary bits two’s complementary representation. The method for conversion is shown below.
3.1.1
Positive number xi > 0
(1) Convert the magnitude of xi to binary.
(2) Pad 0’s to 6-bit size.
3.1.2
Negative number xi < 0
(1) Convert the magnitude of xi to binary.
(2) Pad 0’s to 6-bit size.
(3) Invert bits to achieve 1’s complement.
(4) Add 1 to achieve 2’s complement.
By using the formulas above, the decimal number 1 can be converted into ’000001’, the decimal number -1
can be converted into ’111111’, and so on. However, for the smallest negative number, even though it can be
represented as ’100000’, it cannot be reached in this design. Because after converting this binary number into
positive, it will be the same, and the hardware will still see it as a negative number, which will compromise the
whole design. Thus, the smallest number is represented as ’100001’ so that it can be recognized as a positive
number after conversion. After having this binary representation decided, the introduction of the hardware
design starts in the following sections.

4
Overall Hardware Structure
In this section, after understanding the expected error-correcting ability of the algorithm and the theories behind
it, a hardware structure based on the decoding algorithm is introduced. As has been mentioned in the introduc-
tion section, the soft decoder for the oFEC EEC is the main focus of this thesis. Thus, a hardware soft decoder is
what is implemented in this project. Figure 16 illustrates the hardware implementation of the algorithm. In this
implementation, the number of soft bits is chosen to be 5, and the binary number is represented in the 2’s com-
plement way. For constructing test patterns, 8 LRPs are found from the deinterleaved data, and among them, 3
are flipped in each test pattern, which means the implementation can create up to 93 test patterns if it is needed.
Figure 16: Hardware Structure of the Soft Decoder
When the soft input data reach the soft decoder, the first block it runs into is the deinterleaver block. The
deinterleaver block is able to take in a 128-data-bits-long signal each clock cycle from the channel deinterleaver
and construct 256-data-bits-long codewords based on the received data. Then the codewords are passed into
the sorting block. In this block, the eight least reliable positions will be marked. The inputs are soft data,
meaning the data is likelihood ratios and can be compared.
After finding the indexes for the least reliable positions, the indexes, and the codewords will be passed down
to different test pattern sides. There will be a unique bit-flipping block for different test patterns to flip the
values of different least reliable positions. The flipped codeword will then be categorized into ones and zeros for
21
==PAGE:28==
the syndrome calculation. Once the syndrome values S1 and S3 are calculated, they can be transferred to the
BCH decoder, where the error location is calculated, and the data is corrected. In the meantime, during the
flipping procedure and the error correction procedure, some values have been changed. The absolute values of
those changed positions are added together as final weight values, which will play an important role in the soft
output block.
In the soft output block, the final weight values and corrected hard codewords from all the test patterns
are collected for comparison. The soft codeword can be calculated by combining the weight value and its cor-
responding hard codeword from a certain test pattern because the weight value indicates the probability of a
position to be one or zero. Then combining all the calculated soft codewords can give out a single soft data,
which is the best estimate of the soft output.
In the final step, if the data has already reached the iteration limitation, it can be output to the following
hard decoders waiting for it. Otherwise, together with the original soft input data, they are used to generate
the extrinsic information for the next decoding iteration. In the next iteration of decoding, the data will go
through the same process again until it reaches the limitation and can be output to another procedure. Hence,
the hardware architecture suggested in this thesis can be referred to as a stage of a fully operational soft-decision
decoder.
A complete decoding system in this thesis will have five stages. Among them, the first three stages are soft
decision decoders like what has shown in Figure 16. The rest two stages will be hard decision decoders. The
configuration of the system can be changed into four soft decision decoders and one hard decision decoder
according to the needs. But it is expected to have three to four soft decision decoders in the system. Also, in
the theory section, it has been proved that the decoding performance is better after having more soft decision
decoders. However, the size of the soft decision decoder is bigger than the hard decision decoder. Hence, it is a
trade-off between the decoding ability and the hardware occupancy.
The original goal of the project is to have an oFEC decoder that can be fitted into an 800+ Gbps applica-
tion scenario. But it is hard to measure how fast the clock speed in hardware is required since the overall data
speed not only depends on the clock speed but also on how to implement the decoders in parallel. Since that is
the case, the goal has changed into simulating the decoding ability to output BER 10−15 and how long it will
take to finish the simulation. To not make the simulation time too long, each individual block will have 400MHz
clock speed as a goal. All the tests conducted on each hardware block are based on the 400MHz timing constraint.

Detailed Hardware Structure
In the previous section, the dataflow within the system has been introduced. This section breaks the system
into individual hardware blocks and focuses on how each block is implemented, what hardware algorithm they
use, and their performance.
5.1
Implementation of Deinterleaver Block
As introduced in the theory section, the task of the interleaver block is to recover the codewords from the
received input data. So, having a storage part to save the incoming data and proper logic for finding the right
data among the received ones is crucial for this block.
There are two ways of implementing this storage part.
One of the ways will be using RAM to store the
data. This is a traditional way of storing data, and it is easy to implement but hard to read since it is necessary
to find the right address to navigate through the RAM. The other way could be using many registers to store
them. The advantage of using registers is that the incoming data will be passed down through registers, and
it can read the same positions all the time. After figuring out how many clock cycles it takes to decode the
codeword, the number of registers it needs in the register chain can also be certain. But the structure will be
too huge to handle for the selected FPGA board. Besides, registers are way too expensive compared to using
RAM for data storage. So in this project, a RAM block is used to store the data.
However, unlike what is shown in Figure 11, the task is not as straightforward in a hardware environment,
where the RAM block size is finite. Therefore, careful considerations must be made when designing the RAM
structure. The proposed RAM structure in this thesis, as depicted in Figure 17, includes a guarding space and
22
==PAGE:29==
a delay extension space, both equal in size. The number of rows in these spaces directly depends on the delay
of the decoder, ensuring all extracted data have already been decoded both vertically and horizontally. The
rest of the RAM block contains 18 rows besides those two parts. That is inspired by Figure 11, which shows
that at least 18 rows are needed to generate 16 codewords. No codewords can be generated if it has less than
18 rows. The figure is shown in a circle shape because the RAM block has a cyclic property. There is no fixed
starting position for each section, and only their size is fixed. Figure 11 also shows that it needs eight columns
in the RAM, but the hardware is designed to have eight 1-row-width RAM blocks. Each block has one port for
reading and the other one for writing. Thus the data can be read or written faster with all 16 ports working in
parallel.
However, a counter-argument for the RAM design could be having a single RAM block instead of having
it divided. After codewords are read out and composed, it will take 16 clock cycles to put those data into the
decoder. That means it can offer 16 clock cycles to prepare the next 16 codewords. In the end, after considering
the trade-off between bit routing and data fetching speed. It is decided to use one big RAM instead of eight
small RAMs in the design. This method makes the data fetching speed slower, but it is still fast enough for
assembling the codewords and feeding them to the decoder in each clock cycle.
Figure 17: Structure of the RAM component
Each clock cycle, the component receives a new square block of data from the channel, which contains 256 sets
of soft data. It is chosen to be that number because that is also the size of the read and write port of the RAM
block. If a bigger size of data is fed into the component each clock cycle, then a FIFO will be needed to store
the data temporarily. And these sets of soft data are carefully placed in a 16 by 16 square block, adhering to a
particular order, as shown in Figure 18.
Figure 18: Order of placement in the block
When constructing codewords, it is vital to extract the appropriate data from the RAM block. A codeword
can be formed from a specific pattern in the RAM, as shown by the red line in Figure 11. This pattern can be
applied to all codewords, making the process of extracting data for decoding consistent and cyclic, as illustrated

At the beginning of the process, the data in the RAM block will be unknown or not related to the information
that will be decoded. So, the first thing to do is to send the unknown data out to the RAM block in the second

unknown data from the second row will be taken out, and its corresponding pattern will be used to attach the
new coming information from the channel, then they will be sent to the decoder. The same process will happen
all the time. When the codeword has been decoded, it needs to be written back to the location where the formal
data in the RAM was read out in that clock cycle. The process is illustrated more specifically in Appendix 1.
Once a bit has been decoded both vertically and horizontally, it is passed onto a RAM block for the next
iteration. Simultaneously, the RAM block from the first iteration takes in a new set of 8 square blocks from
the channel and continues the decoding process. This concurrent operation of different stages forms a pipeline
structure, allowing a continuous flow of operations without any halts. With each clock cycle, a new square block
enters the first iteration. In sync, the decoder and the second iteration RAM block also receive a square block,

The second iteration RAM block mirrors the design of the first iteration RAM block, complete with stor-

Managing the addresses for storing decoded data back into the RAM is crucial.
Given that the decoder’s
delay is fixed, a series of registers can be used to maintain the addresses currently under decoding. Once data
is decoded, the stored address is sent to the RAM to guide the writing of the decoded result back to its original
position. An alternate method involves using an additional array containing all the shifted addresses in the
RAM. The shift amount corresponds to the number of clock cycles the decoder introduces. The success of this
method hinges on accurately calculating the decoder’s delay and synchronizing the address with the decoded
data.

logical vectors, so that it can be synthesized into Block RAM (BRAM). Converting the self-define signal into

5.2
Implementation of Sorting Block
The sorting block’s objective is to identify the positions in the given array where a few smaller values are
present. Following the assembly of codewords by the deinterleaver block, which utilizes likelihood ratios as soft
inputs, the sorting block must convert the received data to its absolute value for comparison. This conversion
is necessary because when the likelihood is close to 1, the corresponding bit is highly likely to be 1, and when
the likelihood is close to -1, it is more likely to be 0. By applying a 2s’ complement and converting the bits
to their absolute values, the sorting block can identify the ambiguous ones with values close to zero after the
conversion. Those positions are called the least reliable positions.
According to Wang’s paper[7], there are primarily two options regarding the number of marked positions:
the six-position case and the eight-position case. The difference is in the number of test patterns they can gen-
erate. In the six-position case, flipping the values in three out of six positions yields 42 test patterns. Flipping
the values in three out of eight positions in the eight-position case generates 93 test patterns. This hardware
design aims to fit as many test pattern as possible on a single FPGA board. Hence, it has been determined to
mark the eight least reliable positions.

Bitonic sort is a comparison-based sorting algorithm that can be run in parallel. The input values on the left are

Same structure can also be utilized when the number of the test patterns is odd. For example, if there are
93 test patterns, the last test pattern has nothing to compare to in the first stage. It will go directly into the
second stage and be compared to its neighbor. If it does not have a neighbor to compare to, it can go to the
final stage to compare to the final output. The whole process will not change the accuracy of the comparison.
Because if there are not any values in the last test patterns that are smaller than the final output, they will be
ignored after comparison. Otherwise, some bits in the final result will be replaced by the bits from the last test
patterns, which will achieve the same result as the even number test patterns case.

comparing the inputs bit by bit. For example, since the given result from the Sorting component is already
in ascending order, the Merge component can compare the first value of the received array to see which one is
smaller, and then keep the small one at the first position of the output. After that, the comparison will move

the second bit of A and the first bit of B. The smaller one will be taken to the second position of the output
array. The comparison will keep on going until all 8 times of comparison are done. Finally, the output array
contains the 8 smallest values of the two sorted chunks of data.

5.3
Implementation of Bit Flipping Block
The bit flipping process is one of the most important processes in a Chase decoder. As it has been introduced
in the theory section, a Chase decoder starts with an initiate estimation of the codeword. Then it starts to flip
other bits with low reliability to explore different possibilities and to see if it can give a better result.
Back to the hardware, since the locations of the least reliable positions have been found in the sorting block,
what needs to be done in this block is to flip three bits among the rest eight bits. As introduced in the theory
section, flipping three random positions among eight will create 93 test patterns in total. These test patterns
will be corrected individually and compete with other codewords for the final output.
The index signal from the sorting block is passed down to the bit-flipping block in a vector.
Hence, the
bit-flipping block can select the three positions that it wants to flip to form a test pattern. Theoretically, if
all 93 test patterns were decoded in the hardware, all eight indexes would have an equal chance of being used.
However, in this project, only eight test patterns will be tested due to the complexity of having 93 copies of
circuits.
The most challenging aspect of this block involves locating specific information within the 256 data long.
If this retrieval operation needs to be completed within a single clock cycle, it would require a large multiplexer
in the circuit. However, such a multiplexer would be quite sizable and negatively impact the overall hardware
speed. To address this, a more effective approach is to introduce pipelining, which involves reducing the size of
the multiplexer and spreading the retrieval process across multiple clock cycles.
Figure 33 and Figure 34 depict two approaches for locating a single data within a 256-data data set.
In
Figure 33, a massive 256-data in 1-data out multiplexer is used, posing difficulties in fitting the process into a
single clock cycle. To tackle this challenge, the circuit in Figure 33 divides the operation into four clock cycles,
utilizing 4-data in 1-data out multiplexers for all components. By using smaller components, the clock speed
can be improved. Additionally, dividing the processing load into four clock cycles enables faster clock speed
operation.
Figure 33: Circuit for Bit Retrieving in One Clock Cycle

Figure 34: Circuit for Bit Retrieving after Pipelining
Once the value of a specific position is extracted, the sign of the extracted value will be flipped using the 2s’
complement method, and then it will be put back to its original place in the array. This is performed by the
flipping circuit. Since the flipping position is placed in random order and may appear in any location of the
received array, that means the flipping circuit needs to go through all 256 positions in the array, and all three
positions can only be flipped one after the other. Hence, three flipping circuits are concatenated together to
form a flipping circuit. The absolute values of the flipped locations are marked and will be used in the weight
calculation block. More explanation of this part will be introduced in that section. After all the positions have
been flipped, the data clips, which have 4 data long, will be attached to form a 256-data-long segment and sent

5.4
Implementation of Syndrome Calculator Block
In the BCH decoder, the syndrome values play a crucial role in error detection and correction.
The BCH
decoder will pinpoint the error locations based on the calculated syndromes. To compute the syndrome, the
received codeword is multiplied by the generator polynomial of the BCH code. The resulting vector is the
syndrome vector. Each entry in the syndrome vector represents the result of a specific parity-check equation.
The syndrome vector will be all zeros if the received codeword is error-free. If errors are present, the syndrome
vector will have non-zero values, indicating the presence and location of errors.
However, multiplying the codeword with the generator polynomial is a very complicated calculation in hard-
ware. It might be doable, but it is almost certain that the speed of the circuit will be slow. So in this thesis, a
unique method to calculate the syndrome is proposed. Figure 16 shows two syndrome values that need to be
calculated in this block. The calculation of the S1 will be taken as an example below.
According to the earlier explained theory, the syndrome value may vary from 0 to 255, and each individual
bit within a BCH code has a unique influence on the resulting syndrome values. Consequently, it is feasible
to construct a matrix that provides specific information regarding how each bit contributes to the syndrome
values. As mentioned earlier, the syndrome value is zero if there are no errors in the codeword. Since the BCH
decoder considers the scenario where all codeword bits are zero as an error-free codeword, it is possible to iter-
atively flip each bit position to zeros and observe the resulting impact on the syndrome values. Implementing
this procedure in a Matlab script allows the influence of flipping different bit positions to be calculated and
analyzed. The input codewords are shown in Figure 37. And the result is shown in Figure 38.
Figure 37: The Test Inputs to the Syndrome Calculator
Figure 38: The Effect on the Syndrome Value
The first value showing in Figure 38 indicates the effect of when the first bit is wrong, how much impact it has
on the syndrome value. the rest of the data indicates the same thing for their own location. Take the first value
as an example. After converting the value 147 into binary format (10010011), it can be seen that it impacts the
first, second, fifth, and eighth positions of the binary format of the syndrome value. The second value, 71, will
be (01000111) after the conversion, indicating it impacts the first, second, third, and seventh positions of the
syndrome value.
When there is more than one error in the codeword, for example, the bits in position one and position two

combining all the valid corrected codewords and merging them to create the one with the smallest distance to
the received codeword. Hence, a good way to combine all the received codewords is needed in this block. The
tree structure mentioned in the bit-flipping section can be used here.
Figure 53 illustrates how the processing is done. First of all, two sets of input information will perform a
merging depending on if the codeword is valid or not. In the BCH decoder section, it has been introduced
that there are some moments when the errors in the codeword cannot be corrected, and those cases should be
counted as invalid when calculating the final output. That means in this block, if the incoming correction in
signal equals "100", then it will not participate in the generating of the final output.

positions where the value is ’0’. When that happens, the likelihood for that bit to be ’0’ will be given by the
weight value, and the likelihood for it to be ’1’ will stay unknown. In the case of an invalid input codeword,
the likelihoods for the bits to be either ’0’ or ’1’ are all unknown.

6.3
Overall Performance Evaluation
Based on the outcome of Figure 63 and Figure 64, it appears that the total design uses up roughly 32 percent of
the LUTs and 10 percent of the flipflops on the FPGA board that is being evaluated. This suggests that more
flipping patterns can be incorporated into this board. With the current hardware occupancy, the board can
accommodate an additional 8 flipping patterns without causing a significant decrease in speed, resulting in a
total of 16 test patterns. However, if more test patterns are required to fit onto the board, it may be necessary
to reduce the size of each pattern further. The potential risk could be the size of the soft output block because
the decoded data of all the test patterns need to be received by that block at the same time, and then the
comparison will start. If more test patterns are implemented in this design, then the size of that block may
increase drastically.
The maximum clock frequency for the circuit is around 250 MHz, which is acceptable. However, every sin-
gle component has a clock frequency higher than 400 MHz. Normally, it should also be around 400 MHz after
connecting all the components, given that the overall occupancy of the FPGA board is less than 70 percent.
A lot of tests have been done to search for the reason causing it. Ultimately, it can be concluded that it is
because of routing congestion. Routing congestion happens when the available routing resources within the
device cannot accommodate the required interconnections between logic elements. In this design, the soft data
codeword needs to go through the whole system, which uses up most of the routing ability on the board and
causes congestion. After reducing the length of the routed data, the speed was returned to the expected 400
MHz speed. This offers a hint for future design. It would be better to give some speed redundancy to each
block so that the speed drop would be tolerable for the connected design. Developing a new structure that does
not need too much bit routing will also be good. Also, making a generic block for the test pattern blocks should
be considered. If a speed drop has been seen, all the test patterns will be different by changing the code of the
generic block. This would be a better way to debug since it makes changing easier.
Since there are 256 soft bits getting into each test pattern each clock cycle, and speed of the circuit can
run up to 250MHz. It could be considered that the data speed of the design is 64 Gb/s. This speed includes
the process of both information bits and parity bits. Given the fact that each bit needs to be decoded twice,
the actual data speed will drop to 32 Gb/s. If the bit rate is also taken into account, then the data speed of
information bits will be around 27.7 Gb/s. If more decoders can work in parallel in the decoding system, it is
believed that data speed can increase. According to the goal of fitting the design into an 800 Gb/s scenario, it
is a better idea to have this decoder implemented in parallel for speeding up the process instead of using one
decoder several times, which will for sure be a further drop for the data speed of the design. If the output BER
needs to be simulated to 10−15, then it needs to decode 1016 numbers of bits. Under the current data speed, It
will take around 87 hours.
The hardware resources occupancy of having 93 flipping patterns is hard to estimate because the size of some
blocks would increase a lot. Thus, this thesis will give an estimation regarding the 42-test-pattern case. Each
test pattern occupies around 2 percent of the hardware resources of the current FPGA board. Having 42 test
patterns in the design means it will fill up almost 84 percent of the FPGA boards. Given the fact that the size
of the soft output block will increase after having more test patterns, each iteration needs its own deinterleaving
block and sorting block. Thus, a soft decoding iteration can take up to 1.5 times the resources of the desig-
nated FPGA board. A complete decoder means combining three soft-decision decoders and two hard-decision
decoders. Considering the hardware occupancy for the hard-decision decoder is much lower, the guess would be
it will take 50 percent of the board resources. In general, the overall hardware occupancy of a complete decoder
(3S2H) will occupy around six times the hardware resources it has on the board.
There are also some papers mentioned having 15 test patterns in the soft-decision decoder.
It will surely
lead to a decrease in the decoding performance, but because of the circuit size with 93 test patterns, the per-
formance it sacrifices is a trade-off. The performance can be compensated by having more decoding stages in
the decoding system. For having a soft-decision decoder with 15 test patterns, it is estimated to take around
60 percent of the LUT resources on the board.

Conclusion
The recently introduced error-correcting algorithm known as the oFEC standard demonstrates a decent capabil-
ity to rectify errors resulting from channel noise. By incorporating a staircase encoding structure and utilizing
two interleavers, it enhances the resilience of codewords against noise. The task of this thesis is to build a
soft-decision decoder on a hardware platform that can be used to decode the oFEC codeword. In combination
with the knowledge from the Chase-Pyndiah decoding algorithm and the interleaving structure introduced in
the oFEC standard, a soft decision decoder can be made.
A Matlab simulation has been developed to see if the decoding algorithm can actually have a good perfor-
mance. The result shows that after 5 iterations of decoding, the output BER can go below 10−6 when the input
BER is around 0.02, which proves that the decoding algorithm works. After that, a hardware structure has been
proposed based on the software algorithm, so it is expected that the hardware has the same decoding ability as
the software. The hardware in this thesis is a soft decision decoder that can perform one iteration of decoding.
The data speed of the information bits in the design is 27.7 Gb/s. The implementation simulation on Xilinx
xcvu9p-flga2104-2L-e shows the clock frequency it achieved is around 250 MHz. In future work, if the design
needs to be fitted into an 800 Gb/s scenario, more soft-decision decoders like this need to be implemented in
parallel for speeding up the decoding process. It is estimated that it may need 25 copies of this design to reach
the data speed requirement. However, as it has been discussed in section 6.3, if more test patterns are needed
in the decoder, then the structure may need to be redesigned to avoid the number of routing bits on the FPGA
board, which introduces speed drop during the upscale. Also, now the design has only 8 test patterns instead of
93 or 42, and it is already occupying around 30 percent of the LUTs, so it will for sure require more hardware
resources to have all the test patterns implemented. A full decoder that has 3 SISO plus 2 HIHO decoding
schemes are estimated to occupy six times the hardware resources it has on the designated board under the
42-test-patterns case. The industry has already made the oFEC decoder works on ASIC. But having it on
FPGA means a more precise result can be simulated, and it is more flexible to make changes to the blocks and
structure. One of the goals is to simulate the decoding performance to 10−15, which means it needs to decode
at least 1016 bits. Under the current design, decoding that amount of bits will take 87 hours.
Overall, the project fulfilled its objectives. It selects the Chase-Pyndiah decoding algorithm to decode oFEC
and meets the requirement of having an oFEC soft decision decoder on the hardware platform. After fixing
the routing congestion problem, there will be a better opportunity to have this design used in the 800 Gb/s
situation with more test patterns installed. It also fulfilled the goal of having a faster and more precise decoding
performance simulation. A single soft-decision decoder in hardware can simulate the output BER to 10−15 level
in 87 hours, it is believed that after having more decoders like this running in parallel. The time consumption
of the simulation will reduce more.

8.1
Future Work
Despite having most of the necessary components for a soft-decision decoder, additional enhancements and
features can be incorporated into the design to make it practical and functional.
For example, each time when a set of data is received from the channel, it needs to be deinterleaved first
by the channel interleaver before being deinterleaved on the decoder side. Neither theory nor practical hard-
ware design has been mentioned in this thesis about that. So, future work could be having this part investigated
and proposing a doable hardware implementation.
The deinterleaver on the decoder side is mentioned in this thesis, but the hardware implementation is still
undergoing. There might be some flaws in the proposed idea, such as the bit routing between the RAM com-
ponent and the interleaver control block. Bit routing is one of the main reasons that prevent the design from
achieving a higher speed. So, routing too many soft-decision data in between might not be the best solution,
even though it has a clear structure and faster data-extracting method. The future work for this part could be
having this block implemented and tested on the designated board to see which plan can fit into the high-speed
scenario required in this thesis.
The entire design also has some flaws. As mentioned in the overall performance section, there is a big speed
drop when all the blocks are connected, which is unexpected. But it is concluded that the bit routing is the
main reason causing this. In the future design, it is necessary to investigate this part to make fewer bits routing
in the middle so that the components can be placed on the board easier and bring the speed up.
It is important to consider the potential risks associated with the size of the circuit. As all flipping patterns
must be implemented in parallel on a single FPGA board, the size of this component becomes critical. Now the
size of that part has exceeded 1 percent of the total resources, which makes having all the test patterns on one
board hard. Also, when the hardware occupancy of the board reaches 75 percent, the clock speed of the circuit
would face another drop. So the next step will be reducing the circuit size to let more patterns fit into the board.
To enhance the clock speed of the design, it would be beneficial to incorporate multiple decoders running
parallel to each other, thereby processing incoming data at the same time. Additionally, pipelining the blocks
could accelerate the circuit and should be considered for future designs. However, it is important to acknowl-
edge that pipelining has limitations, as it requires more registers to achieve a higher speed. Consequently, if the
circuit needs to be replicated on the FPGA board, the hardware size of a single circuit must also be evaluated.
In such cases, it may be necessary to redesign certain blocks to reduce their size and fit them into a higher-speed
core network.
Another thing that can be done is to make a simulation system in hardware by including an encoding pro-
cess and a channel model. In this case, the decoding ability of the hardware design can be measured directly.
Having the entire system tested on a real FPGA board is also necessary. By conducting this test, the real clock
speed and data processing speed can be measured. These results will be very important if the design is put into
real use.

8.2
Another Way of Implementation
As it has been stated in section 6.3, the speed of the circuit dropped drastically after having more test patterns
in parallel. It was concluded that it was due to routing congestion. In this section, it will discuss if any other
methods of implementation can solve this problem.
One of the reasons causing the speed drop could be because the blocks are not pipelined enough.
If the
speed margin is big enough when the individual circuit is tested under 400MHz, there is a chance for the whole
system to run up to 400MHz after all the connections. Normally the margin should be about 20 percent of
the designated speed of the whole circuit. Meaning each block should have the capacity to run under 480MHz.
But for some of the blocks, such as the Sorting block and the Weight_2 block, it is not easy to be pipelined
because the structure of the block needs to be changed entirely for performing pipelining. In the Weight_2
block, since it needs to select the corrected data bits, it separates the whole data set into small chunks to have
smaller multiplexers to select the right bit. In the current design, the size of the multiplexer is four to one,
which is small enough. So in the next version of the design, instead of using multiplexers in all the stages, it
could use the shifting method in the early stages of separation. Then in the last stage of selection, it can use the
four-to-one multiplexer to finish selecting the value at the correcting locations. Combining these two methods
saves the usage of multiplexers and could be able to boost the speed.
There is another key block that needs to be improved.
The size of the Soft Output block changes when
more test patterns are implemented in the design. This will be devastating to the design since the hardware
occupancy and circuit speed can never be guaranteed if the number of test patterns is not fixed. Thus, one
alternative way to implement this block could be to separate this block into different subblocks. For example,
having a comparison subblock for each of two output codewords from the test patterns. And the following
stage will take two outputs from the previous stage to perform the comparison, just like the tree structure it
has now in the block. However, having it into many subblocks could help to measure the hardware occupancy
more precisely. After knowing how many test patterns are needed in the decoding system, the size of the whole
comparison process can be estimated easily. That gives an important point to separate this block into smaller
subblocks.
Furthermore, an American telecommunication company Acacia proposed another way to implement the output
calculation[14]. As introduced in section 5.7, for generating the final soft-decision output, it needs to go through
all the bits in the competing codewords to find the best solution for each bit. That means the final soft-decision
output is generated by combining all the codewords instead of selecting one of the decoded codewords that are
already there. After all the comparisons, the algorithm takes the best codeword with 0 and the best codeword
with 1. Then the final output is the difference between the weights of these two codewords. The extrinsic
information is calculated using the final output minus the original input. When the extrinsic is used in the
following iterations, it is scaled by a constant α.
However, in Acacia’s code, they take a number of possible codewords from the hard-decision decoders with
different test patterns. But they only use the two best of them, not for each position but for the complete word.
By selecting an entire codeword, it does not have to conduct comparison bit by bit, which saves a lot of com-
putational power and is more friendly in hardware implementation. It uses the weight value of the second-best
codeword to subtract the weight of the best codeword, which leads to a positive weight value marked as δ. Then
it goes through all the positions of those two codewords to check if these two codewords agree with each other.
If the value of the bits at a certain location is the same, then the soft output of this location will be adjusted to
the value of δ, and the sign is based on the sign of the bit they agreed on. If the codewords do not agree on the
value of some bits, then the best codeword needs to conduct some checks with the original input. If they agree
to this position, the output is the input adjusted by a constant αd times δ minus the input. The adjustment
has a sign corresponding to the sign of the input. If the best codeword disagrees with the sign of the input for
this position, the output is the input adjusted by a constant αd times δ plus the input. The adjustment has a
sign corresponding to the opposite sign of the input.[15]
If this method is applied to the current hardware design, it can at least enhance the Soft Output block. Now
the size of the Soft Output block is troubled by the number of test patterns in the decoder. After using this
method, it only needs to compare the weight value to select the two best codewords. Then, the comparison
between those two codewords can be conducted directly instead of using the cumbersome tree structure. The
performance of this method is supposed to be worse than the one that is applied in this thesis since it does not
consider the best result for each bit. But according to the introduction from Acacia, the decoding result of their
method is surprisingly good and even satisfies the estimated decoding ability of oFEC. In the next version of
the code, it might be a good idea to try to convert their algorithm into a hardware implementation.

Appendix 3: Keywords explanation
Error Correcting Code (ECC): An error correcting code (ECC) is an encoding scheme that transmits messages
as binary numbers, in such a way that the message can be recovered even if some bits are erroneously flipped.
They are used in practically all cases of message transmission, especially in data storage where ECCs defend
against data corruption.
Codeword: It refers to the output produced by the encoding process for a given set of data, typically represented
as a sequence of bits. The encoding process takes the original data, often called a message or information word,
and adds redundant bits to it to form the codeword. This addition of redundancy allows the codeword to have
specific properties that enable error detection and correction during transmission or storage.
Braided code: It refers to Braided Block code. It operates on continuous data streams and is constructed
by the interconnection of two-component block codes.
Hard decision decoding: Hard decision decoding algorithms, which means that for every input and output
signal, a hard decision is made whether it corresponds to a one or a zero bit.
Soft decision decoding: Unlike the hard decoding algorithm, the inputs to it may take on a whole range of
values in between. The information besides the sign of the value is called extra information. This extra infor-
mation indicates the reliability of each input data point and is used to form better estimates of the original
data. Therefore, a soft-decision decoder will typically perform better in the presence of corrupted data than
the hard-decision decoding algorithm.
Additive white Gaussian noise (AWGN): It is a basic noise model used in information theory to mimic the
effect of many random processes that occur in nature. The modifiers denote specific characteristics: Additive
because it is added to any noise that might be intrinsic to the information system.
Gb/s: Gigabit per second.
GB/s: Gigabyte per second.
Scrambler: In telecommunications, a scrambler is a device that transposes or inverts signals or otherwise
encodes a message at the sender’s side to make the message unintelligible at a receiver not equipped with an
appropriately set descrambling device. Whereas encryption usually refers to operations carried out in the digital
domain, scrambling usually refers to operations carried out in the analog domain.
