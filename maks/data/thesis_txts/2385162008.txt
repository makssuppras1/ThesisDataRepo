TECHNICAL UNIVERSITY OF DENMARK
Abstract
DTU Compute
MSc in Digital Media Engineering
by Carsten M. Rosenkilde Nielsen
In this thesis we develop a docking window framework (DWF) for managing windows in
a professional game editor application: Io-Interactive’s Glacier 2. We analyse existing
DWF solutions and conclude that they do not satisfy the user requirements set by IO
Interactive.
We then design a new DWF based on a principle of minimising layout
changes not directly requested by the user.
We describe our implementation of the
framework in WPF and its integration in Glacier 2. We exemplify the practicality of
our design through speciﬁc use-cases. By comparing the new framework with existing
solutions, we ﬁnd that our DWF better maintain the user’s layout, thereby satisfying the
requirements set by Io-Interactive, with the aim of improving work-ﬂow and achieving
more design ideals.

Chapter 1
Introduction
This thesis investigates the current state of docking solutions in game editors, as well
as related applications. We propose the static layout design to improve the precision of
interactions related to modifying the layout, which we validate by implementation.
Docking is a user interaction in a WIMP (windows, icons, menus, pointer) interface that
attaches two or more windows together. Interacting with a group of docked windows will
then aﬀect all of the windows docked together. Depending on the docking framework,
actions such as moving, resizing, closing, etc. now aﬀect the whole group, as if they were
a single window. The reverse operation, undocking, returns a window to the normal,
ﬂoating state.
The docking interaction was introduced as part of window managers (WM’s), in order
to help with the growing complexity of tasks performed on the computer. It became
common to use multiple resources simultaneously, and so workspaces became more clut-
tered. Docking was an attempt at dynamic layouts, to tailor the workﬂow to the user’s
needs. Some features of modern WM’s have kept similarities to docking like the Aero
snap[1] feature in Windows 7, 8 and 10.
The early WM’s were mostly split between two approaches: tiled and overlapping[2].
Tiled systems place windows in a non-overlapping grid structure with no wasted space[3,
4]. Overlapping is probably the more known version today, as it is the one used today
in commercial systems such as Windows and MacOS, where windows are independent
and free-ﬂoating.
Tiled systems had its merits however, and has been shown to be superior in some cases.
Speciﬁcally, tiled approaches spent less time constructing a layout required for a task
with many diﬀerent window-resources required[5, 6].
1
==PAGE:9==

2
This is very relevant for large-scale editors using an array of diﬀerent tools, such as those
related to editing text, images, video, 3D-models, code and so on. Such editors have
adopted a merge of tiled and overlapping approaches using docking, like a mini-WM
inside the application. We will call this a docking window framework (DWF). Examples
of such applications are Microsoft Visual Studio[7] and Photoshop[8].
Level editors for video games were no exception to this development. Video game devel-
opment involves many resources and disciplines, requiring assets from 2D- and 3D-artists,
sound-, game-, and level-designers, AI- and gameplay-programmers, and more.
A level editor is therefore required to handle many diﬀerent assets and to this end
contains a lot of diﬀerent tools and resources. In the end, it can become quite cluttered
to work with solely ﬂoating windows, and is a great candidate for a DWF.
Docking in DWF’s required some fundamental changes from its original WM form. One
of the most profound of these, is that instead of attaching ﬂoating windows to each
other, windows are now docked into other windows with pre-deﬁned containers, which
we call a docking space. This has altered a lot of the interactions and user-experience
with docking, a lot of it diverging from normal windowing interactions. The concept of
a docking space is further explained in section 2.1.1.
In this thesis, we seek to identify and improve the undesirable behaviors brought with
the transition to a docking space approach. The thesis was done in collaboration with
Io-Interactive (IOI), and we are integrating their proprietary editor, Glacier 2 (G2), as
a case study for the design and implementation of a DWF prototype.
1.1
Project Description
The project stakeholder is IOI, who wishes to improve the level editor for their engine,
G2. IOI has tried several commercial solutions on the market which they have found
inadequate for a variety of reasons, so they need to build their own.
Speciﬁcally, IOI needs a docking system, where docked windows are always kept static
except when the user directly moves or resizes them.
As it turns out, this is quite
diﬀerent from the norm, where resizing a window can aﬀect others unintentionally. IOI
has found this to be a source of frustration and time waste through ad hoc reports from
their employees.
IOI has developed a set of speciﬁcations and design goals they desire from the new
docking framework. We distilled the crux of the problem into the following hypothesis
for the project goal:
==PAGE:10==
Introduction
3
It is possible to build such a DWF, using the docking space approach, where a docking
window is only resized either by direct user interaction of dividers or resizing the
host window.
Brieﬂy explained, the dividers are the edges between docked windows, and the host
window is the window containing the docking space. Both of these terms are explained
in section 2.1.1.
It turns out that this is not a trivial problem as it raises a number of new use cases
and technical challenges. The key diﬃculty is the direct user interaction limitation, i.e.
moving one edge should not aﬀect others. Resizing the host window is an exception,
as it will resize the docking space that frames the docked windows, something we also
expand on in section 2.1.1. To our knowledge, such a design has not been constructed
in a DWF before.
There are three speciﬁc use cases we want to address in order to verify the hypothesis
and improve the DWF to ﬁt IOI’s needs. These are dragging dividers, undocking and
resizing the host window. In these three cases the existing DWF solutions we examine
fail to meet the conditions of the project, as unrelated docking windows can be resized.
Beyond this, the speciﬁcations also included converting the editor from using Win-
dows Forms (WinForms) to the more recent Windows Presentation Foundation (WPF).
IOI was also interested in visual stylistic improvements and wanted some fundamental
changes in design, diﬀerent from the oﬀ-the-shelf frameworks they previously used. Fi-
nally, they were interested in controlling the source code to the docking framework, as
they were using an external solution that proved diﬃcult to maintain or modify. How-
ever the focus of this thesis will be on the hypothesis, as these details are tangential to
the greater scope of the project. The full project speciﬁcations can be found in Appendix
A.
Finally, we want to make it clear that user experience (UX) is not a focus for this thesis.
Rather, the focus is on designing a system that achieves the ideals of the speciﬁcations.
The crucial next steps of the development of the DWF, will of course be evaluating user
performance and user experience, however this is outside the scope of the project.
1.1.1
Motivation
The Docking framework gives the user control to shape their workstation. Allowing
them to structure the layout and therefore their workﬂow themselves, has been shown to
==PAGE:11==
Introduction
4
improve productivity a great deal when working on complex tasks or tasks in parallel.[5,
9]
Docking frameworks are usually a balance between automation, control and ﬂexibility.
They aim to give the user as many options as possible, but also aim to reduce repeated
and tedious tasks.
Little academic research has been published on this topic; we discuss this in section 1.2.
Additionally, it is also an area where even small improvements in the workﬂow could
pay dividends in time saved for IOI in particular, as most of their staﬀuse the Glacier
2 editor many hours a day.
The static layout we seek to implement, is possibly a step towards reducing redundant
automation done by docking frameworks while increasing the precision of user interac-
tion.
1.2
Related Work
The concept of a window manager was introduced with the release of the X Window
System, and has been the subject of some research. Most of the older works focus on
analysis and comparisons of the tiled and overlapping approaches and the conventional
WIMP systems most in use today[2] (See cite and the references therein).
The more recent work focus on novel alternatives or non-WIMP window managers[10].
Others have of course previously experimented with improving the spatial like this thesis.
Examples include the importance-driven composition of windows by M. Waldner et
al.[11] and the constraint based window manager SCWM by G. J. Badros et al.[12] A.
Savidis et al.[13] presents a compositing system that allowing the use of multiple, nested
window managers. 3D window managers have been introduced; G. Robertson et al.[14]
and their Task Gallery is an example of an early pioneers.
There is also interesting studies on workﬂows in general. D. R. Hutchings et al.[15] test
the eﬀects of multiple moniters and their interoperality with window managers. B. A.
Bell et al.[16] presents a 2D space management approach for use in both 2D and 3D user
interfaces.

Chapter 2
Theory
Before we delve into analysis, we will need to introduce the terminology and concepts
we will use in the following chapters. The main focus is on deﬁning docking and docking
window framework. We also cover the WPF technology that is used in implementation,
but is also relevant in analysis. Throughout we deﬁne our use of the terminology.
Some of the content will be repeated in later chapters, but this provides an overview of
the ﬁeld.
2.1
Docking
In this section, we will explain the terminology and common features of a docking
framework.
Docking is the act of attaching two or more windows together, such that they behave as
a single window. This is usually done such that one or more of their edges are always
connected and move together.
A window that can be docked we call a docking window. Since we focus on application-
wide docking rather than WM’s with docking, the applications have to implement their
own kind of docking window. This means that docking windows are usually not normal
windows, and in some cases just have window-like behavior without actually being one.
This distinction is very important in implementation of docking, but is also relevant in
some other cases.
Docking operations are conducted much in the same way as a drag and drop operation.
The docking window has some kind of handle, usually a header resembling a title bar.
Dragging the handle will start moving the window with the cursor. Depending on the

the diﬀerent components.
design, the window has to be released over some area or widget to dock the window.
Figure 5.8 shows a screenshot of this.
A docking window which is not docked behaves like a normal ﬂoating window in a
overlapping WM. In some cases the docking window appearance may change, or even
become embedded in a separate window that handles the ﬂoating window behavior. In
the latter case, it becomes ambiguous which window is referred to as the ﬂoating window,
but we generally mean the docking window.
We call an application that implements docking a docking window framework (DWF).
2.1.1
Docking Space
The original WM docking systems docked by attaching ﬂoating windows together, such
that they are placed side by side.
DWF’s however generally dock windows into an
existing space inside certain windows. To our knowledge this space has no agreed upon
name, but we call this a docking space, and such a window containing it a host window.
Figure 2.1 shows an annotated screenshot of these.
The docking space behaves like a tiled WM. The space becomes partitioned as further
windows are docked, such that each window has its own, non-overlapping space. The
edges between windows are called dividers and act as the resize handles of normal
windows. Docked windows are embedded into the docking space like any other element
in a window, and are moved and resized with the window. Resizing the docking space
will force some or all of the docked windows to resize as well, something we discuss
further in section 3.3.
==PAGE:15==
Theory
8
While the windows are non-overlapping, multiple windows can be docked into the same
space via tab groups. These function like normal tab groups, where only one window is
visible at a time.
We call the contents of the docking space the docking layout. We refer to this when we
want to exclude all the external elements of the host window like toolbars and such.
In the scope of an application with a DWF, there is usually one large main window that
is a host window. Depending on the DWF, ﬂoating windows can be host windows, and
therefore also hold their own docking layout. Otherwise, the only docking layout will be
in the main application window.
This summarizes the core mechanics that make up a complete, but simple, DWF. There
are still some additional features that are important to discuss, but they vary a lot
depending on the implementation. We will elaborate on these elements in Chapter 3,
where we will examine some speciﬁc examples from existing applications.
2.2
WPF
In this section, we will cover the basics of WPF, which are important for us in the
implementation. The reader may skip this section if you are already familiar with it.
Windows Presentation Foundation (WPF) is a framework released in 2006 for rendering
MS desktop applications, written in C#. It has a large library of all the standard MS
Windows controls, such as buttons, textboxes, windows, and so on. Even though WPF is
old, it is still widely used. Its predecessor is Windows Forms, usually called WinForms,
which it has a lot of similarities to.
We will use WPF to implement the DWF, and it is commonly used in MS Windows
applications.
There are speciﬁc patterns and behavior one can ﬁnd across diﬀerent
applications, that stems from using standard library components from WPF. This is
brought up a few times in chapter 3.
2.2.1
Hierarchies
The components that make up a WPF control, usually referred to as visuals (as they all
derive from the visual class), are all part of a tree-hierarchy called the visual tree. The
root element is always a window, and therefore there are multiple, separate hierarchies.
==PAGE:16==
Theory
9
The elements share the z-order used by their window-handle, so the hierarchy order is
determined by window z-order - if one element is in front of another, it must also be in
front of all other elements in that tree.
A XAML page (see 2.2.3 for more) speciﬁes the internal hierarchy in a single control.
Whenever this control is instanciated, it introduces its internal hierarchy into the visual
tree as children. Some of those children may have their own hierarchies, which are also
instanced, and so on.
There is a simpliﬁed tree called logical tree, that only includes the top-level controls that
has been instanced by the application. This can be a helpful abstraction to match the
tree with the logic, or work with elements as black-boxes, however usually it is better
to go with the more correct visual tree.
The tree structure has a lot of implications in implementation. Controls usually have
access to everything related to their children, but cannot easily access their parents
without a lot of casts and checks.
Working with WPF therefore favors a top-down
approach to designing the interface. The most common example is that un-childing is
easy, but un-parenting cannot be done generically.
2.2.2
Routed Events
A big part of WPF is routed events, which traverse the visual tree until they are marked
as handled. Controls can listen for diﬀerent events depending on what they need. A
common example is listening for mouse- or keyboard input. Marking those events as
handled ensures that only one control will act on such input. Some events are raised
automatically by WPF (such as input related events), while others can be raised by
controls themselves.
There are three diﬀerent kinds of routed events, which dictate how they traverse the
visual tree. Bubbling events move upwards from the raiser. Tunneling events moves
from the root to the raiser. Direct events only trigger the raiser.
Routed events can be used to ﬂag changes or send data throughout the tree, and is one
of the easy ways to traverse upwards in the tree.
2.2.3
XAML
XAML (Extensible Application Markup Language) is an XML based language, used in
WPF to easily deﬁne a control’s visuals, including child controls and their layout.
==PAGE:17==
Theory
10
WPF uses the Model-View-Viewmodel (MVVM) pattern. Brieﬂy explained, the visuals
of the application, the view, is separated from the backend structure, the model. The
connector between these two is the viewmodel. This primarily helps keeping the imple-
mentation structured properly. While the viewmodel is deﬁned by WPF, the view is
deﬁned by XAML, and the model is deﬁned in C#.
XAML uses templates to deﬁne a generic view for a WPF control, that could later be
overridden for a diﬀerent look. This makes it quite ﬂexible, as a single viewmodel could
be used for many diﬀerent views. A similar concept is styles, which changes variable
values, such as margins, colors, fonts and so on.
Templates and styles are important when implementing ﬂexible controls. They are key
to implement MVVM-friendly controls and also easily allow for skinning the editor in
diﬀerent visual styles.
2.2.4
Dependency Properties
Working with MVVM opens up for challenges regarding moving data between the dif-
ferent parts. Without simple, consistent and safe implementations, it quickly devolves
into an un-maintainable mess, or breaks the MVVM pattern.
Dependency properties are special properties that solve this and other issues. Depen-
dency properties are deﬁned in the viewmodel and can be used like regular properties,
however they can also be referenced by the view (XAML pages).
Furthermore, they can call a callback method whenever their value changes, such that
viewmodel and model can be easily synchronized.
The true value behind dependency properties however, comes from binding with other
dependency properties. Binding dependency properties synchronizes their values, either
one- or two-way. This makes it easy to synchronize values between view and viewmodel,
without breaking the MVVM pattern. The view can update its visuals according to the
viewmodel, or the viewmodel can retrieve values gained the view (such as user input).
Moreover, diﬀerent components in the application can bind properties together. Using
this, we can easily update data throughout the visual tree whenever a change happens,
and with very little implementation required.
A textbox is an example of binding data between the view and model, where the user
input is captured in the view and is processed by the model. Tabs is an example of
binding between components, as the tab group body needs to be updated whenever a
new tab is selected.

2.2.5
Layout
To layout and render the visual tree, WPF has a process that goes through three steps.
First the entire visual tree is measured. This involves a visual determining how much of
the available space it desires to use. The measure method is recursively called from the
root to the children, such that a parent control can use the results from its children to
measure itself. The measure method takes the possible available space as an argument,
and then returns the desired size of that. The desired size is required to not exceed the
available.
After the measure calls has been processed, arrange is called in the same manner.
Arrange is given available size as an argument and returns the ﬁnal size of the visual.
When it is called on a given visual, it is expected to determine how it wants to be
displayed in the given space.
Finally, after all arrange calls has been processed, the visuals can be displayed. The
RenderSize ﬁeld of each visual are changed to match the arrange values, which prompts
the OnRenderSizeChanged method to be called in the same order as the others.
This process is important when implementing controls with custom layouting needs,
which is quite relevant for a DWF.

3.1
Docking Layouts
Before we delve into the speciﬁc elements of a docking framework, we should examine the
overall layout of an application with docking. We also give an outline of the components
in use.
All the applications we examined used the docking space (see: section 2.1.1) approach
and had their main window as a host window, and then usually additional ﬂoating
windows could be constructed.
A given docking layout partitions its space to contain and separate all the docked win-
dows in the related docking space. The windows never overlap and they all retain the
rectangular shape like normal windows. The edges between windows become dividers,
an element the user can drag to resize the neighboring spaces, much like the edges of a
normal window. There are many kinds of behaviors related to these, which are discussed
in detail in section 3.3.
In the following sections, we will describe the two common ways of deﬁning the division
of space in a docking layout.

3.1.1
Nested Layout
The more common layout is what we will call the Nested Layout. The space is recursively
partitioned horizontally and vertically into rectangles that contains individual docking
windows or tab groups. It can be modeled as a k-dimensional tree with two dimensions,
such that the space is cut into rectangles. Figure 3.2 shows a simple representation
of this with a matching tree structure. Figure 3.1 shows an example of this in Unreal
Engine.
Each partition is deﬁned by a single divider, such that the divider position is the partition
position.
Moving a divider then translates into changing the position of the related
partition.
This approach allows every possible conﬁguration of non-overlapping rectangular shapes
to be constructed. Notice however, that a given partition-edge can border any number
of rectangles. A given divider could therefore aﬀect many windows when moved, and is
not always just placed between two.
This means a user is sometimes unable to resize a window in a given direction, without
inadvertently aﬀecting other windows.
Obviously, in a limited space with non-overlapping rectangles, enlarging one must take
up the space of another, but in this case we can aﬀect many windows.
This is seemingly the only limitation of this layout. However it does not seem very
disruptive and the solution is quite simple, so it is not surprising this layout is the
common choice.

3.1.2
Document Layout
Some docking frameworks use what we call a document layout, which has a special
main document area the entire layout revolves around. This is often used when the
application has some speciﬁc windows that are of higher priority than others, such as
the text ﬁles in text editors. In this setup, the main document area takes precedence
before other windows. Resizing the host window for example, will usually only aﬀect
the main document area while keeping other windows at static size.
Docked window placement then becomes deﬁned as a relation to the main document
area, and the layout can be modeled as a nested layout with the main document area
as the root.
It is important to note that the layouts can be identical to a nested layout - the main
diﬀerence is how windows are docked and how the window reacts to being resized.
Therefore, the main document area can be used if users are expected to prioritize some
window(s) more than others when constructing a layout. Visual Studio is an example of
this, where the main document area usually contains code ﬁles. Windows can be docked
either inside or outside the main document area. When resizing the host window, only
windows inside the main document area will be resized, if possible. Floating windows
in Visual Studio just use the nested layout.
Unity is an example using nested layout, even though some windows like the scene view
could be considered more important than others.
In case the application has ﬂoating windows with their own docking layouts, they usually
behave like a nested layout.
3.2
Docking Windows
The main focus of the docking framework is the docking windows(see: section 3.2. As
the DWF’s we are examining are parts of applications rather than general WM’s, only
the windows they own can be docked. In addition, they are usually diﬀerent from normal
windows in MS Windows, both mechanically and visually. Therefore we are exclusively
looking at docking windows related to DWF’s.
As we brieﬂy mentioned in section 3.2, there is a very important distinction between
normal and docking windows, as a docking window is technically not always a window.
==PAGE:23==
Analysis

We simply call it a window, because it often behaves as one, and because traditionally
docking was done with actual windows.
This diﬀerence is important for two reasons. Firstly, while docking windows emulate
normal window behavior, they are usually embedded in a standard window (the host
window) when docked, sometimes even when ﬂoating. They behave more like a panel,
while the host window contains usual window behavior. Secondly, in WPF windows are
considered roots of their visual trees. There is no native support for windows as child
elements, so normal windows cannot be embedded into other windows.
3.2.1
Headers
While the contents of a docking window depends on the application, they need some
visual representation in the framework - at least enough to be interacted with regards
to docking.
This is often done by a header above the contents of the window, which mimics the
functionality of a normal window title bar. It usually also contains buttons like a window
title-bar.
However, sometimes there is both a header and title bar visible, depending on both the
window state and the DWF.
Unity is an example where the visuals are static.
Here the docking windows never
change visually whether they are ﬂoating or docked, always using the same header. This
is visually very similar to WM docking.
The docking system by Shibata et al.[9] and also in Visual Studio shows ﬂoating docking
windows resembling normal windows with title bars. When docked however, a new host
window wraps around the docking windows with a new title bar. The docking windows
also retain their separate headers (which Shibata et al. calls ﬂoating bars), although
they change slightly in form and function. Figure 3.10 shows the two forms of Visual
Studio ﬂoating windows.
As an aside, Kandogan et al.[18] visualizes the whole hierarchy with headers. This allows
users to undock and dock at any point in the hierarchy, giving a lot more options to the
user. Observing the screen-shots of their WM however, it is very expensive in screen-
space and introduces a lot of complexity. If we expect docking to be a rare operation
performed by the user, it is unnecessary to display the hierarchy at all times. While it
is useful in some instances, the gain is quite small compared to the cost in screen space.

In all the cases we examined, the hierarchies were almost invisible and only had their own
headers visible. It could be interesting to explore other ways to allow users to dock and
undock hierarchies together. There was one group-undocking method however, which
we describe in the next section.
3.2.2
Tab Groups
Tabs are closely related to the subject of headers. Often associated with web browsers,
tabs allow for quick switching between windows that share the same screen space.
Tabs are a core feature to docking frameworks. Combined with the ability to deﬁne
the layout, tabbing allows the user to group similar functions together, or provides the
ability to easily hide functionality with workﬂow changes.
Depending on the DWF, either tabs replace the header or they are separate. The latter
is most often the case, where tab and header functionality are distinct. Usually because
the tab manipulates the single related window, while the header manipulates the whole
tab group. This allows the user to undock the entire group at once using the header, or
just a single window using the tabs.
On the other hand, merging headers and tabs reduce clutter and spends less screen
space. If the separate functionality is not required, merging them is the most eﬃcient
solution.
Visual Studio is an example where both cases exists. In Visual studio with its document
layout, headers and tabs are either separate or merged depending on how they are docked
related to the main document area.
3.3
Dividers
The edge handles between windows are what we call dividers. They are handles that
the user can drag to resize the windows, and are equivalent to the resize handles around
a normal window.
In a WM, the combined edges of windows act like a divider, such that both are moved
together. In a DWF however, the dividers are a single element and separated from the
windows.
Both WPF and WinForms use a component called splitter, that is functionally equivalent
to a divider. These are more general however, and are used in any layout with dynamic
resizing between elements contained in a grid (such as a table).

we can refer to ancestors and descendants for dividers. Usually, we use the super- for
ancestors and sub- preﬁx for descendants.

dividers (section 5.2.1 expands on this). However, it turns out that this detail shows
in the interface of the DWF’s we examined. Because of how we deﬁne the containers,
dragging a divider essentially resizes a container and some of its sub-containers. How
this aﬀects sub-dividers, depend on the divider behavior.

3.3.1
Elastic Dividers
Elastic dividers determine their position proportionally to the holding container’s size.
If the container is resized, this will therefore move all internal dividers relative to the
container, but will keep the proportions between sub-containers constant[18]. Microsoft
Visual Studio 2017 is an example of an application with this in use, see ﬁgure 3.4.
This is probably the most common divider behavior and splitter behavior as well. How-
ever, it is not necessarily the best solution, and has been found to be confusing to users.
Speciﬁcally the problem is that moving one divider will move any dividers that are in the
sub-containers. Shibata et al.[9] found that this was confusing users with which divider
they are actually interacting with.
Depending on how the hierarchy is implemented, neighboring dividers may be considered
elements of sub-containers, the super-container or on the same level. If this is incon-
sistent, it may be even more confusing. This is the case in Visual Studio for example,
where neighboring dividers do not move each other, but sub-containers with dividers in
the same orientation are moved. See ﬁgure 3.5 for an example where dividers are static
to movement.
It is also problematic that all dividers move when the container resizes, as the maximal
amount of dividers change position (both relative to screen and the window). This occurs
for example when resizing the host window, shown in ﬁgure 3.6. This makes it more
diﬃcult to conﬁgure a layout to exact sizes, as any movement will aﬀect all descendants.
Whenever the layout is changed, a user has to maintain the sizes of all windows they
want to be constant.
Elastic splitters are natively implemented in the standard WPF library, so they are
very easy to use. This might be the explanation for the common usage rather than a
preference to this behavior. This is also exactly the kind of behavior we are trying to
avoid with regards to our hypothesis.

creates more areas to dock in. Using that, the user can partition the docking space as

3.4.1
Undocking
Undocking windows is the process where a docked window is removed a docking layout,
usually moving into a ﬂoating state.
As mentioned in section 3.2.1, headers usually function as a grip for the windows, just
like title-bars for normal windows. In docking frameworks however, they undock window
when dragged if they are docked.
When a window is undocked, its space is automatically ﬁlled by a neighboring window
(unless there are no other windows). The divider that is part of the immediate super-
container is removed, such that the other sub-container now inherits all of the space
that was previously split. Figure 3.8 shows an example of this.
In case of multiple neighboring dividers with the same orientation, it would be ambiguous
which divider would be removed from the user’s perspective.
This is because some
layouts can be constructed in multiple ways. Figure 3.9 shows an example of this.
Visual studio solves this by sharing the space between all nearby dividers in the same
orientation. By nearby, we mean either ancestors or descendants, that does not have a
divider of the opposite orientation between them in the tree.
Unity does similar, but only shares the space between the area to the left and right or
above and below depending on the divider orientation.
In either of those cases, even though the same layout could be constructed in diﬀerent
ways, the outcome is still the same when undocking. On the other hand, the undocking
aﬀects more windows.
Tab headers usually also function as an undocking grip. In Visual Studio for example,
dragging the header will undock the entire tab-group the window is part of, if any.
Dragging the tab header though, will only undock that window in particular.

Tab Docking
As mentioned in section 3.4, windows can be docked into tab-groups. In all the cases

end of the tab group.
Every case examined also had the ability to dock directly into a speciﬁc position in

browsers.
Instead of using widgets, the user can dock a window by dragging over the tab headers.
The new window will be placed in the group, such that the header will be placed where
the user docked the window in the tab headers. This allows the user to easily order tabs
and visualize the end result prior to docking.
Using this feature, users can move tabs into any order they desire. Undocking and dock-

with another we call tab sorting. Instead of immediately undocking a tab when the user
drags it, tabs can be dragged into new positions directly. This makes ordering the tabs
even easier, and skips the entire docking procedure.
If the user drags the tab far enough away from the tab-group, it will return to normal
undocking procedure.
Chrome is an excellent example of tab sorting, where tabs are animated to smoothly
move into new positions as a tab is dragged.

our own DWF speciﬁcation in order to fulﬁll IOI’s requirements, as well as verify our
hypothesis. Since we are not concerned of other DWF’s anymore, we will from this point
refer to it as the DWF.
In this chapter we will describe the DWF design we constructed for this project. We
will use existing solutions from chapter 3, as well as introduce some of our own. The
goal is to develop a set of speciﬁcations that we can use to implement a prototype later.
4.1
Design Goals
Recall that our goal is to create a static layout, where windows are only resized or moved
when the user directly interacts with them. The rule for our design is then, that no other
interaction should change the size of windows.
From our analysis, we have identiﬁed the points where windows can be resized and
moved, and thus also found the speciﬁc cases where the rule is broken.
These cases are moving super-dividers and undocking. All the case-studies from analysis
could resize un-involved windows during either of these interactions, and we therefore
have no known solutions to these cases yet. There is also the case of resizing the host
window, which should resize some but not necessarily all docking windows.
In the following sections, we deﬁne the DWF by its components in a similar manner to
our analysis. In addition to this, we will address these three speciﬁc cases, by introducing
new components and ideas.

4.2
Docking Layout
We decided to use the nested layout, since it was both simple, ﬂexible and common.
Since there is no need for a main document area, we did not need to use the document
layout.
Floating windows each have their own docking space, similar to Visual Studio. This
allows them to also contain complex docking layouts, using the nested layout approach
again for consistency. They can of course still just hold a single docking window like
normal.
4.2.1
Minimal Interference Problem
We considered improving upon the nested layout, but it required solving the minimal
interference problem, mentioned in section 3.1.1. Speciﬁcally in the nested layout, it is
sometimes impossible to resize one window without aﬀecting multiple others. This is
because the windows share edges.
This is somewhat sub-optimal, as it gives the user fewer options. The most optimal
layout, would be the one where all windows could be resized freely, while still being
non-overlapping and rectangular. However, this becomes diﬃcult to achieve.
As we mentioned, resizing one rectangle must aﬀect at least one other, given that they
share the same limited space. The optimal solution would must then be one with min-
imal interference of other windows. That is, resizing any edge aﬀects at maximum two
windows: the one being reduced and the one being enlarged by the resize.
It turns out that this is diﬃcult to solve optimally with strictly rectangular shapes and
non-overlapping windows.
We can demonstrate this with a simple example. To solve the problem optimally, we
must be able to move any given edge between two windows. Assume we have such an
approach. Assume we have a setup like in ﬁgure 4.1. A large window ”A” neighbors
two smaller windows, ”B” and ”C”, on the same side. The edge A has to both B and C
is divided into two sub-edges, one bordering A and B and the other A and C. With the
assumed approach we can move either individually without moving the other.
No matter which way we would move one of the edges, we would end up in a situation
where we introduce new edges between windows, and need new areas to keep windows
rectangular. Not only are we increasing the complexity of the layout, but we are also
creating wasted space to keep them rectangular.

4.3
Static Dividers
Recall the two divider behaviors we described in the analysis: the elastic divider and
the top-left bound divider.
The divider behavior determine a great deal of how the
docking window’s position and size themselves. Unfortunately, the existing solutions
are inadequate to create the static layout we need.
The elastic divider keeps proportions of neighboring containers constant, but this re-
quires both moving and resizing all of the docking windows.
==PAGE:41==
Design
34
The top-left bound divider does mostly keep window sizes constant, but all descendants
to the right and below are moved when the divider is moved. If the host window itself is
resized from the left or top, all windows are moved and the bottom-most or right-most
are resized instead.
The problem with both of these is that resizing any container will aﬀect all sub-container
positions or sizes.
To avoid these pitfalls, we introduce a new hybrid-behavior called the static divider.
This divider keeps its screen position constant instead, so it is no longer relative to any
of the contents in the host window. This makes it compatible with nested hierarchies,
as the dividers will no longer aﬀect each other as their positions are not related to their
parent-containers. Moving any divider will never move any others, achieving a more
static layout.
Additionally the host window will now be context-aware to resizes, as only the docking
windows bordering the manipulated sides will be aﬀected. Resizing the left side of the
host-window for example, will resize only the left-most docking windows, and the same
goes for any other side.
The reason we call it a hybrid-behavior, is because there is an exception when the user
moves the window. In this case, the dividers should move with it, behaving like elastic-
and top-left bound dividers.
Otherwise, moving the window would keep the dividers static to the screen, resizing all
docking windows bordering the edges in a confusing manner. Moving the host window
should never aﬀect its contents, including the layout, as this is not the point of the move
operation. It would also be inconsistent with all the cases examined.
Like with the other behaviors, the movement of the static divider should be limited
by other dividers, such that dividers cannot be dragged past each other. Unlike other
behaviors however, static dividers also has to account for sub-dividers.
Both of the
other behaviors avoid this problem, as the sub-dividers are always moved out of the way
automatically.
So in addition to the dividers having consistent positioning no matter where they are
located in the nested hierarchy, they also have consistent relations between ancestors
and descendents. This means that visually, the entire layout is ﬂat - that is that no
hierarchy is apparent to the user.

4.3.1
Boundary Problem
Recall the problem described in section 3.3.2, where top-left bound dividers could be
caught outside their container boundaries. This renders part of the window invisible,
which is of course unacceptable for the DWF to perform.
This occurs because the
position is not related to the container’s size, which incidentally is also the case with the
static divider. Unlike top-left bound dividers however, this can only occur when resizing
the host window, not any internal divider.
In the aforementioned section, we also listed some solutions. The only solution we know
of, where windows are not resized, is by having a minimum size for a window. This is
set to the smallest bounding box that covers all dividers, such that none of them can
be caught outside. Unfortunately, this was found too restrictive, and a simple case of
shrinking a window can take many interactions.
We found the simplest and least intrusive one to be bringing the divider into view
immediately whenever it would be caught outside. This does have some consequences
however. As we originally noted, the layout could unintentionally be changed, and is
not always easily restored.
More importantly however, we are introducing a new way to resize the docking windows,
which could go against our hypothesis. The argument here, is that the host window edges
also counts as the edges of the docking windows that border it. It is already the case
that enlarging the host window in a direction should also aﬀect the windows on that
side. Shrinking the host window then, must be the same. The diﬀerence is that once a
bordering window reaches minimal size, the next window in line must be aﬀected.
In the end, we accepted this as a case where windows could be resized. Restricting
window resize depending on the layout was too confusing and frustrating. Moving the
dividers automatically could lead to unintended resizing, but no better solution was
found. Leaving the problem unsolved would simply be worse.
4.4
Docking
As usual, docking is initiated by dragging the headers of docking windows, but we
also have two additional docking interactions. One way is double-clicking a header will
instantly undock turn the docking window into a ﬂoating window in-place. The other is
through a context menu with the undock action, which appears when right-clicking on
the header. So there is a total of three ways to undock a window.
==PAGE:43==
Design
36
This redundancy serves multiple purposes. It aids discoverability, increasing the chance
for the user to ﬁnd the docking functionality.
The three options also have slightly
diﬀerent use cases. The double-click for example, is sometimes faster if the docking
window should simply be a ﬂoating window rather than docked somewhere else.
Docking is initiated through icon-widgets like Visual Studio. This was chosen for its
simplicity and clarity. Like the Visual Studio widgets, we also use a set of inner- and
outer-widgets.
4.5
Undocking
We found in analysis that when a docking window becomes undocked, the neighboring
docking windows will expand to ﬁll the empty space. Only in case that there are no
neighbors will it remain empty, which is only possible if no other windows are docked in
the docking space. This was the behavior of every docking framework we found.
This is both potentially confusing and reduces the precision of the user interaction.
Expanding surrounding docking windows is not always what the user intends, as those
windows might have been conﬁgured to speciﬁc sizes. There are also ambiguous cases,
where there would exist multiple ways to ﬁll out the gap with neighbors. As we found
in section 3.4.1, in this case the space would usually be distributed equally to neighbors
- but this is just a compromise of not being able to guess the user’s intention. Finally,
this is also one of the problems we want to solve to verify our hypothesis.
Instead, we propose to leave the space empty, meaning none of the other docking windows
change shape. This way we do not have to move the dividers, rendering them truly static
and interfering less with the user’s layout.
Obviously, leaving empty space is wasteful, but we can assume the user has a plan for
using this space since they undocked the docking window in the ﬁrst place. We cannot
guess how the space is to be used, so we would rather leave it up to the user completely.
While it could be argued this is slower in some instances where we could guess correctly,
it leaves more decisions to the user and avoids unintended solutions.
This is also what we mean by precision of user interaction. By making the interac-
tions simpler and granular, the user’s decision space becomes more precise. The user
sometimes has to perform more interactions, but it becomes easier to reach a certain
layout-state without unintended side-eﬀects.
There is another interesting beneﬁt speciﬁcally to swapping a window with another,
while retaining size and position of the space. With the normal approach, if the old
==PAGE:44==
Design
37
window is undocked ﬁrst, the space will disappear. The user must dock the new window
as a tab and then undock the old to swap the window.
With our approach, either ordering works, as the layout remains unchanged. Swap-
ping windows positions can be diﬃcult if the DWF does not implement it explicitly,
because undocking inﬂuences the layout. The cloud based DWF by S. Yoshino et al.[19]
implements a swapping operation for example.
4.5.1
Divider Deletion
Because we leave spaces empty, we now have a new issue that the automatic-ﬁll solution
avoided. In the case the user wants to ﬁll the empty space with a neighbor instead of
docking a new window, they now have to get rid of the divider between them. In none of
the examined cases was this a problem, as the dividers automatically disappeared when
no longer dividing two diﬀerent windows.
To solve this, we developed a new technique for deleting dividers and manually ”closing”
empty spaces.
When the user wants to completely ﬁll an empty space with a neighbor, they can drag
a divider all the way across the empty space to the opposite bound. The empty space
should no longer be visible, and the divider is now redundant. We can safely remove the
divider and erase the empty space, eﬀectively covering it with the resized neighbor.
Using this technique, the user can quickly expand neighbors to ﬁll out the spaces in
whatever fashion they want.
It is obviously not as fast as the DWF doing it auto-
matically, but we avoid having to guess the user’s intention and potentially frustrating
them.
This style is also more true to the usual ﬂoating window-behavior. Windows that are
not docked, would remain unchanged if a neighbor was closed or moved. We achieve a
more static layout, that also is more similar with normal window behavior with which
the user should be familiar with.
This method is close to the frame edge deletion by S. Yoshino et al.[19] in their cloud
based DWF solution. In their example, deleting a frame edge causes the neighboring
window frame to expand over the empty frame.
==PAGE:45==
Design

4.5.2
Divider Culling
Similarly to the problem solved by divider deletion, we can end up with dividers that are
between two empty containers. Since dividers are meant to mimic window edge-handles,
they are redundant when no window is present, and should be removed. This will merge
the two areas, into a single container.
Note, that the two containers should be completely empty. We cannot remove the divider
if the areas are simply semi-empty. This means cases where the containers have further
divisions, with some of the areas empty. Do note however, that after we have culled
and merged two containers, new dividers could turn out redundant, and should also be
culled.
4.6
Docking Windows
We want to mimic normal window behavior when possible, such that they feel familiar
to the user. Especially ﬂoating windows should be close to normal windows. We want to
keep default functionality and the maximize, minimize and close buttons. The simplest
way to do this, is to keep the Windows visuals, including title-bar.
We decided to not diﬀerentiate ﬂoating-windows depending on whether it contains one
or more docking windows. This way the visuals are more consistent, and there is no
confusion between when a docking window is a ﬂoating window.
As a consequence of this, the ﬂoating windows’ title bars never collapses with the docking
window header.
By re-using title bar buttons, the docking window headers can be
simpler, only requiring a title and a close button.
The close-buttons are only shown when hovering over the header, something used in
Visual Studio. With a lot of docking windows open at the same time, this helps reducing
clutter. Closing a docking window, while obviously a very important feature, can be
assumed to be a rare interaction compared to working in the application. It should be
easy to discover, as hovering over the header happens whenever the user needs to undock
the docking window.

4.7
Design Summary
We have now gone through the most important speciﬁcations of the DWF, while intro-
ducing some new concepts. The most important of these are the static dividers, divider
deletion and the undocking leaving empty spaces.
The static dividers ensures that moving any divider will not move others, and also
makes the host window more context aware. One downside we discovered however, is
that resizing the host window is sometimes ambiguous to resizing its docking layout. It
was decided to count resizing the host window as resizing whichever window happened
to be bordering the edge, even though this could aﬀect multiple edges and docking
windows.
By leaving spaces empty, we now also ensure undocking does not modify neighboring
windows. We added the technique of manually deleting dividers, such that the user now
decides how to best use the vacant space.

be brought up in the next chapter, which covers implementation.

depth to the docking layout. It is therefore important that this illusion is not broken in
order to not confuse users, and the tree structure should not be apparent.

5.2
Dividers
As we decided in chapter 4, we are using the static behavior for the dividers.
This
means the dividers should stay still on-screen, except when dragged or the host window
is moved.
It turns out this is more diﬃcult to implement than it would seem.
To
understand this, we ﬁrst have to clarify how we use WPF to position the controls.
5.2.1
The Grid Control
On of the standard library components included in WPF is the useful grid control, which
is very ﬂexible for layouting controls. Its main function is to partition content into rows
and columns, like a table. The widths and heights can be deﬁned as either proportions,
==PAGE:51==
Implementation
44
absolute distances or just automatic ﬁll. Individual cells or a span of them can then
contain child controls.
The gridsplitter control can be used to create divider handles between cells. The grid-
splitter works by resizing adjacent row or column lengths when dragged by the user,
such that the splitter follows the cursor.
By having a grid with two cells divided by a gridsplitter, we can construct a splitpanel.
However, we still need to set the position of the gridsplitter, such that it behaves as a
static divider.
If we naively set the row or column distances to some proportion, the result will be
elastic dividers. On the other hand, setting the distances as an absolute distance at a
constant value, will just result in top-left bound dividers.
This is because the distances are set relative to container, not the window or screen. In
fact, all positions in WPF are always deﬁned relative to the container.
Therefore, whenever any container is is resized, it will cascade down all the descendants.
For any sub-dividers, their positions will now have changed relative to the screen.
The ﬁrst step is to make the divider position adjustable. Depending on the orientation
of the splitpanel, we call the left-most or top-most cell the ﬁrst container, and the other
cell as the second container. By adjusting the length of the ﬁrst container, and simply
setting the second container to ﬁll, we can control the exact position of the splitter, as
a distance from the top-left origin. Adjusting the length of a cell, is done by setting
a distance for the related column or row, again depending on the orientation of the
splitpanel.
We now have a way to construct splitpanels, that can be used recursively. The content
of either of the containers, can either be more splitpanels or docking windows. With the
splitpanels, we can also control the exact position of the divider relative to it, so now
we just need to ﬁgure out which position achieves the desired static behavior.
5.2.2
Static Dividers
To achieve the static behavior of a divider, we need to catch the relevant events where
the divider is moved involuntarily, and then correct it.
Recall the WPF layout process from section 2.2.5 The OnSizeChanged event is called
on a control whenever its size has changed. Unfortunately, there is no context for this
==PAGE:52==
Implementation
45
resize other than the change in dimensions. We do not know the change in position, if
any, or the sides that has been resized.
If we store the divider screen position from before a resize, we can compare it with the
new screen position to ﬁnd the distance it has moved. This can then be used to correct
the position of the divider, by moving it in the opposite direction by the same distance.
Retrieving the current screen position is readily available in WPF. We need to store
the old divider screen position however, ﬁrst when the control is initially displayed, and
then every time it is resized (after correcting the position obviously).
This is not immediately compatible with moving the window however, as the dividers
need to follow the window. To solve that case, we also need to update the old divider
position whenever the window is moved.
This way dividers will ”accept” their new
position, and react correctly when moved and then resized.
There is an event for window movement as well, although it is not immediately exposed.
WPF Windows has a message loop, which every window event passes through and
methods can be hooked into.
The divider simply has to hook into this loop when
instantiated and detect the move message.
Note, that like the built-in resize event there is also a location changed event in WPF.
This is not usable in our case however, as it is called in too many cases. Speciﬁcally the
event is raised whenever the window is moved, or the container is resized from the top
or left (which moves the origin). As this event is called separately form the size-change
event, there is no way to safely detect whether the window has moved and resized, or
just resized. Therefore we need the window move event, as it isolates the cases.
We now successfully have static dividers, that automatically correct their position on-
screen when resized.
5.2.3
Limits
Rows and columns cannot be reduced to below zero length. Since we move the dividers
by changing the size of their neighboring rows or columns, that means they are limited
by the boundaries of the container. This even works with multiple dividers in the same
grid, such that they cannot be dragged past each other.
However, this does not account for any sub-dividers. By default, the dividers ignore
the contents of the cells. This is not a problem with the other behaviors, because as
previously stated sub-dividers simply move out of the way. This is not the case with

WPF has some built-in tools that we can use here, although they are fairly limited. The
grid control allows for minimum- and maximum sizes to be set for any row or column,
which restricts the movement of the dividers.
Using this in the splitpanel, we can set a minimum and maximum size of the ﬁrst
container to deﬁne a range of permitted movement for the divider. By setting the right
min and max values, we can set this range such that the divider will appear to be
restricted by sub-dividers.
The solution is identical for both vertical and horizontal dividers, but with diﬀerent
values, so we will just describe the vertical divider solution. The interval we want to
create for a given vertical divider, is the area between the closest divider to its left and
the closest to its right. These dividers correspond to the previous and next vertical
nodes in our layout hierarchy. See ﬁgure 5.4 for a visual representation.
Assuming we have the distance to both the previous and next dividers, relative to our
splitpanel container, we can calculate the range. Recall that the WPF origin is the top-
left corner, so these distances are relative to the left boundary. The minimum limit is
==PAGE:54==

then equal to the previous divider’s distance added by the divider width. The maximum
limit is equal to the next divider’s distance, minus the divider width.
Note that you could instead use the minimum and maximum of both containers, but
that would be slightly redundant and also caused us some glitches.
So if we are given the distances for previous and next dividers, we can ﬁnd the limits.
The next part is ﬁnding these distances. Since these are values that we need to maintain
at all times and update often, a tree search would be very expensive.
Instead we create a custom dependency property used by splitpanels and tabhosts in the
tree called limits, that will store the values we need. This includes the distance from the
left-bound of the container to the right-most divider among itself and its descendants.
We call this value left. Similar values right, top, and bottom are found in the same way
for the other directions.
The idea is we use the dependency property to maintain these values, updating them
whenever any changes occur.
Recall from section 2.2.4 that we can create callback
functions for dependency properties. Using this, whenever any changes occurs to the
limits dependency value in a child, we can automatically update it in the parent. This
way we have a network of automatically maintained values, that are only updated when
necessary.
The only case where we need to explicitly update the value, is when a
splitpanel is resized.
Now we just need to ﬁnd these values. We obtain them recursively in two steps.
For a given tabhost, these four values are always zero. Since the tabhosts are leaves in
the tree, there are no dividers or sub-dividers at this container. We then choose to deﬁne
this distance as zero, which is the relative distance to the boundaries of the container.
For a given vertical splitpanel, we calculate a compound left, right, top, and bottom
values from its two sub-containers. Call the ﬁrst container A, second container B. We
are given the values width, as the full container width, divider x-position Divx, divider
width Divwidth. The compound values are then calculated as follows:
left = Divx + Divwidth + Bleft,
right = width −Divx + Aright,
top = max(Atop, Btop),
bottom = max(Abottom, Bbottom).

The left and right values are found by recursively using the left-most and right-most

5.2.5
Host Window Maximization
One of the new use-cases we have to address when using static dividers, is the results of
maximizing the host window.
Elastic- and top-left bound approaches can just follow normal behavior when maximized.
That is, when a window is maximized, the dividers will behave as if the window was
manually resized to ﬁt the screen. This is both intuitive, and usually leads to good
results.
However, since static dividers are context-aware to resize directions, we have more com-
plex behavior. Depending on how you interpret the intention of maximization, you could
expect diﬀerent results.
One way to think of it, is resizing the window from its current position until it ﬁlls the
screen. Another is ﬁrst moving the window to the top left corner, and then resizing it
to ﬁll the screen.
With the previous divider approaches, both operations would result in the same layout,
so this never comes up. With static dividers, the ﬁrst option would expand all bounding
docking windows, but leave dividers where they are. The second option would only
expand the bottom-most and right-most windows, while also moving the dividers towards
the top-left.
==PAGE:58==
Implementation
51
So we need to pick an approach, which we expect will most often lead to the best results.
While just resizing without moving is the simpler and more consistent choice, it turns
out that it has some issues.
For one, since static dividers are screen-position constant, if a window is partially outside
the screen prior to maximization, anything caught outside would be cut oﬀ. Then, any
dividers caught outside after maximization would automatically be pushed into view
(see section 4.3). This could severely disorganize the layout. It is not necessarily a rare
use-case either, especially when using multiple screens.1
Moreover, because the dividers are screen-static, the resulting layout would be diﬀerent
depending on where the window was positioned prior to maximization. Since none of
the common divider behaviors has this factor, there is no way the user could expect this
behavior.
Taking these arguments into account, we chose to interpret maximization as ﬁrst moving,
then resizing. The behavior will then be slightly artiﬁcial, but generally should be more
intuitive. While the solution may be inconsistent with general static divider behavior,
it is instead consistent with usual maximization behavior.

5.4.1
Detection
To execute the docking operation, we need to retrieve the container which is currently
being hovered over. WPF has has a ﬂexible method called a hit-test, which can be used
to traverse the tree at a given pixel.
It works by traveling downwards from the root towards the deepest visual that contains
the pixel. During this, it can be set to call methods for each control it passes, and also
which controls to ﬁlter. The callback method can also stop the hit-test if needed.
So to retrieve a container at a speciﬁc pixel, we need a control to look for to mark an
area as dockable. We created the hostsite control for this, which is placed at the root
of the docking space, and also as the two children of a splitpanel. This way, it is placed
exactly like our deﬁnition of a container from section 3.3.
So performing the hit-test, to detect the containers we simply ﬁlter any other controls
than hostsites. As we are using our own callback method for each hit, we can store
any information we need from the hit-test. In case we hit something that should not be
dockable, like a divider, we simply cancel the hit-test and store the result as un-dockable.
Since the hit-test travels downwards, the ﬁrst hostsite found is the highest, and the last
one found is the lowest. We will need both results when implementing widgets as we are
using both outer- and inner-widgets.
Visually, the hostsite control is invisible and simply ﬁlls its entire container, such that
the hit-test will hit it in every pixel of an area.

5.6
Manual Divider Deletion
Recall the manual divider deletion feature, explained in section 4.5.1. We want to delete
a divider when it resizes an empty, neighboring container into having either zero width or
height. We can use hostsites to detect this, since they follow the structure of containers.
Since we already have events for resizing controls, we can easily detect when a hostsite
meets the conditions. A hostsite is empty if it its child is a tabhost with no docked
windows in it. In other cases, it either contains further splitpanels or has docked win-
dows. It has no visible area if it has either zero width or height. Whenever the hostsite
is resized, we check if these conditions are met and if the user is currently dragging a
divider.
Once the user releases a drag on a divider, if a child hostsite meets the conditions, we
want to remove the splitpanel.
Like with divider culling, we implemented a simple version of this.
We remove the
splitpanel that has the empty sub-container, replacing its spot in the tree with its other
child. This will eﬀectively delegate all the space that was previously split to be owned
by the other child, and will remove the divider.
However it is not optimal. There are cases where we could remove either of two dividers.
In those cases, we would like to always remove the divider the user is currently dragging.
To improve this case, we could use a similar solution to the improved divider culling (see
section 5.5). This involves replacing the splitpanel by the child, whose sub-tree contains
the empty space, then moving the other child into that empty space. This is exactly the
same operation as the one shown in ﬁgure 5.13, but where the LCA is replaced by the
splitpanel owning the divider the user is currently dragging.

5.7.1
Layouts
An important feature of any layout customization mechanic is preserving it between
sessions. Otherwise, the user would have to spend time recreating it every time they
start the application. There are also other beneﬁts, such as sharing layouts between
users or quickly switching layouts.
In our case, we would have to save the hierarchy, including ﬂoating windows. We also
need some satellite data like window positions, states and divider positions to recreate
a layout exactly.
Recall that XAML is used to deﬁne controls (see section 2.2.3). XAML is XML based, so
any given layout of WPF elements can already be deﬁned in a XML format. Converting
the hierarchy to XML is easily done by using the same style as XAML when writing to
a ﬁle.
Saving the layout is then easily done linearly by passing through the tree, traveling from
top-down and left to right. Loading the layout can be done recursively in a bottom-up
approach.
5.7.2
Docking Window Deﬁnition
As the DWF uses WPF it is quite ﬂexible. It can display any WPF element without
further integration, but as it turns out there are some important features to connect
between the windows and the DWF.
In the previous section we covered loading and saving the hierarchy, but we still lack
the ability to load and save the docking windows themselves. To this end, we need a
consistent way to construct docking windows on demand.
Our solution gives each window a unique string key to identify it with. When saving the
layout, we only save the key for each window. Before loading the layout, we construct
a hash table of docking windows, where each window entry corresponds to their key.
When loading the layout, it then uses the keys to retrieve the corresponding window.
This approach only works if all the windows unique, rather than generic.
As an aside, a factory method pattern could be stored in the hash table instead of object
instantiations. However, WPF is optimized already to only perform layout operations
when the element becomes visible.
Instantiating the objects themselves is therefore
cheap, and unused windows will simply never perform the expensive layout operations.

Discussion
We now have the design and implementation of a DWF, based on the analysis of existing

Recall our hypothesis:
It is possible to build such a DWF, using the docking space approach, where a docking

Our aim was to eliminate all of the varying interactions, where the user was either forced
to or could unintentionally interfere layout.

6.1
Analysis
Our analysis in Chapter 3 examined a batch of contemporary DWF’s in use of editing
applications. We identiﬁed common concepts in DWF’s and their challenges in both
design and development.
We were also introduced to some additional features and
quality-of-life functions beyond the bare-minimum docking system.
A particular detail we found was that some of the common components among the
DWF’s had strong similarity to the standard library components from WPF. The best

example of this are the elastic dividers, which was the only divider behavior we found

6.2
Design
Using the ﬁndings from analysis, we built a set of speciﬁcations for our DWF in Chapter
4. We ﬁrst identiﬁed the problems we needed to solve to verify the hypothesis. To create
a static layout, we had to address the cases of host window resize, divider moving and
undocking, as these all caused indirect resizing of docking windows.
We introduced the static divider, which had the quality of super- and sub-dividers
acting consistent, such that moving either would not aﬀect the other. This also makes
the behavior more similar to normal windows, as resizing any window has no aﬀect on
others.
Undocking was altered to leave empty spaces rather than automatically resizing neigh-
bors to ﬁll. This allows the neighboring windows to stay static, which avoids modifying
the rest of the layout. It raised the issue of removing dividers, which we addressed by
introducing the divider deletion technique and automatic culling. Again, this concept is
similar to docking with a WM, where detaching a ﬂoating window would simply leave
the space empty.
There were a few compromises required to make these new concepts work, especially
with regards to host window resizing. As the dividers were now screen-static, resizing
the window could resize a container such that a divider was now outside its bounds.
The best solution we found, was to automatically move the divider back to the closest
valid position. Its only issue is that the changes are not easily reversible. Shrinking a
window and then enlarging it to its previous size could change the layout. With elastic
dividers, it will always return to the same layout after such an interaction.
This might turn out to be an important quality in the DWF, especially since user’s are
probably not used to a resize operation that is not immediately irreversible. We tried a
solution based on one of the top left bound divider variants, described in section 3.3.2.
Dividers would store the last position the user had moved them to, and would move
to the closest, valid position to this. In case a host window resize would push dividers
==PAGE:74==
Discussion
67
out, they would still store the old position. In case the host window was restored to
the previous size, they would return to their original position, thus making the resize
interaction reversible again.
However, we found this to be too confusing, as at any given moment it is not apparent if a
divider had stored a diﬀerent position than its current one. Therefore, resizing a window
or even moving other dividers, would sometimes move other dividers unintentionally. We
therefore decided against using it.
6.3
Implementation
Finally in Chapter 5, we described the most relevant details in implementing the DWF.
Implementing the DWF was important to demonstrate the validity of the design, and
prove that the theoretical rules were applicable. It also shows the practicality of the
new concepts, as the solutions used simple data structures and functions. Future DWF’s
should be able to implement similar solutions in reasonable time.
6.4
Evaluation
The new DWF design solves all the problems we identiﬁed with the existing solutions
related to indirect resizing of windows. As such, we consider the hypothesis successfully
veriﬁed, as our design demonstrates that it was possible create a DWF, where windows
were only resized by direct interaction.
The use-cases speciﬁcally improved by our static layout, are resizing the host window,
dragging dividers and undocking. In all three of these cases, we have either removed
or minimized the interference with the rest of the layout. Compared to the DWF’s we
examined using elastic dividers, in all these cases, unrelated windows would be moved
or resized in these use cases.
We believe that the static layout lead to more pleasing behavior, and also ﬁnd that the
use cases are not contrived, but occur regularly with use of a DWF. This was conﬁrmed
through ad hoc testing with a few employees from IOI, who would be future users of
the DWF. It is our impression that those users found the changes intuitive and easier
to use.
Beyond this, the DWF prototype was implemented in full according to the speciﬁcations,
both from IOI and from our designs. It was also completely integrated in the Glacier 2
editor, which was partially converted from WinForms to WPF to accommodate it.
==PAGE:75==
Discussion
68
The logical next step in the development of the DWF is usability- and user experience
(UX) testing. As we previously mentioned, UX is outside the scope of this thesis, but
it would be interesting to compare the static layout with its contemporaries.
We would mostly be interested in tests related to conﬁguring the environment, although
the testing the task executions are probably also important, just to verify there is no
discrepancy. In testing these cases, we are interested mostly on time spent, but also user
preferences and possibly amount of interactions required.
As an example of testing a docking system, Kandogan et al.[5] recorded subject’s time
spent on not only executing speciﬁc tasks, but setting up a task environment and switch-
ing task environments. By testing diﬀerent task complexities with their elastic windows
system and another WM, they could then compare the performances. For IOI’s case,
the best DWF to test against would be the current DWF they are using. In a larger
scope, it would probably be most useful to test against similar environments to the ones
we examined.

Chapter 7
Conclusion
In this thesis we have examined several existing DWF’s and classiﬁed the various ele-
ments such that we could deﬁne a DWF by them. We extended this with new designs,
most importantly the static dividers, empty containers and divider deletion technique.
Using these elements we constructed a new DWF with what we call a static layout.
Recall our hypothesis:
It is possible to build such a DWF, using the docking space approach, where a docking
window is only resized either by direct user interaction of dividers or resizing the
host window.
We were speciﬁcally interested in three use cases: dragging dividers, undocking, and
resizing the host window (i.e. the window containing the docking space). In these three
interactions, the DWF’s we examined could alter unrelated docking windows.
The static layout successfully veriﬁed our hypothesis as it addresses these three cases.
Dragging any static divider will never aﬀect others, undocking will never move any
dividers, and resizing the docking space no longer necessarily aﬀects all windows.
The design was fully implemented in WPF and integrated into Glacier 2, thereby com-
pleting the project goals set in collaboration with IOI and also validating the design.
The new DWF has received positive feedback in ad hoc testing by future users.
We avoided the common issues found in our analysis by heavily customizing controls
instead of using the standard library components. In our implementation we especially
focused on the new element designs, as their behavior were not natively supported by
WPF.

Future Work
At this point, the DWF is completely functional and integrated in Glacier 2. However,
some of its features could be improved, such as divider culling and manual divider
deletion. It is not deployed yet either, as it needs to be tested for both correctness and
usability.
The next step would be user performance and user experience testing, comparing it to
the current DWF in use for Glacier 2. We are interested in testing the time spent both

improvement. We are also interested user preference and validating our ad hoc tests, by
observing whether users prefer the static layout over the previous one.

1 Introduction 
A docking framework is a library of software that, when integrated into a GUI application, allows the 
user to dock/undock and arrange his windows and workspace as he likes.  
For reference, here is examples of 3​rd​ party docking frameworks:  
XCeed: ​https://xceed.com/xceed-docking-windows-for-winforms/ 
Actipro Software: http://www.actiprosoftware.com/products/controls/wpf/docking 
This document will cover the requirements for our own implementation of such a docking 
framework. 
2 Motivation 
Over the 11 years of life-time of Glacier 2 we have used 3 different 3​rd​ party window docking 
frameworks but every single of them had problems. The first (Unknown) and second (Weifenluo 
docking framework) was open source, but not very good. Third did not include source code so we 
could not fix bugs within them. We managed to get limited source code access to the last one 
(XCeed WinForms docking framework) but since we only have one license this is very convoluted to 
work with. And getting source code licenses for the entire team is very expensive. 
Furthermore, XCeed docking framework is overly complex and has a number of issues that annoys 
people. For instance, there is no support for multiple master panels and docking between floating 
panels behaves weirdly. 
To solve this problem long term we would like us to make our own that we can have ownership and 
that we can improve upon as needed. 
We do not want to use an existing open source project since they typically are overly complex and 
have features we don’t want or need. 
 
3 Criteria for design 
When making the design and implementation the focus should be on making a clean and simple 
design (both code and user experience) over feature richness. The code must be easy to understand 
and maintain. We explicitly want to avoid overly abstract code or generic code – keep the code as 
simple as it makes sense. 
The user experience should be aimed at allowing the users to organize the workspace in any way 
they see fit while focussing on minimize the amount of surprise. The user should not experience that 
windows move or change layout in unexpected ways. 
2 
 
==PAGE:81==
 
4 Requirements 
The list of requirements in this document should not be taken as rule book. If certain elements turns 
out not to be ideal we should change them. It should be taken as our current knowledge of what the 
system should do. 
Glossary: 
●
The new library will be referred to as ​the DFW​ (Docking Framework) in this document.  
●
G2 is short for Glacier 2 – the technology used in HITMAN. 
 
4.1
Platform  
●
The DFW must be written in C# for .NET and must use WPF (Windows Presentation 
Foundation) 
●
Its build setup must be done using into the Premake build system used on HITMAN. 
●
Although the main purpose is to use it for the G2 Editor it must be written in an application 
agnostic manner so we can use it for other applications within the G2 framework as well. It 
is ok to make use of G2 base .NET libraries. 
 
4.2
Visual style 
●
For skinning DFW must use the G2 common WPF stylesheets. This will allow us to skin the 
entire editor later on – for instance to give it a dark skin. 
●
The editor should have modern look. Work with our Tools Designer (Robin-Yann Storm) on 
settling on the exact style. 
 
4.3
Master panel  
●
The application does not use the docking framework windows as the ​application main 
window​. The application is expected to provide its own main window.  
●
Anywhere within the application a single ​Master panel​ can be placed.  
●
Note: The separation between main application window and master panel will allow custom 
layout of the main windows, e.g. to add a status bar, a tool bar and a menu bar without 
having to build that into the docking frame work. 
●
The master panel can be used as a hosting area (see below) for tool windows (see below). 
●
When the master panel is not hosting any tool windows it will show a blank (e.g. gray) 
background. 
 
4.4
Tool windows 
●
A ​tool window​, is a window that can be docked in a hosting area. 
●
It is owned and drawn by the DFW. 
●
A tool window is drawn using a thinner-than-normal title bar. 
●
The title bar only contains a close button (x). There should be no “pin”, “minimize” or 
“maximize” button. 
3 
 
==PAGE:82==
●
The contents of the tool window is defined by the application using DFW. The client 
application should have full control over the client area of the tool window. 
●
Paint/Update messages should only be sent to tools windows that is at least partially visible. 
It is important for performance that the client code for invisible tool windows does not 
perform any processing. 
●
A tool window has no border except for title bar. (Splitters and any host window (see below) 
will provide the need borders for resizing, etc.) 
 
4.5
Hosting area 
●
A ​hosting area​ is a top level panel that can contain a hierarchy of tool windows. 
●
A hosting area can be a master panel or it can be placed inside a host window (see below) 
 
4.6
Tool Window docking hierarchy 
Within a hosting area all Tools Windows are organized in a binary tree hierarchy where each node in 
the tree represents the region. An inner node represent either a horizontal or vertical split of the 
region and a leaf node represents a Tools Window or multiple tabbed Tool Windows. We call this the 
Tool Window docking hierarchy​.  
Each hosting area has its own Tool Window docking hierarchy. 
 
4.7
Undocking tool windows  
●
The user can grab a tool window by holding and dragging either (1) the title bar of the tool 
window (2) The named tab if the tool window is tabbed.  
●
Only one tool window can be undocked/dragged at a time; this is relevant in the case of 
tabbed tools windows. This also implies that it is not possible dragging something that 
represents an inner node in the Tool Window docking hierarchy. All tool windows are leaf 
nodes. 
●
The user can then move the mouse (and hence place the window) either in a docked 
position or leave it as a floating window. 
●
During the undocking/moving procedure the Tool Window should follow the mouse cursor – 
relative to the place it was when the undocking started. 
●
When a tool window is undocked or closed, it is replaced by an empty area in the Tools 
Window docking hierarchy. Undocking should *never* rearrange or resize other tool 
windows or regions. 
 
4.8
Floating tool windows / Host Windows 
●
If the users releases an undocked tool window outside the area of current application it 
becomes a floating window. 
●
A floating window is a tool window located as a top level node in the tool window docking 
hierarchy for the hosting area placed inside a Host Window  
●
The Host Window is created by the DFW on demand. 
●
The Host Window has a Close, Maximize/Restore, and Minimize button. When it is closed all 
tools windows within the Host Window is closed.  
4 
 
==PAGE:83==
●
A Host Window does not contain tool bars, status bars or menus. 
●
A Host Window is *not* dockable. Only tool windows are dockable. 
●
The title of a Host Window is a concatenation of the titles of the tool windows. 
●
When the last tool window has been removed from a host window the host window will 
automatically close. 
●
If a host window only contains a single tool window, the two title bars should behave 
equivalently. Do not collapse the hosting window title bar and tool window title bar into 
one. 
4.9
Docking tool windows 
If the user is dragging a tool Window and the mouse is hovering over an empty hosting area or tools 
window within that hosting area then the ​docking selection widget​ should be shown. It should look 
something like this: 
 
A window will only be docked if the mouse is released while being over one of these 5 icons. While 
hovering over an icon the blue shadow should provide a preview of where the tool window will be 
docked. 
Note: Since this icon only appears on tool windows that essential mean you can only dock onto leaf 
nodes in the Tool Window Docking Hierarchy. 
●
The middle icon indicates that the tool window will be tabbed with the existing tool window. 
If a tab list already exist here an extra tab is added to the existing tab list. 
●
The top icon indicates that existing tool window will be split in half (top and bottom) with a 
horizontal splitter control in middle. The existing tool window will be halved in size and then 
occupy the lower part. Other tools windows should *not* change size or position. The 
dropped tool window should be in the upper part and be resized to match. 
●
The Bottom, Left and Right icons works equivalent to the Top icon. 
 
4.10
Tabbed tool windows 
Whenever a tool window is tabbed with other tool windows a tabbed list should appear at the ​top. 
This will replace the tool window title bar and the tab should contain the title. 
Note: We discussed if it should be at the bottom but that uses more screen real estate. By placing it 
at the top we avoid having both title bar and tab list. This idea is used with success on most modern 
browsers. 
●
Double clicking a tab will undock that tool window and make it floating. 
●
The tab has no bottoms – in particular no close button. 
●
When there is only one tab left, the tabbed list disappears and a regular tool window is 
shown instead. 
5 
 
==PAGE:84==
●
If there is more tabs then there is space the tabs should be made smaller (truncating the text 
on each tab).  
●
Tabs should never span multiple rows. 
 
 
4.11
Splitters 
●
Whenever a window is docked a horizontal or vertical splitter is created separating the two 
tool windows. The splitter is used to resize the split space in one direction. 
●
When the mouse hovers over the splitter window the appropriate splitter mouse cursor 
should be shown. 
●
When a splitter is moved it *cannot* effect the position of other splitters – those should stay 
at the same position on screen. If another splitter (of the same orientation) blocks 
movement the splitter cannot moved further. If another splitter of the opposite orientation 
connected with the moved splitter then it resize its length to match the available space. 
 
4.12
Resizing tool windows 
●
Host windows and the main application window (and this the master panel) can be resized 
by dragging any of the sides or corners. 
●
A tool window can *statically* indicate what its minimum and maximum sizes are. This must 
be given at window creation time (so it can different for each window instance) 
●
Operations that causes these restrictions should not be allowed. E.g. it is not possible to drag 
a splitter so a windows becomes smaller than its minimum size. And it should not be 
possible to dock a window in a hosting area that is bigger than the maximum size. 
 
4.13
Context menus 
●
Right clicking a tool window title bar will show a context menu showing a “undock” (if 
docked) and “close” button. 
●
Right clicking a tab will show a context menu showing a “undock” and “close” button.  
●
Right clicking a host window will show a context menu showing the regular window 
commands (move, resize, minimize, etc.) 
 
4.14
Maximize / Minimize / Restore 
●
Only host windows can be maximized and minimized. Tool windows and tabs cannot. 
●
A maximized host window will occupy the entirety of the current screen. Only one host 
window can be maximized on the same screen at a time. If one is already maximized the 
maximize bottom of the other ones on the same screen should be greyed out. 
●
In general there should only be one icon in the task bar of Windows for the full G2 Editor. 
There is no task bar icons for host windows; except when they are minimized. 
●
When a maximized or minimized Host Window is restored it should return to the position it 
had before it was maximized or minimized. 
 
6 
 
==PAGE:85==
4.15
Save/load/reset layout 
●
The DFW should have API calls that allows the main application to make menu points that 
allows the user to save, save as.., load and reset window layouts respectively. 
●
Layouts should be saved to and loaded from a folder specific in a local configuration file. This 
will allow users to share layouts more easily. 
●
When loading a layout; it should be verified that allow window positions are visible on 
screen and if not the user should be warned and the windows pulled into the visible area. 
This can be important if users have different monitor layouts or have changed their monitor 
setup. 
●
Resetting the layout will bring it back to the default layout provided by the application. 
 
4.16
 Workspace management 
Often perform different tasks over the time of a project and they might change frequently between 
these tasks. As a result they often have all the views open as required to support all those tasks. This 
takes up a lot of screen real estate (over 3 monitors) and also effects performance. 
The DFW should support that the user can rapidly change between different layouts – one for each 
type of task that needs to be worked on. 
We imagine that the top of the Master panel can contain a bar like this (called ​The Workspace Bar​): 
 
With this you can switch between workflows and, create, rename, delete, and reorder them. If the 
“lock” is toggled one can only switch between workspaces – not change this existing set. 
Switching to a workspace mean changing the window layout to use the one saved with workflow. 
The “+” means saving the current window layout as a new workspace. 
Each workspace button should have a right click menu with “rename”, “delete”, “update from 
current layout”. 
It should also be possible to reorder the workspace buttons by drag-dropping. 
 
4.17
 Performance 
●
Overall all operation should feel fast and responsive.  
 
4.18
 Integration with G2 Editor 
●
All the existing tool windows in the G2 editor should be changed to use the new DFW. 
●
It should be verified that all the existing tool windows does not do any updates when they 
are not visible. If they do, then this should be changed.

5 Potential requirements 
This section list ideas that we might want to implement in the future; that is not settled yet. They are 
*not* part of the project requirements. But if time allows it would be nice. 
 
5.1
Optionally: User roles 
It would maybe make sense to introduce the concept of user roles. We already have the start of this 
with the Asset Browser user profiles. Then we could have a different set of default 
layouts/workspace for level designers than for, say, artist. 
5.2
Optionally: Locking support 
We are not sure but it might be too easy to move windows around by mistake. If this turns out to be 
true then we should add locking functionality. When DFW is locked all tool windows and splitters 
should be locked in place and cannot be moved. 
 
5.3
Optionally: Support multiple master panels 
In the G2 editor we have multiple sub editors that each consist of multiple panels. For example: the 
Shader Editor, the Sequence Editor, and the Resource Analyzer. These panels below together so they 
should not be individual tool windows. It would be ideal if we could use DFW to rearrange these 
panels within the appropriate sub editor. 
●
Applications should support multiple master panels. 
●
Each tool window is assigned a master panel owner – this is set at window creation time and 
cannot change. 
●
Each master panel maintains its own tool window docking hierarchy. A tool window can 
*only* be docked in hosting areas belonging to its owning master panel. And a hosting areas 
implicated created by a floating tool window will belong to the same master panel.  
●
Note that the new master panels can be placed in any location; including inside a tool 
window in the hierarchy of another master panel. 
●
When the window containing the master panel is closed all the host windows owned by it 
should also be closed. 
 
6 Features that we don’t want 
 
Some docking frame works support features that we do not want. For completeness we will here is 
lists some of those common features we are *not* interested in. 
 
8 
 
==PAGE:87==
6.1
Undo support 
The DFW should *not* have Undo support. The G2 Undo support is designed to work with changes 
to the main application document. Windows layouts are not part of this and should not be placed on 
the undo stack. 
 
6.2
MDI overlapping windows 
In some older Windows application it was common to allow floating windows within the master 
panel. These windows could be moved among each other and could be overlapping. There would 
then typically also be “tile horizontally”, “tile vertically”, “cascade” options in the Windows menu. 
We don’t want any of this. In DFW a window is either floating (in Host window) or docked. 
 
6.3
No special main document area 
Some docking frame works (e.g. Visual Studio 2015) have special area that I here call the “main 
document area”.  
This area is used to show documents (e.g source files) and you can have an unlimited set open at any 
given time. We want no such special treatment of the main document. All tool windows are equal. 
 
6.4
Auto hiding windows 
Some docking frame works (e.g Visual Studio 2015) have the support for “auto hiding” windows. 
These will slide into view when the mouse is over the placeholder bar. The user can then use a pin 
button to disable the auto hiding. 
The DFW should not support auto hiding tool windows and we should not have a “pin” button. 
 
6.5
Embedded toolbars and menu 
Toolbars and menus on the Main Application windows is out of scope for the DFW. They are 
important but that is for another project.
