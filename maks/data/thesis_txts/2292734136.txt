Abstract
This thesis deals wit the implementation of a framework for analysis of dierent
evolutionary algorithms on pseudo-boolean problems.
Dierent search spaces are introduced: both pseudo-boolean and combinato-
rial. Classical examples for these, such as the Travelling Salesman Problem are
further explored.
The dierent algorithms used are Evolutionary Algorithms, Genetic Algorithms,
and Ant colony Optimization algorithms. All of the algorithms are implemented
to run on both the pseudo-boolean and combinatorial search spaces.
Dierent methods of analysing the performance of the algorithms are also ex-
amined and methods of measuring these are implemented in the program. The
analyses include the expected performance of the algorithm after a set amount
of time, and results which are useful in assisting with theoretical analyses of the
algorithms.
The program is designed to run in parallel, so as to allow for analysis of large
problems.
Lastly a number of tests of the program par performed.

Chapter 1
Introduction
Evolutionary Algorithms (EAs) are metaheuristic optimization algorithms which
are inspired by evolution in nature. EAs can often nd good approximations for
problems even if the problems are very complex and the underlying structure is
unknown.
Theoretical analysis of evolutionary algorithms can be hard to use in practice,
for example due to guarantees on the runtime which are asymptotically good,
but practically infeasible. In order to bridge this gap, theoretical results are
sought that do not neglect the real-world performance.
This project is about creating a framework for experimental analysis of Evo-
lutionary Algorithms. Some of the analysis includes tracking tness levels and
running time, and drift analysis.
Multiple dierent algorithms and problems will be explored. The rst types
of problems are pseudo-boolean functions, where the output of the function on
a bitstring is optimized (minimized or maximized). Another type of problems
are combinatorial, for instance the Traveling Salesman Problem, where a path
between a set of points is minimized. The rst algorithm implemented is the
(µ+λ) EA, where a population of µ individuals (solutions) mutate and produce
λ new individuals, of which the best µ are selected to form the new population.
Later other algorithms such as Ant Colony Optimization and Genetic Algo-
rithms will be implemented.

2.1.2
Combinatorial problems
Another common type of problems are combinatorial problems. In these, the
search space consists of permutations of some sequence which then maps to a
real number.
(Probably) the most famous combinatorial problem is the TravellingSales-
manProblem (TSP), which seeks to minimize the length of a Hamiltonian cycle
of a set of points.
In other (more popular?) words, the aim is to minimize the travelling distance
of a salesman who has to visit a number of cities, and the return to home.
The permutation describing this problem instance is simply the list of vertices
in the order they are visited.

Algorithms
There are many dierent types of evolutionary algorithms.
2.2.1
Evolutionary Algorithms
A common type of evolutionary algorithm is the (µ + λ) Evolutionary Algo-
rithm (EA). The algorithm has a (currently best) population of µ candidate
solutions (individuals). At each step an intermediate population of λ individuals
is created by mutating individuals from the previous population.
All individuals are then evaluated by a tness function as described above, and
the best µ individuals are selected to be the new best population.
Algorithm 1: (µ + λ) EA
1 Let P be a set of µ uniformly chosen elements of {0, 1}n;
2 repeat

7

P ←µ f-maximal elements from P ∪O;
9 until forever;
The simplest and most analysed EA is the (1 + 1) EA. Here µ and λ are both
set to 1, meaning that there is a population of a single individual, and only one
new individual is mutated per generation. Furthermore the new individual is
chosen if and only if its tness is at least as good as the population's.
A similar algorithm to the (1 + 1) EA is Random Local Search (RLS).
Here there is also only a single new individual each time, but the mutation is
local. For bitstrings this means only ipping a single bit, instead of each one
independently.
Another algorithm which is very similar to the (µ + λ) EA is the (µ, λ) EA.
In this algorithm the new best population is only chosen from the intermediate
population of λ individuals. λ must therefore be at least as large as µ in this
algorithm, to maintain the population.

2.2.2
Genetic Algorithms
Genetic Algorithms are algorithms which are inspired by evolution in the sense
of mating and mutation in populations. This includes some sort of selection
scheme, and a way of combining two individuals in order to create ospring.
One such algorithm, known as the Canonical Genetic Algorithm[Whi94],
has an initial random population. From this population an intermediate pop-
ulation of parent individuals are chosen. Each pair of parents are combined
producing two ospring individuals, which are mutated.
This new ospring
population is the population of the next generation.

a selection scheme is used.
In many cases the intermediate population is chosen dependent on the tness
of the individuals. There are a few ways to determine this tness value, which
can eect the selection outcome.
One example is linear scaling where the tness of an individual is directly or
linearly scaled.

A couple of terms as dened in [Bak87] are used to describe to performance of
the selection schemes:
Bias is the absolute dierence between an individual's actual sampling prob-
ability and the expected value. The optimal, zero bias is when the sampling
probability equals the expected value.
Spread is the number of possible values for the number of ospring of an indi-
vidual. The minimum spread is the smallest spread which permits zero bias.
The simplest selection scheme is the Roulette-wheel selection (RWS), which
is also called Stochastic Selection with Replacement[Bak87]. Each individual is
mapped to a linear scale according to their tness value. A random point on
the scale is then chosen, and the corresponding individual is added to the in-
termediate population. This is analogous to placing the individuals around a
roulette wheel where each slice is proportional to their tness, and spinning it.
This procedure is repeated until the desired population size is reached. Roulette-
wheel selection has zero bias, but does not guarantee minimum spread, as any
single individual with f > 0 could be chosen for the entire intermediate popu-
lation.
Stochastic Universal Sampling (SUS)[Bak87] is an alternative to roulette

intermediate population. A tournament of size t is held by choosing t individuals
from the population uniformly at random, and picking the individual with the

2.2.3
Ant-Colony Optimization
Ant-Colony Optimization (ACO) is a type of algorithm inspired by the be-
haviour of ants leaving their nest in search of food.
When ants go in search of food, they initially wander randomly, and when they
nd food, return to the nest while leaving pheromone trails on the path. If other
ants who also wandering around nd these trails, they are more likely to follow
them, and if they nd food, reinforce that path.
Over time the pheromone trails evaporate, so they are not as attractive as newer
trails. If an ant follows a long path while gathering food, the trails have a lot
of time to evaporate before the ant comes back. If however, the path is shorter,
the trails will evaporate less before getting rewalked, and will thus reinforce a
shorter path more than a longer path. Because of pheromone evaporation, there
is also a chance of escaping local optima. If there were no evaporation the rst
ant to nd food would leave a pheromone trail which most other ant would
follow even if shorter paths were found, as more pheromones are already present
on the original trail.
The aim then, is to have an algorithm which simulates this behaviour, and
mimics ants walking around and laying down pheromones.
One such algorithm is the Max-Min Ant System (MMAS), keeps track of
the pheromones between each pair of points and constructs a path from the
pheromones. The pheromones are then updated accordingly. The signicant
dierence between MMAS and other ACO algorithms is that the pheromones
in MMAS have a minimum and maximum value. This ensures that one path
will not be reinforced so much that no other paths can conceivably be taken.
The algorithm starts by initializing all the pheromones at an equal value. Then
a path is constructed using the pheromones, and the pheromones are updated
according to that path. Now the algorithm enters into its main loop, which runs
forever (or until some halting condition), in which a path is constructed and
evaluated. If this new path is better than the previous best, it is remembered.
The pheromones are now updated according to the (potentially new) best path.
The construct procedure starts at an initial node and uses a weighted probabil-
ity to choose which unvisited node to visit next. At each step a node is chosen
until no more nodes are left.

The tness landscape is the tness of the individuals in a neighborhood of an
individual. The analysed individuals can either be the entire neighborhood or
a random sampling of it.

the individual; The individual is a local maximum.
2. The minimum tness of the neighborhood is larger than the tness of the
individual; The individual is a local minimum.
3. The average (and maximum or minimum) tness of the neighborhood is
the same as the tness of the individual; The individual is on a plateau.
4. None of the above; The individual is on a slope.
By sampling a large number of random individuals and looking at the tness
landscape, one can estimate the number of local maxima and minima using the
above four classications.

3.2.4.4
Drift
The drift is estimated by taking a population and mutating it independently a
large number of times. The tness values of these new populations can then be
averaged, and are an indicator of the expected improvement of a population.
This should be done by copying the algorithm along with its population, and
then mutating the population without replacing the current one. This new in-
termediate population is evaluated and discarded, with the tness values count-
ing towards the total average. This repeats until enough samples have been
recorded.
Both the drift of the best individual of the population [HY01] and the average
tness of the whole population are recorded in this manner.
When running the program on multiple cores, the algorithm will be copied to
each process which then estimate the drift for a portion of the total number of
required times. The number of times each process need to calculate the drift is
split evenly. If an even split is not possible, the remainder is spread among the
rst processes (the remainder will always be less than the number of processes).
3.2.4.5
Fixed Budget Analysis
Fixed budget analysis looks at the expected value after a certain number of
steps. The calculations can be seen as the `inverse' of the drift calculations, as
drift looks at the expected improvement at each step.
Where drift copied the algorithm and population at each step, xed budget
copies the algorithm a number of times, but with a new random population for
each. Each of these algorithms is then run in parallel, and the tness values are
averaged together at each step. The algorithms continue logging after reaching
optimal solutions so the mean population tness can still be determined, but
stops when all algorithms have found an optimal solution.

The cluster servers at DTU Compute have a message passing library known as

The basic idea of Open MPI is that the program runs independently on a number
of processors, and each program instance knows which processor it is running
on. The dierent instances can then pass messages to each other by specifying
the destination process.
The messages passed between processes in MPI are all arrays of primitives
(double, int, etc), which means that complex objects cannot be passed along.
Because of this all the data being sent back and forth must be encoded into
arrays.
Some examples which are trivial include the individuals (arrays of booleans or
integers), and the calculated values which can be stored in arrays of doubles.
More advanced are the dierent options for the algorithms. These could either
be passed along as strings (arrays of characters), or a lot of constants could be
dened which correspond to each dierent option.

4.1.2
RNG
In order to conduct the experimental analysis, a random number generator
(RNG) is needed. The RNG should be suitable for Monte Carlo experiments
(eg. when sampling the tness landscape), and should be seedable to allow for
repeated results.
The default pseudo-random number generator in java has a relatively low period
(248) which makes it unsuitable for large Monte Carlo trials, especially for very
large runs.
An alternative in Java is the SecureRandom1 which is a cryptographically strong
RNG. There are two problems however: rstly the generator is quite slow (see
Appendix B.1) and secondly seeding the algorithm does not guarantee the same
output, which makes it impossible to perform identical repeated experiments.
A popular pseudo-random number generator is the MersenneTwister[MN98].
The standard implementation MT19937 has a period of 219937 −1.
The specic implementation of the algorithm used is a Java implementation
of the MT19937[sea].
Two implementations are provided:
MersenneTwister
and MersenneTwisterFast.
The fast version of the RNG is algorithmically
identical to the normal version, but has a lot of hard-coding and the methods are
not synchronized, which means that the same RNG cannot be shared between
threads.
A comparison of the dierent RNG's can be found in Appendix B.1. The speed
of the algorithms are compared by generating a large number of random values,
and next the generators are seeded with the same value twice, to ensure that
they produce the same output.
The results show that the SecureRandom is about ten times slower than both
Java's default RNG and the Mersenne Twister. The fast implementation of the
Mersenne Twister is twice as fast as the normal implementation.
According to the documentation of the Mersenne Twister, MersenneTwister is
approximately 1/3 slower than Java's Random, which is in turn about 1/3 slower
than MersenneTwisterFast.
The RNG chosen in this project is the MersenneTwisterFast, although it would
be possible to make the RNG substitutable, either by using MersenneTwister
which is a subclass of Java's Random or making a custom wrapper class which
could use multiple dierent RNG's.

4.5.3
Drift
Each dierent thread is initialised with a copy of the algorithm and its popula-
tion.
At each step the current population is sent to each thread, which then calculate
the drift, and return the drift value. The results from each thread are combines
together as a weighted average, where the weights correspond to the number of
samples calculated in the corresponding threads.
The constructor of the thread calculating the drift initialises an Algorithm ob-
ject, and stores the relevant parameters (drift samples and references to queue
and RNG).
The drift calculation itself is as described in the design section. First it gets the
current population from the main thread, and sets it as the population of the
stored algorithm class. The population is then mutated to get an intermediate
population, from which the best and total drift are calculated. More intermedi-
ate populations are generated and analysed until the desired number of samples
is reached.
The drift values of all the samples are averaged together, and returned along
with the number of samples examined, which is used by the algorithm for the
weighted average.
Specically the values are returned in an array of doubles.

The xed budget analysis of the algorithm is implemented in the same overall
way as the drift calculations, where each thread gets passed information from
the main thread at each step which prompts the calculations.
The calculations themselves are of course a bit dierent.
One of the parameters in the constructor is an algorithm object. This algorithm
is then copied with a random population multiple times, and each are added to
a list.

The calculations of the xed budget are similar to the calculations of drift, but
instead of the population being passed to the thread via the queue, the current
time step is passed. Each algorithm in the list is then advanced a single step,
and the tness values are calculated.
The best and mean tness of each algorithm are averaged together to form the
resulting tness values.
The reason for the time step being passed to the thread is so that the tness
values of the initial step can be returned correctly. If step=0, the algorithms do

As with drift, the values are returned in an array of doubles. The specic return
values are the best tness, the average tness, the number of algorithms, and a
ag indicating whether or not all the algorithms have reached an optimal value.
When all algorithms in all the threads have reached an optimal value, there is
no need to run the algorithms any more, and the main thread can skip the rest
of the loop.
4.5.5
Fitness landscape analysis
The tness landscape analysis has two dierent types of analysis. One analyses
the neighborhood of the best individual of the current population, and the other
estimates the number of local optima in the search space.
At each step of the algorithm, the neighborhood of the best individual is either
found or sampled. The tness values of the neighborhood are then calculated
and logged. In particular the average, the minimum, and the maximum tness
values are logged.
4.5.5.1
Sampling the neighborhood
If the requested number of samples exceeds the size of the neighborhood, it
is more accurate to look at the entire neighborhood instead of sampling. The
Neighborhood operators contain methods which calculate the size of the neigh-
borhood of an individual.
The dierent Neighborhood operators have methods which either randomly sam-
ple the neighborhood, or return all the individuals in it.
For the bitstrings, the neighborhood relation chosen is a single bit-ip, which
==PAGE:51==
4.6 Parallelization
39
means the size of the neighborhood is the length of the bitstring. When sam-
pling, a random bit is ipped, while each bit is ipped in order when taking the
while neighborhood.
The neighborhood relation for the combinatorial search space is the 2-opt oper-
ator. As determined in Section 3.2.4.6 the size of the neighborhood is N(N −1).
The full neighborhood is found with two nested loops, which take every pair of
indices unless they are the same. Randomly sampling is done by applying a
single 2-opt mutation at random to the individual.
4.5.5.2
Number of local optima
Estimating the number of local optima in a search space is done by sampling a
large amount of random individuals in the search space. The tness of neigh-
borhood is calculated, and the minimum and maximum values are recorded.
The number of points which are local minima or maxima are normalized by the
amount of samples, and scaled by the number of possible points in the search
space.
If the size of the search space exceeds the largest double value in Java, the
result is Infinity, but the normalized value can still be used as a percentage
of points.
If no local maxima or minima are found, the algorithm looks at whether any
points are found which are not on a plateau. If there is such a point, it can
be safely assumed that there it at least one local (and global) minimum and
maximum. If all sampled points are on a plateau, the resulting estimation is
that there are no optima.

Logging
The logging of the analysed data is handled by a class Logger.
The main data structure of the log is a hash map which maps from the time
index (step) to an object containing the logged data at that time. The objects
are LogPoints, which simply contain an array in which the data can be stored.
As there are many dierent types of data that can be logged, an enum was
implemented, which contains a eld with each type of data. Each eld contains
a string description, which is used when the log is saved to a le.
When a LogPoint is created, the array is initialised to have its length the same
as the number of types of loggable data (as per the enum LogType).
Because the enum is constant, the type can be used as an index in the array
when adding data to a LogPoint.
When logging data at a certain time, the Logger nds the LogPoint for that
time, or creates a new one if not available. The data is then added to the point,
which is put back into the map.
Because not all types of data are logged at each run, a boolean array is kept in the
Logger class as well. When data of a certain type is logged, the corresponding
boolean value is set.
Writing the data to a le is done at the end of the run of the algorithm.
First some information about the rune are written to the le such as algorithm
name, problem type, and so on. Then the boolean array is checked, and the
name of each type of data which has been recorded is written to the le separated
by tabs. Then all the data (sorted by time index) is written to the le in the
same manner.
For simplicity and to avoid collisions with existing les, the le name is chosen
to be the current date and time with .csv as extension4. If the le does exist,
the log is appended to the end of the le such as not to overwrite any important
