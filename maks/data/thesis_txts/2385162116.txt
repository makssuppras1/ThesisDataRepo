Abstract
This project is split in two parts, ﬁrst of which is testing the Pixhawk PX4FLOW Smart Camera for
usage in an unmanned ground vehicle (UGV) to help reduce the position error gained when the UGV has to
do an emergency stop. The camera was tested at multiple altitudes, on multiple ﬂooring types and with extra
lighting in a stationary situation and ﬁnally mounted on a driving mobile robot. The conclusion was that
when standing still, the position drift caused by noise on the velocity measurements from the ﬂow camera
meant that the ﬂow camera was not suited for the desired use as drift was too high and non systematic and
therefore could not be calibrated.
The second part of the project was calibrating a SICK LMS100-10000 laser range ﬁnder, a laser based
2D ranging system, to attempt to better gain better accuracy for navigation based on laser data. Data was
gathered by placing the laser range ﬁnder in front of a door at speciﬁc distances and determining calibration
functions from polynomial regression. When tested the calibration functions worked very well for the set of
data of which they were produced and proved to have a small positive effect on a second set of data measured
with the laser range ﬁnder. Furthermore the calibration of the 2D LiDAR sensor was tested by having an
autonomous mobile robot drive towards a door and stop when the LiDAR sensor meausred the door within
a speciﬁc distance.More testing is necessary as the project was primarily focus on improving the distance
measurement directly in front of the LiDAR sensor and the sides of the sensor was not tested.

CHAPTER 1
Navigation using PX4Flow Smart Camera
1.1
Introduction
Initially the project was supposed to be navigation based on data from a Pixhawk PX4Flow Smart
Camera1 (will be referred to as "the camera"), in collaboration with wheel based odometry for an autonomous
mobile robot. In many cases a well calibrated wheel based odometry in combination with an localization
algorithm will work exceptionally well, however in this situation the camera was supposed to be used for
emergency stops, where the wheels of the robot would slip, as they were instantly stopped in an emergency
situation, with the result being that the odometry would no longer be fully reliable.
The issue was very apparent in testing, where a test person would walk in front of the driving robot,
it would then do an emergency stop causing the wheels to slip. If the robot only did a single emergency
stop, the localization system would still be able to determine the position of the robot, however if multiple
emergency stops where done, the position of the robot would become so skewed that the localization system
would be unable to identify the positioning from the surroundings. The idea was then to use the ﬂow camera
to gain information about the positioning of the robot during the skidding, so as to be able to accurately
determine where the robot ended after the emergency stop.
1.2
About the PX4Flow Smart Camera
The PX4Flow Smart Camera by Pixhawk, which can be seen in ﬁgure 1.1, is an image based ﬂow sensor
that is intended to work like an optical sensor of a mouse for autonomous drones. By registering the change
in images taken by the camera, the movement done by a drone can be determined.
The optical ﬂow is determined from cropped area of the CMOS sensor running at 400 Hz, in which sets
of 4 pixels are combined to increase the sensitivity to light, also known as 4x binning.
Each image is processed on the small on-board CPU, which then determines differences between frames
and calculates the ﬂow based on the known metrics of the sensor. The camera also features an on-board
sonar sensor3 to determine the distance to the ground and a on-board gyroscope to help determine rotation.

1.3
Connecting the Flow camera
The Pixhawk PX4Flow Smart Camera offers four different options for connections, I2C, JTAG, USART
and USB. Initially the idea was to use a Teensy Development Board4 to gather and relay the information, as
it was already used for the primary positioning measurements on the autonomous robot for which the system
was needed.
So, the Teensy was programmed to simply relay the information gained over USART to the USB, and
the information was read on the computer, however the expected clear lines of information, where very
scrambled as well as missing parts of the information.
From the information page about the ﬂow camera it was indicated that a total of three messages should
be expected; OPTICAL_FLOW, OPTICAL_FLOW_RAD and HEARTBEAT, all in a package structure by
the name of MAVLink.
After unscrambling the lines from the Teensy, it became apparent from the index of the MAVLink
message, that the only information gained was a simple heartbeat and the OPTICAL_FLOW_RAD message,
but full of zeros, except for the id and checksum.
In an attempt to troubleshoot, the ﬂow camera was connected through USB to a computer running
ROS5 with the px-ros-pkg6, which is a ROS-node that has a driver for the PX4Flow Smart Camera and
parses MAVLink messages, converts them to ROS-messages and publishes these for use within ROS. A
simple program was then written to log all the information from the camera, and the data then studied. The
HEARTBEAT message and the OPTICAL_FLOW message was received, and the ﬂow message included
data. The OPTICAL_FLOW_RAD message was simply not part of the px-ros-pkg node and all attempts at
adding the message to the driver, simply failed with the message never being received.

1.4
Testing the Flow Camera
1.4.1
Stationary testing
In order to test the camera for use in precise navigation with a mobile robot, the camera was mounted
on a robot running the simple ROS logging program. The camera was mounted on the front and facing the
hardwood ﬂoor at an altitude of 0.376 m. To test for position drift and check for the possibility of calibration
of the PX4 Flow Camera, data was logged while the robot was in stationary position. The data collected
was then studied in MATLAB. The position was calculated using the velocity measured by the ﬂow camera
and the delta time between measurements which in most cases was between 40 and 80 ms, at an average of
65.92 ms, however the largest deviation was a delta time of 320 ms and the standard deviation between them
was 23.83 ms.
As the position is calculated by integrating the speed, some drift in the position is expected and therefore
a possible calibration of the stationary velocity error would be ideal. An example of the velocity ﬂow
measured over time be seen in ﬁgure 1.2.
Figure 1.2: Velocity ﬂow measured over time on stationary robot
The data is just a single example of multiple measurements, where the direction and magnitude of the
erroneous velocity measurements all varies. Because of the, in this example, similarity between the the
positive and negative velocities on the x-axis, the position calculated from the measured velocity ends up
relatively close to the actual position on the x-axis, however due to the average of the velocity on the y-axis
being primarily on the negative side, the resulting position ends up some distance away from the actual
position. The effect of the erroneous velocity on the position over time can be seen on ﬁgure 1.3.
From just one of these measurements over time, it appears as if the drift in position could be eliminated
by doing a calibration routine which removes the effect of the slow drift on the y-axis, however when doing
the exact same measurement a second time, the position drift was in a different direction. Running the same
test multiple times showed that the drift in position was not systematic and therefore not possible to calibrate.

1.4.2
Experimentation complications
The Pixhawk PX4 Flow Camera is produced with airborne drones in mind, and therefore in this exact
use-case, the limitations that could be a non-issue in aeronautic use, can pose as problems when using the
camera at a constant distance to the ground. There was a number of factors which could have inﬂuenced the
measurements to introduce the erroneous velocities measured in the initial tests, these include, but are not
limited to;
1. Floor type
2. Measurement distance
3. Lighting
4. Interfering objects & camera focus
1 - Floor type
As the initial measurement was simply done on a random, dusty, generic hardwood ﬂoor in the AUT
laboratory, the contrast and the details of the ﬂoor was limited. As the ﬂow camera captures images and
compares the differences, if the details are blurry or there is a general lack of contrast, e.g. due to dust or
lack of grain in the wood, it can be difﬁcult for the camera to exactly determine the changes.
2 - Measurement distance
The measurement distance in the initial testing was a set speciﬁc altitude, due to the camera being
stationary mounted on a unmanned ground vehicle (UGV). Initially it was obvious that the distance measured
by the on-board sonar sensor on the camera was not intended for a UGV as it was limited to measurements
of at least 0.3 m, meaning anything less was still measured as 0.3 m.

According to the paper An Open Source and Open Hardware Embedded Metric Optical Flow CMOS
Camera for Indoor and Outdoor Applications by Dominik Honegger, Lorenz Meier, Petri Tanskanen and
Marc Pollefeys of ETH Zurich, Switzerland7, the camera uses the sonar distances measurements to convert to
metric velocity values from the optical ﬂow measured from the images, which means the limitation of 0.3
m could have an impact on the magnitude of the velocity measured at distances lower than that. From the
paper, it states that the translational velocity vm,trans in m/s, can be determined from the ﬂow camera using
equation (1.1) when the distance Z is constant, between two consecutive images, and the angular velocity is
either zero or can be compensated for (as in this case due to the on-board gyroscope).
vm = v · Z
f
(1.1)
where vm is the velocity of the movement, v is the optical ﬂow, Z is the distance to the ground and f is the
focal length of the camera.
From that, a conclusion can be drawn, that an erroneous distance measurement, likely due to the
limitations of the sonar on the Pixhawk PX4FLOW Smart Camera, can impact the validity of the velocity
measurements at low altitude.
3 - Lighting
In the publicity material of the Pixhawk PX4FLOW Smart Camera8, it is stated that due to the nature of
the fast updating and super light sensitive CMOS sensor, the camera should work both indoor and outdoor
and in both low and high illumination situations. Despite this, an experiment with adding an external light
source was planned.
4 - Interfering objects & camera focus
Having mounted the camera on the top, front-side of a robot, it is possible for another part of the robot to
actually interfere with the measurements. To avoid interference being an issue, the camera was mounted on
a bar extruding from the body of the robot. The mount can be seen on ﬁgure 1.4.
To ensure the body of the robot was not in the frame of the camera as well as the ﬂoor actually being in
focus, the software QGroundControl9 was loaded up on an external computer, the camera was connected
and the image from the camera was visually inspected and approved.
1.4.3
Further stationary testing
The issues mentioned in the experiment complications were considered and further stationary testing of
the Pixhawk PX4FLOW Smart Camera was done to investigate the issues raised. Primarily the ﬂoor type
and distance measurement were to be tested more systematically, by mounting the camera on a frame with
varying heights and running tests on different ﬂooring. In order to mount the camera on varying levels and
do a large number of tests, a small ofﬁce lamp was re-purposed to become the stable mounting point for the
camera at varying altitudes, it can be seen on ﬁgure 1.5.
Three types of ﬂooring was chosen and those were vinyl ﬂooring, chosen as it most closely represents
the ﬂooring in the situation where the camera information was needed, hardwood ﬂooring, to easily compare
with previous measurements, and a high contrast ﬂooring made from black and white printouts normally
used for camera based guidemark recognition, to have the ideal situation with very prominent contrast and

Results of High contrast ﬂoor testing
In the case of the high contrast ﬂooring, the measurements proved much better than the equivalent of

Realistically the high contrast ﬂooring is not represented in areas where it is desired to have an autonomous
unmanned ground vehicle running, e.g. a hospital or a warehouse.

Conclusion
In conclusion to the vinyl testing, vinyl did in no particular height, within the limitations for use in the
desired case, prove a good surface to measure the optical ﬂow on. With the majority of the measurements
being zero-quality, the camera would not provide any viable or credible information. The use-case dictates a
speciﬁc robot driving in multiple situations, including driving along hallways of a hospital with vinyl ﬂoors,
and looking at the results from testing, the camera would not provide any additional movement information,
and therefore it is not suited for that situation.
For hardwood ﬂooring, although there were actual measurements, the amount of non systematic errors in
the velocity proved that any use-case where the camera would need to be used to help determine movement,
the amount of drift in position over time would mean that it should be implemented in a way that all
information gathered while the robot was stationary should be ignored. That could easily be done by having
a state-machine that only considers the information from the Pixhawk PX4FLOW Smart Camera at those
times.
Testing on high contrast ﬂooring was primarily done in an attempt to gather some accurate movement
information from the camera, but although the drift in the position was up to a magnitude 6 times less than
the measurements done on the hardwood ﬂoor, there was still drift in various directions, which in turn is not
possible to calibrate.
The Pixhawk PX4FLOW Smart Camera is intended for use in aerial drones, where the errors in
positioning is not as mission critical, as it is in the emergency stop use-case. Therefore for the case of
determining the position changes during emergency stops on a UGV, the camera is not suited.
Alternative solutions
To solve the problem of lacking movement information during an emergency stop, a few different
solutions are possible.
One option would be to, instead of instantaneously stopping the wheels during an emergency stop,
instead decelerate the robot. By decelerating as opposed to just stopping the wheels, the amount of slipping
==PAGE:24==
12
CHAPTER 1. NAVIGATION USING PX4FLOW SMART CAMERA
should be severely reduced, this the odometry staying on course. This could have a negative effect on
the stopping distance, which depending on safety regulations might not be legal, as the robot has to stop
operations completely in an emergency.
Another option is to use a inertial measurement unit (IMU) to help determine the positioning when the
system raises the emergency stop ﬂag. By using the combination of a accelerometer and a gyroscope, the
velocity of the robot can be determined in the situation and, like done for the camera, the position could
be updated by using the time integral of the velocity. As with the ﬂow camera, this option can introduce
drift in the position, although it should be minimal if the IMU is only relied on in situations with spikes in
acceleration or deceleration.
Another option could be to change the localization algorithm to something that can use guide marks or
beacon signals to identify current position if lost.

2.2
About Light Detection And Ranging (LiDAR)
A LiDAR sensor is a range sensor based on illuminating a target with a laser, and measuring the returning
light and the timing in between. The range is estimated from the time taken for the light to return, based on
three different possible methods of measuring the distane.
The ﬁrst is to use a pulsing pulsing laser diode and directly measure the elapsed time, however due to the
cost of electronics capable of measuring picosecond differences, this method is not commonly used.
The second method is measuring the beat frequency, or the interference, between the reﬂected light and a
frequency-modulated continuous wave.
The third and most common method is measuring the phase-shift between the transmitted and reﬂected
signal. As the signal being used is a laser light at a speciﬁc modulated frequency, the wavelength has to obey
the following equation:
c = f · λ
(2.1)
where c is the speed of light, f is the modulating frequency and λ is the wavelength of the signal.
Using f = 2.5MHz, and λ = 120m the distanced travelled would be:
D′ = L + 2D = L + θ
2π λ
(2.2)
where D′ is the total distance traveled by the emitted light, D is the distance from between the mirror and the
object and L is the distance between the mirror and the photodiode and θ is the phase difference measured.
From that the distance is calculated by
D = λ
4π θ
(2.3)

The type data gathered from a 2D LiDAR sensor is polar coordinates, meaning the measured data is a
distance combined with an angle. Most measurements and calculations for a mobile robot is done using a
standard (x,y)-plane and therefore to ease the use the LiDAR data in combination with other measurements
from a mobile robot, the polar data has to be converted to Cartesian coordinates, which is done using the
trigonometric functions sine and cosine, see equations (2.4).
x = r · cos(θ)
y = r · sin(θ)
(2.4)
Where x and y is the position in the Cartesian coordinate system and r and θ is the distance and angle of the
measurement in a polar coordinate system.
When the polar coordinates has been converted to Cartesian, the laser data is still in the frame of the
LiDAR, meaning the distances are measured from the center of the 2D LiDAR sensor, and not the robot’s
reference frame. When using the LiDAR data for calculating a route, the data used for movement has to be
directly relatable to the position of which the routing is done, i.e. the center between the driving wheels of a
differential drive robot.
This conversion from the 2D LiDAR sensor’s frame to the robot’s reference frame is known as extrinsic
calibration and includes translation and rotation of the coordinate system, depending on the mounting of the

testing the laser measurements were done by logging every ﬁfth complete laser scan from the sensor. Driving
with the robot used, is done through SMR-CL and a plugin called "zoneobst" is used for gathering simple
laser scan data for use within SMR-CL.

was positive, with a small reduction in the largest errors and large improvements for the average error of
the measurements, as seen on ﬁgures 2.8 and 2.9 respectively. However when used on a separate data set,
the improvements from calibration were a lot less, and in some cases, the calibration of the measurement
largely increased the error as seen for the second order polynomial calibration of the 0.943 m measurement
in ﬁgures 2.10 and 2.11.
For the driving tests, there was a measurable improvement in the consistency between the distance
measured to the door by the LiDAR, and that measured by hand, for the ﬁrst and second order calibration
equations. For the 0.4 meter test, the maximum error was down to just 4 mm for the ﬁrst order calibration
with an average of 0.37. For both the 0.5 and 0.6 m tests, the second order calibration managed to stay at 5
mm in absolute error.
The other data gathered from the driving test was very similar to that of 0.4 meters with the most notable
thing being a complete misreading for 0.5 meters, third order equation, causing the robot to stop with more
than a meter to go. As the very large deviation from the norm was completely inconsistent with the other
measurements, the test was rerun and on the following three runs, the robot stopped within the expected

2.7.2
Future work
N The method of calibration done in this project only considers the error in the distance measurement
of the LiDAR and only the front part of the sensor has been tested. The testing should ideally also have
included experiments studying the measurements from the sides of the sensor. Experiments could have been
done using the same method as the stationary measurements, by simply placing the LiDAR sensor a speciﬁc
distance away from a door and record the data, although at an angle. That way any irregularities, such as
asynchronous measurements of the sensor, could have been determined, furthermore it should be noted that
the calibration determined is only for the x-axis of the sensor, and has not been tested for the y-axis, although
it should also work if the systematic error is only due to the method of measuring the distance and does not
originate from an angular error, i.e if the error comes from errors within the pulse modulation.
To ensure the measurements were as precise as they could become, the θ angle of the laser scanner
should also be considered, as a small error in the angle has a large impact on the position when converted to
a Cartesian coordinate system. One method of determining the accuracy of the angle could be placing a thin
object at precise angles relative to the LiDAR sensor and comparing the known angle with that measured
angle.
Ideally more navigation using the calibrated 2D LiDAR sensor data should have been done. One method
could have been to do landmark localization using the laser scanner to ﬁnd an object of a speciﬁc shape to
determine accurate location relative to the landmark.
Another option could be to use the calibrated LiDAR data to localization. Running Kalman ﬁlter
localization without calibration versus with calibration and seeing if the calibration had an impact on the
quality of the localization.
2.8
Conclusion
To conclude on the possibility of calibrating 2D LiDAR data, the data gained from my experiments
show that at least in some speciﬁc cases, the calibration can have a possitive impact. In testing the SICK
LMS100-10000 laser range ﬁnder, the magnitude of the errors measured was small before starting, however
it was still possible to reduce the both the maximum error and average error in most cases after calibration.
When using the calibration for a test of autonomous driving, the results were positive with the statistics
showing improved accuracy on for all three orders of calibration equations.
Over all the primary project problem was to investigate the posiblity of calibrating and improving the
accuracy of 2D LiDAR sensor measurements, which has determined is possible. Secondly if improved
accuracy had an impact on navigation for a mobile robot using the sensor, which was tested to be true as
well, although the magnitude of the improvements was not major as the error to start off with was very small.
