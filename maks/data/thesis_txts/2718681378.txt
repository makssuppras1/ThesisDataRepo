String Indexing with Compressed Patterns
Mathias Søndergaard
Supervisors: Philip Bille and Inge Li Gørtz
Technical University of Denmark
16th January, 2023
Abstract
We consider the problem of indexing a string S to support subsequent compressed pattern queries. The
problem is similar to the well-known string indexing problem, but with the twist that the patterns are given
in compressed form. Recent results show that the problem can be solved efficiently by preprocessing S into
a compact data structure known as the phrase trie. Let n be the length of S and let Φ be the sum of
the longest-common-prefix array over S. Our main theoretical contribution is that of a new preprocessing
algorithm which constructs the phrase trie over S in O(n + Φ) expected time and O(n) space. Moreover we
present observations regarding the Lempel-Ziv-77 compression scheme that may be of independent interest,
including a lazy algorithm that compresses all suffixes of S. Additionally, we show that in practice the phrase
trie outperforms naive approaches when answering the compressed pattern queries.

Introduction
In the string indexing with compressed patterns problem, or SICP for short, we are given a string S and
the compressed form of a string P, with the objective of reporting all occurrences of P within S. More
specifically, we want to report the starting positions in S where P is a substring.
Within the realm of
computing, the problem has several intersting applications, such as server-client communication. Here a
client reduces bandwidth by compressing P before sending it as a request to the server. Naively, the server
would then decompress P, transforming the SICP problem to the well-known string indexing problem, which
introduces unwanted overhead.
Recently, novel solutions for the SICP problem have been proposed. In [3] the concept of phrase tries were
presented, which in essence is a compact data structure over S supporting fast SICP queries. Phrase tries can
be constructed with different compression schemes. In this study we will focus on the popular Lempel-Ziv
77 scheme [22] (LZ77). To parameterize the complexities of the phrase trie, we first introduce the following
variables. Let n be the length of S, occ be the number of elements reported and let zP be the length of
LZ(P), where LZ(P) denotes the LZ77 compressed form of P. The phrase trie supports O(zP +log n+occ)
query time and a space complexity of either O(n) or O(n2), with the former being known as the phrase
trie with linear space and the latter known as the phrase trie with quadratic space. In comparison, a naive
solution would maintain a suffix tree over S. To answer a query, we would decompress LZ(P) and search
for P in the suffix tree. With m denoting the length of P, the naive solution has O(m + occ) query time and
O(n) space and preprocessing time [10, p. 122].
As the phrase tries were recently discovered, and to the best of our knowledge, the tries have not been
tested in practice. Thus, our first goal was to determine if the phrase tries function in practice, and more
importantly, we investigate whether the complexities hold in a practical setting. Let zi denote the length of
LZ(Si), where Si is the ith suffix of S. An algorithm to construct the phrase trie with quadratic space was
given in [3] using O(n√log n+Pn−1
i=0 zi log log n) expected time and O(n log log n) auxiliary space. In regards
to the phrase trie with linear space, no non-trivial solution exists. A naive solution would be to transform
the phrase trie with quadratic space to the phrase trie with linear space. However, this is infeasible for large
n, as O(n2) space is needed at some point. Hereto we can present the main theoretical finding in form of
theorem 1.
Theorem 1: Given a string S, let n be the length of S and let Φ be the sum of the longest-common-prefix
array over S. The phrase trie with linear space can be constructed in O(n + Φ) expected time and O(n)
space.
To obtain theorem 1, we will first explore preprocessing algorithms for the phrase trie with quadratic
space. Whilst doing so, we will show an algorithm that solves the problem of compressing all suffixes of S.
Finally, we will combine the covered material with properties of LZ77 parsing to show theorem 1.
Our next finding is that of a practical result. First we demonstrate that the phrase tries function in
practice and that theorem 1 holds in a practical setting. Thereafter we let S be a string of protein sequences
where n = 106. We construct the phrase trie with linear space and the suffix tree over S. We show that SICP
queries applied on S with patterns that are longer than 429 (m > 429), are computed fastest when utilizing
the phrase trie with linear space. We note that for smaller patterns, the two data structures perform almost
equally.

Preliminaries
Let S and P be two strings of lengths n and m respectively. P is commonly referred to as the pattern and
S is referred to as the indexed string. In the SICP scenario, we assume n is greater than m and that both
strings are drawn from the alphabet Σ. In this project, we treat a string as an array of characters, where
accessing the ith character of S is denoted S[i]. We define the (i, j)-substring of S, denoted S[i, j], as the
subarray representing the characters from the ith position of S to the jth position (including both). The ith
suffix is defined as S[i, n −1] and denoted Si. The ith prefix is defined as S[0, i] and denoted Si.

Suffix array
The suffix array, sometimes named SA, describes the lexicographical order of all suffixes
belonging to a string. SA is sorted in increasing order. That is, SA[0] stores the starting position of the
suffix with lowest lexicographical value. The length of SA is equal to the length of the string it describes.
The inverse suffix array SA−1 is defined as: 0 ≤i < n : SA−1[SA[i]] = i. To construct the suffix array, all
suffixes need to be sorted. In this project the DivSufSort algorithm [7] is used. We assume the size of the
alphabet is linear, thereby the suffixes can be sorted in O(n) time [14]. We will occasionally use the notation
SAi to denote the suffix array of a specific suffix Si, with SA0 denoting the suffix array of S.
Range minimum queries
Range minimum queries, or RMQ for short, solves the problem of find-
ing the minimum element occurring between two indexes (i, j) of A.
This is defined as RMQA(i, j) =
mink∈A[i,j]{A[k]}. Different data structures support RMQ in O(1) query time with O(n) space and prepos-
sessing time. The solution shown in [6] is used in this project.
Longest common prefix
The longest common prefix of two strings S and S′, denoted LCP(S, S′), is the
maximal i, such that the ith prefix of the two strings are equal. We are only interested in the LCP between
suffixes of S. Furthermore, we are only interested in the length of the prefix, and not the actual prefix, so
we assume that LCP(S, S′) returns an integer corresponding to the length of the longest common prefix.
Longest common prefix array
The LCP-array, named LCPA, is an auxiliary data structure of the suffix
array. It stores the length of the longest common prefix between a suffix and its predecessor in the suffix
array. Formally this is: 0 < i < n : LCPA[i] = LCP(SSA[i−1], SSA[i]). The LCP-array can be computed using
Kasai’s algorithm [12], which given the suffix array requires O(n) preprocessing time.
Longest common prefix data structure
The LCP-array can only answer LCP queries between two
suffixes adjacent in the suffix array. However, we are interested in answering LCP queries between any two
suffixes. That is, for any two suffixes Si and Sj, the goal is to answer LCP(Si, Sj) in O(1) query time using
a data structure with O(n) space and prepossessing time. From now on, it is assumed that any LCP query
is computed by the following:
We start by preprocessing SA, SA−1 and LCPA. Afterwards a RMQ data structure is constructed over
LCPA, returning RMQLCP A. Finally the LCP(Si, Sj) query can be answered by computing:
LCP(Si, Sj) = RMQLCP A(SA−1[i] + 1, SA−1[j]).
The data structures used to answer the query require O(n) prepossessing time and space. The query
time is O(1) by the use of the RMQ data structure. The appendix contains a further description of why the
method is correct.
Lempel-Ziv 77 compression scheme
The Lempel-Ziv 77 representation of a string S is a sequence of
z phrases. This is defined as LZ(S) = f1, f2, . . . , fz. We use the notation given in [11]. A phrase f = (p, l)
is a tuple which encodes a substring S′ starting in position x of S. Given fi = (pi, li), the substring which
fi represents is S[pi, pi + li]. So S′ = S[pi, pi + li] = S[xi, xi + li]. Moreover, fi must be selected such that
li is maximized in a greedy fashion and xi > pi. If no such phrase can be found, so li = 0, S[xi] must be a
character not occurring in S[0, xi −1]. Here fi is encoded as fi = (S[xi], 0).
If xi −pi < li then fi is called a self-referencing phrase. In this project we allow phrases which has this
property. In some literature, the definition of a phrase is f = (r, l). We will strictly use f = (p, l), however
these can be interchanged by f = (r, l) = (x −p, l).
The KKP3 algorithm presented in [11] can be used to compute LZ(S). Given the suffix array it uses O(n)
time and space. It is assumed that all compression algorithms are deterministic. We note that LZ(S) is
used as an array in this project. So LZ(S)[k] is the kth phrase and LZ(S)[i, j] is a subarray of phrases.
Finally, given a position x in S, we define LZR(x, S) as the restricted LZ77 compression of S. Assuming
that x is contained in phrase k of LZ(S), we have that LZR(x, S) = LZ(S)[0, k]. So, the second last phrase
of LZR(x, S) contains x, as we assume 0-indexing.

Tries
Let D = {d1, d2, ..., du} be a set of strings. A trie is a rooted tree which contains the set. Each edge
contains one character from the alphabet as its label, denoted el. Let v be a node, di be a string from D
and let str(v) be the string obtained by concatenating the edge labels of the root-to-v path. If str(v) = di, v
contains a pointer to di. The main requirements of the trie are given as follows: All strings in D are pointed
to once. A node v has a pointer to di iff str(v) = di. And that for all shared prefixes between strings in D,
a shared prefix is represented by one path starting from the root in the trie.
If the strings of D are prefix free, the pointers to D will be contained in leaves of the trie. The trie uses
O(Pi=u
i=1 |di|) space. For routing we assume that the outgoing edges of a node are stored in a dictionary
(hash map). This is known as the children dictionary. Here a key is a character, and the value is an edge
with the corresponding label.
Compact Tries
Let D be defined as above. For simplicity we assume that the strings are prefix free.
The compact trie is a trie with one modification. Rather than saving one character per edge, we now save a
substring of a string in D (el is a string instead of a character). We do so maximally, such that each internal
node in the compact trie has at least two children. That is, given a trie, we obtain the compact trie by
the following: For each path in the trie consisting of nodes with one child, we replace the path by a single
edge. The label of a new edge is the concatenation of the characters from the replaced edges. We update the
children dictionary to reflect these changes: A key to the children dictionary is a character and the value is
an edge whose label starts with that character. The theoretical space of a compact trie is equal to that of a
trie, since the edges still contain the strings of D. However, the number of vertices and edges is reduced to
O(|D|).
Suffix trees
Given a string S and the set of suffixes D = {S0, S1, S2, ..., Sn−1}, the suffix tree is a compact
trie over D. Opposed to the compact trie, the suffix tree can be stored in O(n) space, as each edge label
can be stored by keeping two integers [21]. It is assumed that S[n −1] is an unique character. With this
assumption the leaves of the suffix tree and their root-to-leaf paths represents the suffixes of S. The suffix
tree can be constructed in O(n) time using Ukkonen’s algorithm [20] or by using LCPA and the suffix array
[17].
Matching in tries
The matching algorithm, named match, is used as a subroutine in many trie algorithms.
The following is an explanation of how it functions: Given a pattern P and a trie, we set k = 0 and begin
from the root. We check if the key P[k] exists in the children dictionary. If so, we obtain an edge e. We set
i = 0 and check whether P[k] = el[i], incrementing i and k by one for each successful character comparison.
At some point i = |el|, and we repeat the process from the child node. The algorithm terminates when a
character comparison is unsuccessful or when P has been fully matched. We define a pointer named locus(P)
by the four cases which occur while matching:
The matching failed in edge e: P[k] did not equal el[i]. Here we set locus(P) = e.
The matching failed in node v: P[k] did not exist in the children dictionary of v. Here we set locus(P) =
v.
The matching was successful, ending in node v. Here we set locus(P) = v.
The matching was successful, ending in edge e = (parent, child). Here we set locus(P) = child.
In this project the matching algorithm returns whether the match was successful or unsuccessful and
the locus pointer, denoted locus(P). Using perfect hashing [8] for the children dictionaries we can perform
look-ups in O(1) time. The time complexity for the match algorithm is then O(m). Finally, as an example
we show how to answer the string indexing problem using the match algorithm. Given a pattern P and the
suffix tree over S we apply match on P. If match is successful, we report the pointers to D located in leaves
below locus(P) as the starting positions of P in S. If match is unsuccessful we report nothing.

Phrase Trie
In this section we give an overview of the two phrase trie variants. Both process SICP queries in O(zP +
log n+occ) time. This overview is a simplification of sections 3 and 4 in [3]. The goal is to highlight omitted
details which are relevant for implementing the phrase tries in practice.
3.1
Phrase Trie with quadratic space
Given a string S the phrase trie with quadratic space is a compact trie over D = {LZ(S0), LZ(S1), .., LZ(Sn−1)}.
Opposed to the compact trie, we now store phrases as the edge labels instead of substrings, so el is an array
of phrases and a key to a children dictionary is a phrase. It is again assumed that S[n −1] is a special
letter which is lexicographical greater than the other letters. Before exploring the phrase trie further, two
important observations regarding compression and LCP are described in the form of corollary 1 and lemma
1:
Corollary 1: Given two strings S and S’, let LCP(S, S′) = c. Assume that S[c] is contained in LZ(S)[k]

1. LZ(S)[0, k′ −1] = LZ(S′)[0, k′ −1] and LZ(S)[k′] ̸= LZ(S′)[k′] or
2. LZ(S)[0, k′ −2] = LZ(S′)[0, k′ −2] and LZ(S)[k′ −1] ̸= LZ(S′)[k′ −1].

3.1.1
The data structure
We enhance the described phrase trie with auxiliary data. Each internal node v stores a range over the
suffix array, which the leaves below v cover. Each leaf stores an integer which corresponds to the suffix it
represents. As the phrase trie is a compact trie, each node also contains the children dictionary. Moreover,
the suffix array and a LCP data structure over S are stored.
Figure 1: Phrase trie with quadratic space over S = ababa$.
3.1.2
The query algorithm
Given LZ(P), we want to report the starting position(s) of P in S. The algorithm is split into 4 cases, which
cover all possibilities. In all cases we begin by applying match on LZ(P) with the phrase trie.
Case 1: If match returns success, that is all the phrases in LZ(P) were matched, we obtain the range [i, j]
stored in locus(LZ(P)) (due to the definition of match, the locus is always a node in this case). We then report
suffixes corresponding to this range and the following: Starting from SA[i] we check whether the condition
LCP(SSA[i], SSA[i−1]) ≥|P| holds. If it holds, SA[i−1] is also reported. Afterwards LCP(SSA[i], SSA[i−2]) ≥|P|
is checked and so on. If the condition fails or the end of SA is reached, the search to the left of SA[i] is
terminated. Similarly we check to the right of SA[j]. Simply put, we report suffixes that neighbor the range
and fulfill the condition.
It is now assumed that match fails. Here, locus(LZ(P)) can either be an edge e = (parent, child) or a
node v. Let LZ(P)[k] = fk = (pk, lk) be the phrase which failed during the matching, and let xk be the
starting position of fk in P.
Case 2: Independent of the locus, if fk is a single character, we report nothing.
Case 3: We assume that locus(LZ(P)) = e, and define Sy as a suffix representing a leaf below e. Then
we check whether xk + min(LCP(Sy+xk, Sy+pk), lk) = |P| holds. If it holds we report the elements in SA[i, j],
where [i, j] corresponds to the range saved in child. If it does not hold, we report nothing.
Case 4: We assume that locus(LZ(P)) = v, and initialize a binary search over the range [i, j] stored in
v: Let m = ⌊i+j
2 ⌋and t = xk + min(LCP(Sm+xk, Sm+pk), lk). We then check if t = |P| holds. If it holds, we
repeat the process of reporting neighbors w.r.t SA[m] shown in the first case (checking neighbors to the left
and right of SA[m]). Otherwise we continue the search. If S[m + t] > S[m + t −(xk −pk)] we continue the
search to the left by setting j = m. Otherwise we set i = m and continue the search to the right. Nothing
is reported if the search terminates without finding Sm fulfilling the condition.
3.1.3
Correctness
We divide the analysis into the four cases.
Case 1: If match is successful, it is trivial that the range SA[i, j] should be reported. Furthermore, simply
reporting the range may result in the omission of other correct positions. This is due to corollary 1, and so
we must also check neighbors that are adjacent to the range. We know that LCP(P, SSA[i]) = |P| as match
was successful, entailing that P is a prefix of SSA[i−1] if LCP(SSA[i], SSA[i−1]) ≥|P|. Moreover, we can stop

also be included in the output.
Case 2: It is trivial that if the failing phrase, fk, is a single character, all suffixes corresponding to leaves
below locus(LZ(P)) do not contain the character at position xk. Thus P cannot be a prefix of any suffix,
and nothing is reported.
Case 3: Here we apply lemma 1 directly with respect to P and Sy. Essentially we compute the following
LCP(Sy, P) = xk + min(LCP(Sy+xk, Sy+pk), lk)
Where Sy corresponds to a leaf below e. If LCP(Sy, P) = |P|, then Sy should be reported. Moreover, we
report the range [i, j]. This is correct as all suffixes SSA[y′] with y′ ∈[i, j] share prefixes maximally with Sy.
Knowing that LCP(Sy, P) = |P| and that the locus is an edge, entails that LCP(SSA[y′], P) = |P|.
Case 4: By corollary 1, we have the possibility of P being a prefix of Sy whilst match being unsuccessful
with locus(LZ(P)) = v. Assuming P is a suffix of Sy, we know that Sy must correspond to a suffix stored in
the range of v. Because the suffix array is sorted in lexicographical order, SA[i, j] is sorted as well. Thereby
we can do a binary search on SA[i, j] to find Sy. This is done by routing the search using t = LCP(Sm, P)
(lemma 1). If P is not a prefix of Sm, the search must continue. Now P is either lexicographical smaller or
larger than Sm. So we check whether Sm[t] = S[m + t] > P[t] = S[m + t −(xk −pk)] holds. If it holds, we
search to the left, as Sy will also be lexicographical smaller than Sm (P is a prefix of Sy). This shows that
routing the search by lemma 1 is sufficient for finding Sy. After finding Sy, we check the neighbors in SA,
which was shown in case 1 to yield the correct occurrences.
3.1.4
Analysis
The use of match is what the cases have in common. This requires O(zP ) time. In cases 1,2 and 4 we also
compute the length of P which can be done in O(zP ) time. Occurrences are reported either by using the
[i, j] range, or by checking neighbors with LCP. Both use O(occ) time. For the fourth case a binary search
is performed on the range, where j −i = O(n). The final time is therefore O(zP + log n + occ). For the
space complexity, the auxiliary data use O(n). Also O(Pn−1
i=0 zi) = O(n2) is used to store the edge labels,
showing that O(n2) space is used in total.
3.2
Phrase Trie with linear space
We now show how to reduce the space from O(n2) to O(n). Trivially, we can no longer store all phrases
of the compressed suffixes. To obtain the desired results, we must instead rely on the structure of LZ77
compression and additional auxiliary information with a modified match algorithm.
3.2.1
The data structure
We store the exact same data structures as in the phrase trie with quadratic space, except a few modifications:
An edge no longer stores phrases, but the length of the substring in S which the edge represents. An edge
also has a pointer to a leaf below it.

3.2.2
The query algorithm
The input and output is the same as in the query algorithm for phrase tries with quadratic space. We can
no longer rely on the match algorithm as before, due to the edges no longer containing the needed phrases.
Consequently, we change match to function on phrase tries with linear space: Setting k = 0 we start from
the root, checking whether LZ(P)[k] exists as a key in the children dictionary. If it does, we set l = 0 and
fetch the corresponding edge e, which contains a leaf below it representing the suffix Sy. e is now matched
on with the available information, during which two cases occur. First, if LZ(P)[k] = fk = (pk, lk) is a
letter (by definition lk = 0, with pk containing the letter), we check pk = S[y + xk] where xk is the starting
position of fk in P. If the condition is true, we increment k by 1, and continue. If not, match is unsuccessful.
Secondly, if fk is not a letter we check LCP(Sy+pk, Sy+xk) ≥lk. If the condition is true, we increment k by
1. Otherwise match is unsuccessful. For each successful match we add lk to l. At some point, l equals the
length stored in e. When this happens, we repeat the process at the child node. Finally, when LZ(P) has
been fully matched, match is successful. We define locus(LZ(P)) as in the preliminaries, that is when match
is successful, locus(LZ(P)) is always a node. Otherwise locus(LZ(P)) is the failing node or edge.
With match fitted to function on phrase tries with linear space, we repeat the exact same steps as before.
That is, we apply match and continue by whatever case locus(LZ(P)) falls into.
3.2.3
Correctness
We only consider the correctness of the modified match algorithm, as the remaining steps are equal to
those seen before.
If fk is a single character, it is trivial that checking for this character in S[y + xk]
is correct. Otherwise we use LCP to check for a match. More specifically, we use lemma 1 to determine
whether the match holds. Recall that fk represents the substring P[pk, pk + lk] = P[xk, xk + lk]. Since
we have matched the first k phrases, we know that P[0, xk] = Sy[0, xk]. So we use LCP and lemma 1 to
check whether Sy[pk, pk + lk] = Sy[xk, xk + lk] is true.
If it is, then LCP(Sy+pk, Sy+xk) ≥lk holds and
P[0, xk + lk] = S[0, xk + lk], showing that fk is successfully matched. We can repeat the argument for all k,
leading to the modified match algorithm being correct.
3.2.4
Analysis
In regards to time, the modified match algorithm applies either one LCP query or one dictionary lookup for
each phrase in LZ(P), requiring O(zP ) for all phrases. The remaining steps are equal to those previously
seen, so the time complexity is O(zP + log n + occ). Besides storing the same auxiliary data structures as
before, we no longer store all phrases, but constant information for each edge and node. Therefore the space
complexity is O(n).

4
Lazy Consecutive Suffix Compression
In order to preprocess the phrase trie with quadratic space, all suffixes of S must be compressed. A naive
approach would be to compress each suffix independently.
Using the KKP3 algorithm [11], this requires
O(P
i |Si|) = O(n2) time and involves computing the suffix array and auxiliary arrays for each single suffix,
which has a relatively large overhead. KKP3 was designed to compress strings which are independent of
each other, whereas we are now interested in compressing suffixes. In [3] a suffix compression algorithm
is proposed. It uses O(n√log n + Pn−1
i=0 zi log log n) time to compress all suffixes and O(n log log n) space.
The idea is built on results given in [13] and [9]. Albeit being fast, implementing and covering the required
data structures are outside the scope for this project. Therefore, we introduce the lazy consecutive suffix
compression algorithm, or CSC for short. The purpose of CSC is to be simplistic and faster than KKP3 in
practice. Moreover it should run in O(n2) time and O(n) auxiliary space.
The contents of this section is as follows: We split the problem of compressing multiple suffixes into three
sub-problems. The first problem is to compute a phrase starting in position x of Si. The second problem
is to compress Si. The final problem is to compress all suffixes of S. The solution to problem 3 will be the
main routine of the CSC algorithm.
4.1
Computing an arbitrary phrase of Si
We use the KKP3 algorithm as inspiration. It computes a phrase by finding NSV and PSV (next and previous
smaller values respectively) [11] for position x in S. We must first redefine NSV and PSV for our given
context, since we are no longer working with independent strings. Recall that the context is to compute a
phrase starting at position x in Si.
Local lexicographical predecessor (LLP): Given Si and the position x, LLP is a suffix Sj of S such
that Sj is the first lexicographical predecessor of Si+x satisfying i ≤j < i + x.
Local lexicographical successor (LLS): Given Si and the position x, LLS is a suffix Sj of S such
that Sj is the first lexicographical successor of Si+x satisfying i ≤j < i + x.
If LLP or LLS do not exist for a given position, it is set as the empty string. Informally, LLP is a suffix
with starting position between i and i + x in S. The notion of first lexicographical predecessor means that
LLP must have a smaller lexicographical value than Si+x. Moreover, if multiple lexicographical predecessors
of Si+x exist, the one most similar to Si+x is chosen (similar meaning that they have the smallest distance
from each other in the suffix array).
Now to compute the actual phrase, f = (p, l), we compare the values of LCP(LLP, Si+x) and LCP(LLS, Si+x).
To determine the length l, we select the larger of the two. Assuming that LCP(LLP, Si+x) ≥LCP(LLS, Si+x)
we select the starting position of LLP in Si as the position p. If both LCP calls return 0, Si[x] must be an
unseen character, so f = (Si[x], 0). Due to the definition of LCP, this method clearly returns the longest
phrase that is available in position x of Si. Further details are given in [4]. We have now shown how to solve
problem 1, albeit no concrete algorithm has been given.
4.2
Compressing an arbitrary suffix
We are now interested in finding all LLP and LLS for the starting positions of phrases in Si. Using the
method shown in section 4.1 we can compress Si.
Given the suffix Si, SA and SA−1, we proceeded as follows. We set x = 0 and compute LLP by doing
a linear search in SA starting from position SA−1[x + i]. We search to the left of this position, returning
the first entry j which satisfies the conditions given in the definition of LLP. LLS is computed in a similar
fashion, but by searching to the right of the position. LLP is correctly computed, as we search left in the
suffix array when finding the predecessor, and return the first entry fulfilling the given conditions. Similarly
for LLS. Thereafter we apply the same steps as described above to yield the phrase. x is then incremented
10
==PAGE:11==
by the length of the phrase, repeating the process until x = |Si|. The time complexity is O(|Si| · n) = O(n2)
as for each character of Si there is a possibility of two linear searches being performed on SA.
The algorithm given here performs excitingly bad on a single suffix, however it solves problem 2. Moreover,
one may question why we use SA and not SAi.
The reason hereto is that we can reuse SA for every
compression, thereby removing some overhead.
4.3
Compressing all suffixes of S
To solve problem 3, one could apply the algorithm shown in the previous section on all suffixes, but this
would require O(n3) time. We now assume that we compress the suffixes in suffix order. That is, we compress
Si, then Si+1 and so on. With this assumption we make the following observations:
Observation 1: The difference between Si and Si−1 is always 1 character, being Si−1[0]. Therefore the
phrases of Si and Si−1 could be similar and reused.
Observation 2: We increment i by 1 for each compression iteration, thus the constraint i ≤j < i + x
behaves in a predictable manner. Therefore we could use the LLP and LLS suffixes obtained from computing
LZ(Si−1) to compute LZ(Si).
To solve problem 3, we now use the two observations in combination with the solution to problem 2.
First we formalize observation 1 in the form of lemma 2.
Lemma 2:
Let f be a phrase of LZ(Si) with starting position x in Si and let f ′ = (p′, l′) be a phrase of
LZ(Si−1). If f ′ has starting position x + 1 in Si−1 and p′ ̸= 0, then f = (p′ −1, l′).
Proof. Without loss of generality, assume that LZ(Si−1) and LZ(Si) were computed by the algorithm shown
in section 4.2. When f ′ was computed, the corresponding LLP’ and LLS’ were found among suffixes with
starting position between 0 and x + 1 in Si−1. As p′ ̸= 0, LLP’ and LLS’ must be found among suffixes
with starting position between 1 and x + 1 in Si−1. For f, LLP and LLS were found among suffixes with
starting position between 0 and x in Si. Now consider the two substrings Si[0, x −1] and Si−1[1, x]. Due to
the definition of a suffix, these are equal. As the compression algorithm is deterministic and the two phrases
f and f ′ were computed while having the same context, being Si[0, x −1] and Si−1[1, x] respectively, shows
that f = (p′ −1, l′).
Unfortunately lemma 2 does not improve the theoretical time complexity, but it does improve the per-
formance in a practical setting. Next up, we formalize observation 2 in the form of lemma 3. The intention
is to use dynamic programming on LLP and LLS.
Lemma 3: Let S0, S1, ..., Sn−1 be the order which the suffixes of S are compressed in. Let k be a position
of S. Finding LLP and LLS at position k for all suffixes require O(n) time.
Proof. Assume that Si is the first suffix which has a phrase starting at position k of S. To find LLPi,k we
do as described in section 4.2, that is we search to the left of index SA−1[k] in SA. Assume that LLPi,k
is found at index c in SA. That is, the entries in SA[c, SA−1[k]] have been checked to find LLPi,k. Now
assume Sy is the second suffix which has a phrase starting at position k in S. Instead of starting the search
from SA−1[k] to find LLPy,k, we start the search from c. Assuming that the search stops in c′, the entries
in SA[c′, c] have been checked. We now claim that LLPi,k and LLPy,k are correctly computed. The former
is correctly computed as described in section 4.2. For LLPy,k we consider the upper bound given in the
definition of LLP. We define this as j < i + xi and j < y + xy for the ith and yth iteration respectively.
As the position k does not change, we have that i + xi = y + xy. Which shows that the upper bound has
not changed between the two iterations. The lower bound, i ≤j and y ≤j for the ith and yth iteration
respectively, becomes more tight between the iterations as i < y. Thus the bounds on LLPy,k are tighter
compared to LLPi,k. Combining this with the fact that LLPi,k was not found in SA[c, SA−1[k]], shows that
LLPy,k cannot be found there as well. Consequently it is sufficient to begin the search from c. Continuing
by induction shows that this holds for any suffix that has a phrase with starting position k in S, due to the
11
==PAGE:12==
suffix order. As the length of SA is n, we will check at most O(n) entries to find LLP at position k for all
suffixes. This argument also holds for LLS, if we search to the right in SA.
A property of lemma 3 is that it can be applied for all positions in S. Thus the time to find all needed
LLP and LLS becomes O(n2). We now combine our results to show the CSC algorithm in its full strength.
4.4
The CSC algorithm
Given S we proceed as follows. First we compute SA, SA−1 and a LCP data structure over S. Then we set
i = x = 0 and initialise two arrays LLPA and LLSA of size n. Initially all entries of these two are −2. Now
we start the compression of Si. First we preprocess an array previous of size |Si|. Here previous[j] = −1
means that no phrase in LZ(Si−1) represents a substring starting in position j + 1 of Si−1. Otherwise
LZ(Si−1)[previous[j]] = f ′ is a phrase where f ′ represents a substring starting in position j +1 of Si−1. For
S0, previous is not initialized and any steps below using previous are skipped.
A phrase is computed by first checking whether lemma 2 applies. If LZ(Si−1)[previous[x]] = f ′ = (p′, l′)
with p′ ̸= 0, the next phrase is f = (p′ −1, l′).
If f ′ could not be found or used, lemma 3 is applied. We access LLPA[i + x] = c to find LLP. Three
possibilities occur:
1. If c = −2 we search for LLP as in section 4.2; by using SA and starting in position SA−1[i + x].
2. If c = −1 we set LLP as the empty string.
3. Otherwise we search as explained in lemma 3; by searching to the left of c in SA.
If no LLP can be found by the searches in 1. or 3., we set LLP as the empty string and LLPA[i + x] = −1.
Otherwise, LLPA[i + x] is updated to store the index in SA where LLP was found.
Similar steps are
applied for finding LLS. Afterwards the next phrase f = (p, l) is computed by comparing LCP(LLP, Si+x)
and LCP(LLS, Si+x) as described in section 4.1. We continue with the next phrase by setting x = x + l.
If x = |Si|, the suffix has been completely compressed and we continue with the next suffix by setting
x = 0 and i = i + 1. After compressing Sn−1 the algorithm terminates.

4.4.1
Correctness
A phrase in LZ(Si) is computed by either applying lemma 2 or by applying LCP twice. Due to the correctness
of lemma 2, the phrases obtained thereby are correct. The method of applying LCP with LLP and LLS was
shown in section 4.1 to return the correct phrase. Also, we correctly find LLP and LLS by lemma 3. Finally,
for each suffix, we iterate over each position that is a starting position of a phrase. Combining these shows
that all phrases are correctly computed.
4.4.2
Analysis
For space, we need to store S, SA, SA−1, LLPA, LLSA and a LCP data structure. All require O(n) space.
During compression we also need to store LZ(Si−1) and previous using O(|Si| + zi) = O(n) space. If we are
to store all compressed suffixes, O(Pn−1
i=0 zi) space is required. However, for the actual compression, these
do not need to be stored. So, in total the algorithm uses O(n) space, and if needed returns the result of size
O(Pn−1
i=0 zi).
In relation to time, we need to compute SA, SA−1, initialize two arrays of size n and compute the LCP
data structure. These require O(n) time. Computing previous requires O(|Si|) time for each suffix, being
O(Pn−1
i=0 |Si|) = O(n2) for all suffixes.
Applying lemma 2 requires constant time by usage of previous.
Lemma 2 can be applied at most O(zi−1) times for suffix Si, resulting in O(Pn−1
i=1 zi) = O(n2) time for all
suffixes. Worst case, all LLP and LLS values need to be found, and by applying lemma 3 on all positions
of S, require O(n2) time. Using LCP to find lengths require O(1) time and can happen at most twice per
phrase. Thus the final time complexity becomes O(n2).
5
Preprocessing the Phrase Trie with quadratic space
We now show how to construct the phrase trie with quadratic space. First, the data structures needed for the
CSC algorithm are computed. This includes SA, SA−1 and a LCP data structure over S. Next we initialize
the phrase trie by adding a root-node. We set i = 0 and compute LZ(Si) using CSC. LZ(Si) is then inserted
into the phrase trie: First we use match(LZ(Si)) to find the locus. If the locus is a node v, we add a node
child to v by an edge e = (v, child). Assuming that k phrases of LZ(Si) were matched with the phrase trie,
the phrases LZ(Si)[k, zi −1] will be contained in e. Thereafter the children dictionary of v is updated to
reflect these changes.
If the locus is an edge e = (v1, v2), we have to split e. Assume that ke phrases were matched with e. We
create two new nodes, leaf and split and three edges, e1 = (v1, split), e2 = (split, v2) and e3 = (split, leaf).
The labels on the new edges are defined:
e1 has the phrases el[0, ke −1], where el are the phrases of e.
e2 has the phrases el[ke, |el| −1].
e3 has the phrases LZ(Si)[k, zi −1], where k is the total number of phrases that were matched between
the phrase trie and LZ(Si).
All children dictionaries are then updated to reflect these changes and e is deleted. We continue with the
next suffix by incrementing i by 1, progressing until i = n.

Correctness
The main steps above, which include the matching, splitting of an edge, etc. are equal to the
steps seen in algorithms for naively constructing the suffix tree [10, p. 93]. These steps are slightly altered
to fit the structure of the phrase trie, without ruining the correctness.
Analysis
The procedure of inserting new nodes involves dynamically updating the children dictionaries.
We use dynamic perfect hashing [5] to do so, which ensures that the query time remains O(zP +log n+occ).
We do at most O(n) insertions to the dictionaries, thus using O(n) amortized expected time. We also match
at most O(Pn−1
i=0 zi) = O(n2) phrases, each being a constant operation, therefore using O(n2) time. All
suffixes are compressed by the CSC algorithm, which also uses O(n2) time. So in total O(n2) amortized
expected time is used. As the algorithm returns the phrase trie with quadratic space, O(n2) space is needed.
6
Preprocessing the Phrase Trie with linear space
The phrase trie with linear space is easily obtained from the phrase trie with quadratic space. Trivially this
requires O(n2) space and time at some point, making the method infeasible for large S. We here show three
preprocessing algorithms for the phrase trie with linear space. What they have in common is that we only
compress phrases which are used for routing during construction.
6.1
Suffix ordered algorithm
Again we start by computing SA, SA−1 and a LCP data structure over S. We then apply the same steps as
shown in section 5 but with one modification. Instead of computing LZ(Si), we compute LZR(ϕi, Si). Here
LZR(ϕi, Si) is given in the preliminaries and ϕi is computed as:
ϕi = max(LCPA[SA−1[i]], LCPA[SA−1[i] + 1])
If LCPA[SA−1[i]] or LCPA[SA−1[i] + 1] are undefined, they are set to 0. ϕi thereby describes the maximal
LCP between Si and any other suffix. Thereafter the algorithm continues with the steps shown in the previous
14
==PAGE:15==
section, but by using LZR(ϕi, Si) instead of LZ(Si). When inserting or updating an edge we also add or
update the length of the substring which the edge represents.
We complete the construction by visiting each edge and deleting their contained phrases. Whilst doing
so, we also add a pointer to an arbitrary leaf below the edge. The CSC algorithm or the KKP3 algorithm is
used for compression. Which one that should be used in practice is discussed later.
6.1.1
Correctness
We may encounter difficulties by using LZR(ϕi, Si) instead of LZ(Si), as more phrases could be needed
for matching (and therefore also when splitting an edge). By the definition of LZR(ϕi, Si), we know that
ϕi is contained in the second last phrase. Due to how ϕi is computed we know that for any given suffix
Sj : Sj ̸= Si, all phrases except the last one of LZR(ϕi, Si) is needed for any matching with LZR(ϕi, Sj).
Moreover, we need the last phrase of LZR(ϕi, Si) to update the children dictionaries correctly when splitting.
Therefore we do not need the remaining phrases of LZ(Si)[k + 1, zi −1] for any matching or routing. These
can be discarded, or simply not be computed, which we do by using LZR(ϕi, Si). Furthermore, we ensure
that the auxiliary information of an edge is correct by dynamically updating the lengths and finally setting
the leaf pointers while removing the remaining phrases. The correctness of the algorithm is thereby given
by combining these arguments with the correctness of the phrase trie with quadratic space.
6.1.2
Analysis
By the definition of LZR(ϕi, Si) we observe that the algorithm compresses the following substrings:
S[0, ϕ0], S[1, ϕ1 + 1], . . . , S[i, ϕi + i], . . . , S[n −1, ϕn−1 + n −1]
Therefore a total of Φ = Pn−1
i=0 ϕi characters are compressed. Using the KKP3 algorithm, this would require
O(Φ) time. Applying the same logic shows that at most O(Φ) phrases are matched during insertion. Deleting
all phrases requires O(Φ) time, as at most O(Φ) phrases are compressed. So, finalizing the construction takes
O(n + Φ) time, as each edge is visited and all phrases are deleted. Moreover, we will also need to create
O(n) nodes and edges, where we again use perfect dynamic hashing for the children dictionaries. Combining
these shows that the algorithm runs in O(n + Φ) amortized expected time. For space, we now only store
O(Φ) phrases during construction. Moreover we still store the auxiliary data structures, nodes and edges,
which require O(n). Thus O(n + Φ) space is used at some point.
We now investigate what Φ is bounded by in terms of n. It is noted that Φ is bounded by O(Pn−1
i=1 LCPA[i])
due to the definition of ϕi. So the question is what structures the LCP-array can take. We show the two
extremes. If S = Σ, that is S is the alphabet, then Pn−1
i=1 LCPA[i] = 0, since no suffixes of S share a prefix.
To maximize the sum, each suffix Si must have a corresponding suffix Sj such that LCP(Si, Sj) = O(n). This
can be achieved by the general structure S = S1S1S2S2, . . . , SuSu. Here u = O(1). Due to such strings,
Pn−1
i=1 LCPA[i] = O(n2). This shows that Φ = Ω(1) and Φ = O(n2).
As a note it is observed that Φ can be found in linear time, which in a practical use-case allows multiple
on the fly decisions. For example, if Φ = Θ(n2) we can use the CSC algorithm to do the compression, as
it uses O(n2) time and is fast in practice. Otherwise we use the KKP3 algorithm on each suffix to compute
LZR(ϕi, Si).
6.2
Lexicographical ordered algorithm, variant 1
Until now, the suffixes have been asserted in suffix order. We know from the realm of suffix trees that
processing the suffixes in lexicographical order results in O(n) space and time constructions [17]. Using this
as an inspiration, our goal is to reduce the space complexity to O(n). Moreover, because the CSC algorithm
was designed to compress suffixes in suffix order, we can no longer use it for compression. In alternative, we
use KKP3 for the algorithms below.

==PAGE:16==
6.2.1
Algorithm
We preprocess the same auxiliary data structures as in section 6.1. We process the suffixes in lexicographical
order, so we iterate over SA instead of just i. Again we use LZR(ϕSA[i], SSA[i]) instead of LZ(SSA[i]). For
i = 0 we insert LZR(ϕSA[0], SSA[0]), resulting in the phrase trie containing a root node and a leaf. For i > 0
we first compute the following before insertion: Let pathi−1 be an array of size |LZR(ϕSA[i−1], SSA[i−1])|.
The element pathi−1[j] is a pointer to an edge in the current phrase trie, such that the edge contains phrase
LZR(ϕSA[i−1], SSA[i−1])[j], and the edge is on the root-to-leaf path, ending in the leaf representing SA[i−1].
That is, pathi−1 is an array which keeps track of what edges contain the phrases from the (i−1)th iteration.
We then compute LZR(ϕSA[i], SSA[i]). Instead of using match to find the locus, we now sequentially compare
phrases of LZR(ϕSA[i−1], SSA[i−1]) and LZR(ϕSA[i], SSA[i]). Assuming that the first mismatching phrase is
found at index k, we use pathi−1[k] to find the locus. Depending on the locus, we either split an edge or add
a new child to a node, as explained in section 5. To finish the ith iteration, we do an intermediate phrase
pruning. Let Ei−1 be the edges from the root-to-leaf path ending in the leaf that corresponds to SSA[i−1].
Ei is defined similarly. Let E = (Ei−1 △Ei) ∩Ei−1, where △is the symmetric difference operator. That is,
E is the set of edges that are contained in Ei−1 and not in Ei. For each edge e in E, we delete all phrases
contained in e, except the first. We then increment i by one, repeating the same steps. Again we make sure
that an edge contains the length of the substring it represents. This can be achieved by maintaining a string
depth on each node.
Lastly when i = n −1, the construction is finalized. We delete all remaining phrases and initialize leaf
pointers and children dictionaries.
6.2.2
Correctness
The algorithm is similar to the suffix-ordered preprocessing algorithm, but has two modifications; using
pathi−1 to find the locus and the dynamic deletion of phrases. Both are correct because we assert the suffixes
in lexicographical order. Consider the ith iteration. We observe that the first phrases of LZR(ϕSA[i], SSA[i])
matches maximally with the first phrases of LZR(ϕSA[i−1], SSA[i−1]) compared to other compressed suffixes.
This is due to SSA[i] and SSA[i−1] sharing the longest common prefix at the ith iteration. Therefore, we
know that the locus can be found by using a pointer in pathi−1. Specifically, the pointer will be at position
k, where k is the first mismatching phrase between the two phrase arrays. Similarly, the phrase trie does not
need other phrases than those in LZR(ϕSA[i−1], SSA[i−1]) for adding a new leaf (e.g. by splitting an edge), as
we are ensured that LZR(ϕSA[i−1], SSA[i−1]) contains enough information due to the mentioned observation.
Thus, all other phrases can be deleted, except the first phrase of each edge.
We note that pathi−1 and Ei−1 can be found by a leaf-to-root traversal, starting from the leaf that
represents SA[i −1]. We keep one phrase per edge, as this is needed for creating the children dictionaries.
6.2.3
Analysis
In regards to time, we again compress a total of O(Φ) characters, requiring O(Φ) time using the KKP3
algorithm.
Moreover, we find pathi−1 at each iteration, which uses O(ϕSA[i−1]) time by a leaf-to-root
traversal. So, O(Φ) is used for finding pathi for all i. Similarly for finding all E sets. To delete phrases,
O(Φ) is used, as at most O(Φ) phrases are created during construction. The sequential comparison requires
O(Φ) for all i, again due to at most O(Φ) characters being compressed. The children dictionaries are no
longer dynamically updated, allowing us to use perfect hashing [8] instead of dynamic perfect hashing. The
length of all children dictionaries sum to O(n). Therefore, O(n) expected time is needed. We also construct
the auxiliary data structures, nodes and edges using O(n) time. Combining these, we obtain O(n + Φ)
expected time to preprocess the phrase trie with linear space.
For space, it can be observed that at any point during construction, at most two root-to-leaf paths fully
contain phrases. Formally this is O(max
i>0 (zSA[i−1] + zSA[i])) = O(n). All other edges contain one phrase. A
node contains information linear in the number of children it has. Thereby the edges and nodes need O(n)
16
==PAGE:17==
space. Moreover we keep pathi−1, Ei−1, Ei and E during the ith iteration. These need O(n) space. So, in
total the algorithm uses O(n) space, proving theorem 1.
6.2.4
Variant 2
As a small theoretical extension we show how to apply a different suffix compression algorithm to obtain a
time complexity dependent on the compression ratio. Instead of computing LZR(ϕSA[i], SSA[i]) by the KKP3
algorithm, we can use the results given in [13] and [9]. Combining these results we obtain an algorithm
using O(|LZ(S′)| log log n) time to compute LZ(S′) where S′ is a substring of S. The time to construct
the data structure is O(n√log n) and the space needed is O(n log log n). Translating this to the relevant
parameters and context here, we get: O(n√log n + Pn−1
i=0 ψi log log n) time where ψi = |LZ(S[i, i + ϕi])|.
Defining Ψ = Pn−1
i=0 ψi we get O(n√log n + Ψ log log n). Ψ thus depends on Φ and the compression ratio.
We now discuss the trade-off between variants 1 and 2. For space, variant 2 uses slightly more due to the
extra log log n factor. It becomes more complicated regarding time. What can be concluded is that with
bad compression ratios, such that Ψ = O(Φ), results in variant 2 performing worse. Moreover, if Φ = O(n)
which entails Ψ = O(n), variant 2 performs worse, due to the additional n√log n term.
6.3
Problems and attempts
What is common between the preprocessing algorithms for the phrase tries is that we compute the LZ77
phrases of the same substrings multiple times. The time will be greatly reduced if the repetition can be
eliminated. In the following we explore possibilities of said elimination, ultimately resulting in problems that
we could not solve. Firstly, we tried to construct the suffix tree and directly transform it into either phrase
trie. This idea was quickly ruined by strings like S = aabbaabbaaabbaa$. The structure of the phrase trie is
significantly different than that of the suffix tree in such cases (see figure 17). Secondly, we consider the O(n)
time suffix tree construction algorithm given in [17]. Simply put, it uses LCPA and SA to traverse the suffix
tree in depth-first order whilst building the tree. We now try to modify this algorithm to function on phrase
tries. To conserve time, we will compress as few substrings as possible. Now consider the ith iteration. Here
SSA[i] needs to be inserted. Assume that an edge e = (parent, child) needs to be split, and that e is found
by the ongoing depth-first traversal. In the suffix tree setting, we could split the edge by information given
in LCPA[i]. In the phrase trie setting we have to answer corollary 1 to find the split. That is, we need to
figure out where LZ(SSA[i]) and LZ(SSA[i−1]) have the first differing phrase. Which is the same as finding
the phrase that contains LCPA[i] for both. To our knowledge, no fast solution exists for this purpose.
Problem 1, LZ77 position indexing: Given two strings S and S′ let LCP(S, S′) = c. Find the phrase
that contains c in LZ(S) and LZ(S′).
Problem 2, LZ77 position indexing on lexicographical ordered suffixes: Given a string S, answer
problem 1 with respect to SSA[i−1] and SSA[i], for all i : 0 < i < n.
Trivially, problem 1 can be answered by computing LZ(S) and LZ(S′) (or the restricted version of these)
which is exactly what we have done until now. In conclusion, any preprocessing algorithm for the phrase
trie with linear space is bounded by Ω(n + Λ) time where Λ is the time for answering problem 2, as this is
the minimum requirement for performing O(n) splitting operations. With the proposed algorithms we can
do so in either O(n + Φ) or O(n√log n + Ψ log log n) time.

Conclusion
We have shown new algorithms for preprocessing the two variants of phrase tries, proving theorem 1. It
is unknown whether the bounds of theorem 1 are optimal, but we show that answering problem 2 given
in section 6.3 is necessary for determining the lower bound. Furthermore, we have presented an algorithm
for compressing all suffixes of a string, which albeit being theoretical slower than known techniques, solely
builds on arrays and has little to no overhead. Finally, we have tested the tries in practice. We conclude that
the phrase tries outperform the suffix tree in most cases when answering SICP queries. The disadvantage is
that the phrase tries take longer time to construct.

Revisting LCP:
In the preliminaries we state that LCP(Si, Sj) can be computed by:
LCP(Si, Sj) = RMQLCP A(SA−1[i] + 1, SA−1[j]).
We here explain why this is correct. LCPA contains the LCP lengths between lexicographical ordered
suffixes. Given LCPA[i] = c and LCPA[i + 1] = c + x entails that LCP(SSA[i−1], SSA[i+1]) = c. This is
due to the suffix array being sorted in lexicographical order.
Thereby, SSA[i−1] has an equal or longer
common prefix with SSA[i] than SSA[i−1] has with SSA[i+1]. So, we must take the smallest of the two (c
and c + x). Which shows min(c, c + x) = LCP(SSA[i−1], SSA[i+1]) = min(LCPA[i], LCPA[i + 1]). Repeating
this argument between positions SA−1[i] + 1 and SA−1[j] in LCPA shows that finding the minimum in
LCPA[SA−1[i] + 1, SA−1[j]] is correct, which we do using RMQ.

Problem definition
Given two strings, S and P, the string indexing problem consists of finding all
starting positions of P in S. Commonly P is known as the pattern with |S| > |P|.
In this thesis we shall investigate the problem where P is compressed, which
is known as the string indexing with compressed pattern problem (SICP). We
will use the LZ77 compression scheme to compress P, with LZ(P) denoting the
phrases obtained by compressing P. A typical use case which is affected by the
SICP problem is in the client-server scenario. Here the server contains S. In
order to save bandwidth, the client sends a request which includes LZ(P), thus
having the server do the indexing.
Previous work
A naive solution would be to decompress LZ(P), transforming the SICP prob-
lem instance to a string indexing instance. Let m = |P| and let occ denote
the number of occurrences of P in S. The naive method results in O(m + occ)
time, as LZ(P) has to be decompressed and the indexing is done using a suf-
fix tree. Both require O(m) and reporting occ elements require O(occ). O(n)
space is required for the suffix tree, with n = |S|. Novel solutions exist which
does not decompress LZ(P) while obtaining better time bounds.
These are
covered in [2], with the main finding being a data structure using O(n) space
and O(z + log(m) + occ) time. Here z = |LZ(P)|.
Goals of the project
The primary goal of this project is to review and implement algorithms / data
structures for the string indexing with compressed pattern problem. These will
be based on state of the art theory, given in [2]. Moreover, they will be tested
and bench-marked, investigating whether the theory holds in practice. To reach
this goal, other data structures and algorithms will be covered and implemented,
including Nearest common ancestor, Longest common prefix, LZ77, Suffix trees,
ART-decomposition ([1]) and their optimal prepossessing algorithms.
